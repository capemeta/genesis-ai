# 聊天模块前后端接口映射

## 1. 目标

本文档用于说明当前聊天模块首批可联调接口，明确：

- 前端页面区域对应哪个后端接口
- 关键字段应该如何使用
- 当前阶段哪些能力是真实可用的
- 哪些能力只是架子已搭好、后续继续增强

当前实现遵循：

- 聊天空间 `chat_spaces` 对应首页卡片层
- 聊天会话 `chat_sessions` 对应进入空间后的左侧会话列表
- 聊天消息 `chat_messages` 对应中间消息区
- 聊天轮次 `chat_turns` 对应一次问答执行快照


## 2. 当前已可联调能力

- 聊天空间增删改查
- 聊天空间能力挂载增删改查
- 会话列表与会话创建
- 消息列表读取
- 发送消息
- 真实调用租户聊天模型
- 若绑定知识库，则向模型注入“模拟知识库背景提示”

当前暂未实现但接口结构已预留：

- 真实知识库检索
- 引用生成
- 工具调用执行
- MCP 调用
- 工作流运行明细联动
- 流式输出


## 3. 页面与接口映射

### 3.1 聊天首页卡片区

页面用途：

- 展示“Q4 财务分析”“法律文档审核”这类聊天空间

接口：

- `GET /api/v1/chat/spaces`
- `POST /api/v1/chat/spaces`
- `GET /api/v1/chat/spaces/{space_id}`
- `PUT /api/v1/chat/spaces/{space_id}`
- `DELETE /api/v1/chat/spaces/{space_id}`

关键字段：

- `id`: 聊天空间 ID
- `name`: 卡片标题
- `description`: 卡片副标题
- `status`: `active | archived | deleted`
- `entrypoint_type`: `assistant | workflow | agent`
- `entrypoint_id`: 入口对象 ID
- `default_model_id`: 默认聊天模型，当前语义是 `tenant_models.id`
- `default_retrieval_profile_id`: 默认检索模板
- `default_config`: 空间默认配置
- `is_pinned`: 是否置顶
- `display_order`: 排序值
- `last_session_at`: 最近会话活跃时间


### 3.2 进入空间后的左侧会话列表

页面用途：

- 展示当前聊天空间下的多个具体会话

接口：

- `GET /api/v1/chat/spaces/{space_id}/sessions`
- `POST /api/v1/chat/spaces/{space_id}/sessions`
- `GET /api/v1/chat/sessions/{session_id}`
- `PUT /api/v1/chat/sessions/{session_id}`

关键字段：

- `id`: 会话 ID
- `chat_space_id`: 所属聊天空间
- `title`: 会话标题
- `summary`: 会话摘要
- `status`: 会话状态
- `channel`: `ui | api | system`
- `visibility`: `user_visible | backend_only`
- `persistence_mode`: `persistent | ephemeral`
- `config_override`: 会话级配置覆盖
- `last_message_at`: 最近一条消息时间
- `stats.message_count`: 消息数
- `stats.turn_count`: 轮次数
- `stats.total_tokens`: 当前会话累计 token
- `stats.last_turn_status`: 最近轮次状态


### 3.3 中间消息区

页面用途：

- 展示当前会话下的消息列表

接口：

- `GET /api/v1/chat/sessions/{session_id}/messages`
- `POST /api/v1/chat/sessions/{session_id}/messages/stream`

关键字段：

- `id`: 消息 ID
- `turn_id`: 所属轮次 ID
- `role`: `system | user | assistant | tool`
- `message_type`: 当前阶段主要是 `text`
- `status`: `pending | streaming | completed | failed | cancelled`
- `content`: 主文本
- `content_blocks`: 富内容块
- `display_content`: 给前端直接显示的文本
- `error_code`
- `error_message`
- `metadata`: 扩展元数据
- `citations`: 当前阶段通常为空，后续做真实检索后会写入


### 3.4 输入框发送消息

页面用途：

- 用户输入问题并提交

接口：

- `POST /api/v1/chat/sessions/{session_id}/messages`

请求体：

```json
{
  "content": "请帮我总结一下 Q4 财务表现",
  "source_channel": "ui",
  "content_blocks": [],
  "config_override": {
    "temperature": 0.7,
    "max_tokens": 1024
  },
  "metadata_info": {}
}
```

返回结构：

- `session`: 最新会话数据
- `turn`: 本轮执行快照
- `user_message`: 用户消息
- `assistant_message`: 助手回复

当前行为说明：

- 会真实调用租户模型
- 若聊天空间挂了知识库，会注入一段“模拟知识库背景”系统提示
- 当前不会做真实召回，不会生成真实引用


### 3.4.1 SSE 流式发送

接口：

- `POST /api/v1/chat/sessions/{session_id}/messages/stream`

请求体与普通发送一致，额外支持查询参数：

- `chunk_size`: 每个 SSE 文本分片大小，默认 `32`

返回类型：

- `text/event-stream`

当前事件类型：

- `turn.created`
- `assistant.status`
- `assistant.delta`
- `assistant.completed`
- `assistant.failed`

当前实现说明：

- SSE 协议和消息状态流转都已打通
- 当前上游模型仍是同步调用
- 也就是说，前端可以先按正式流式协议开发，但底层还不是 provider 原生 token stream
- 等后续模型平台支持真流式后，只需要替换服务层内部实现，不需要改前端协议


### 3.5 右侧 Session Config

页面用途：

- 展示和修改当前聊天空间/会话的默认配置

建议使用方式：

- 空间级长期配置：更新 `chat_spaces`
- 会话级临时默认配置：更新 `chat_sessions`
- 单轮临时覆盖：在发送消息时走 `config_override`

基础接口：

- `GET /api/v1/chat/spaces/{space_id}`
- `PUT /api/v1/chat/spaces/{space_id}`
- `GET /api/v1/chat/sessions/{session_id}`
- `PUT /api/v1/chat/sessions/{session_id}`


## 4. 下拉初始化接口

接口：

- `GET /api/v1/chat/bootstrap`

返回内容：

- `retrieval_profiles`: 检索模板列表
- `workflows`: 工作流列表
- `knowledge_bases`: 知识库列表
- `models`: 可用聊天模型列表

注意：

- `models[].id` 当前返回的是 `tenant_models.id`
- 前端保存到空间默认模型时，应直接写入 `chat_spaces.default_model_id`


## 5. 能力挂载接口

页面用途：

- 给聊天空间挂知识库、工作流等能力

接口：

- `GET /api/v1/chat/spaces/{space_id}/capabilities`
- `POST /api/v1/chat/spaces/{space_id}/capabilities`
- `PUT /api/v1/chat/spaces/{space_id}/capabilities/{binding_id}`
- `DELETE /api/v1/chat/spaces/{space_id}/capabilities/{binding_id}`

当前已做有效校验的能力类型：

- `knowledge_base`
- `workflow`

其他能力类型目前只保留表结构和接口形状，后续再接真实校验与执行。


## 6. 推荐前端联调顺序

1. 调 `GET /api/v1/chat/bootstrap` 初始化下拉数据
2. 调 `GET /api/v1/chat/spaces` 展示首页卡片
3. 创建一个聊天空间并挂载知识库
4. 在空间内创建会话
5. 调 `POST /api/v1/chat/sessions/{session_id}/messages` 发送消息
6. 成功后再调 `GET /api/v1/chat/sessions/{session_id}/messages` 做完整刷新


## 7. 当前最佳实践取舍说明

这版实现是“能稳定联调”的首批方案，不是终局方案。

已经做到的最佳实践：

- 会话层、消息层、轮次层分离
- API 只做接入，业务逻辑集中在 service
- 发送消息会落完整执行快照
- 先使用真实模型调用，不用假回复糊弄前端
- 知识库能力先走模拟提示，避免现在就把检索逻辑写死
- 流式接口优先稳定前端协议与状态机，再逐步替换成上游原生流式

当前刻意保守的点：

- 不做流式
- 不做真实 RAG
- 不做复杂工具调用
- 不做多轮推理编排

这不是能力缺失，而是为了先把前后端主链路打通，并保证后续增强时不需要推翻当前接口结构。
