# Genesis AI 权限系统深度解析 (URAC + RBAC)

本文档旨在详细说明 Genesis AI 系统中的权限设计逻辑。我们采用了 **RBAC (基于角色的访问控制)** 与 **URAC (通用资源访问控制)** 结合的复合架构，以支持从“个人版”到“企业级 SaaS”的平滑扩展。

---

## 一、 核心架构图设计

权限判断遵循以下三层逻辑：

1.  **租户级隔离 (Tenant Isolation)**：物理/逻辑层面的第一道防线。所有查询必须带上 `tenant_id`。
2.  **功能级鉴权 (RBAC)**：决定用户“能做什么”（例如：能不能创建知识库、能不能修改 API Key）。
3.  **数据级鉴权 (Resource Permissions)**：决定用户“能看哪个”（例如：在 100 个知识库中，哪 5 个对他可见）。

---

## 二、 三大支柱详解

### 1. 灵活的组织架构 (Tree-based Organization)
我们使用 PostgreSQL 的 `ltree` 插件来实现组织架构。
*   **设计价值**：支持无限层级的部门树。
*   **权限穿透**：当资源共享给“研发部”时，系统可以自动计算其下的“前端组”、“后端组”成员是否拥有继承权限，查询效率极高。

### 2. 通用资源权限管理 (Universal Resource Access Control)
这是系统最核心的创新。所有的“资产”（资产包括：知识库、Agent、工作流、自定义工具）统一通过 `resource_permissions` 表进行授权。

*   **多主体授权**：
    *   **User -> Resource**：点对点授权（如：共享给 A 同学）。
    *   **Role -> Resource**：按职能授权（如：共享给所有“知识库审计员”）。
    *   **Org -> Resource**：按部门授权（如：共享给整个“销售部”）。
*   **权限等级**：
    *   `view`：只读，无法修改切片和设置。
    *   `edit`：可管理数据，但不能删除资源或授权他人。
    *   `manage`：完全控制权，包括转让所有权和删除。

### 3. 可见性模型 (Visibility Logic)
为了优化高性能检索场景，我们引入了 `visibility` 标记：
*   **Private**：仅所有者 (Owner) 和被授权主体可见。
*   **Tenant_Public**：租户内全员可见（如：公司行政手册）。

## 三、 典型业务场景下的判权流程

### 场景一：用户进行 RAG 对话
当用户发起提问时，系统需要检索向量数据库。过滤条件如下：
```sql
SELECT * FROM documents 
WHERE kb_id IN (
    -- 1. 查找我作为 Owner 的知识库
    SELECT id FROM knowledge_bases WHERE owner_id = :current_user
    UNION
    -- 2. 查找租户内公开的知识库 
    SELECT id FROM knowledge_bases WHERE visibility = 'tenant_public' AND tenant_id = :tid
    UNION
    -- 3. 查找显式共享给我的知识库 (包括共享给我的角色或我的部门的)
    SELECT resource_id FROM resource_permissions 
    WHERE resource_type = 'knowledge_base'
    AND (
        (subject_type = 'user' AND subject_id = :current_user)
        OR (subject_type = 'role' AND subject_id IN (:user_roles))
        OR (subject_type = 'org' AND :user_org_path <@ subject_org_path) -- ltree 路径包含关系
    )
)
```

### 场景二：API 密钥访问
通过 `api_keys` 访问时，系统会检查 `scopes`（JSONB 格式）：
*   如果 `scopes` 为 `["chat", "search"]`，则拦截该 Key 的 `delete` 操作。
*   API Key 绑定的 `owner_id` 决定了该 Key 能看到哪些数据资源。

---

## 四、 审计与追踪 (Who & When)

我们坚持“资产必有主，变动必留痕”的原则：
1.  **所有权 (Owner)**：所有核心表都有 `owner_id`，代表业务负责人，不随修改而变。
2.  **冗余审计**：`created_by_name` 和 `updated_by_name` 被冗余存储，以支持在不进行 Table Join 的情况下快速显示操作详情。
3.  **审计日志 (Audit Log)**：所有的权限变更（如给某人授权、改变知识库可见性）必须记录到 `audit_logs`。

---

## 五、 未来扩展：ABAC (基于属性的访问控制)

由于我们在 `knowledge_bases` 和 `documents` 中预留了 `metadata (JSONB)` 字段，未来可以通过配置解析规则实现更复杂的逻辑，例如：
*   “只有职级 > P7 的用户能访问标记为 `confidential = true` 的文档。”
*   这类逻辑无须改表，只需在后端鉴权层增加一行对 JSONB 的过滤判断即可。
