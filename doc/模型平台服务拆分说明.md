# 模型平台服务拆分说明

## 目标

将原 `model_platform_service.py` 的“设置管理、接入同步、运行时路由、统一调用、响应归一化”拆分为清晰模块，保持现有业务逻辑与接口行为不变。

## 当前结构

主聚合入口：

- `genesis-ai-platform/services/model_platform_service.py`
  - 保留内置厂商定义 `BUILTIN_PROVIDER_DEFINITIONS`
  - 保留 `seed_builtin_model_provider_definitions`
  - 统一回填导出各 service，兼容既有导入路径

子模块目录：

- `genesis-ai-platform/services/model_platform/__init__.py`
  - 模块统一导出
- `genesis-ai-platform/services/model_platform/settings_service.py`
  - 设置中心聚合逻辑：`ModelSettingsService`
  - CRUD 包装服务：
    - `ModelProviderDefinitionService`
    - `TenantModelProviderService`
    - `TenantModelProviderCredentialService`
    - `PlatformModelService`
    - `TenantModelService`
    - `TenantDefaultModelService`
    - `ModelAdapterBindingService`
- `genesis-ai-platform/services/model_platform/provider_integration_service.py`
  - 厂商连通性测试与模型同步：`ModelProviderIntegrationService`
  - 包含模型发现与凭证加载能力
- `genesis-ai-platform/services/model_platform/runtime_profile_service.py`
  - 运行时画像解析：`RuntimeProfileServiceMixin`
  - 默认策略矩阵（`adapter/request_schema/endpoint_path`）
  - `CapabilityRuntimeProfile`
- `genesis-ai-platform/services/model_platform/response_normalizers.py`
  - 响应归一化：`ResponseNormalizerMixin`
- `genesis-ai-platform/services/model_platform/invocation_service.py`
  - 统一调用入口：`ModelInvocationService`
  - 覆盖 `chat/embed/rerank/asr/tts/debug_invoke`

## 依赖关系（单向）

1. `invocation_service` 依赖：
   - `runtime_profile_service`
   - `provider_integration_service`
   - `response_normalizers`
2. `settings_service` 与 `provider_integration_service` 不依赖 `invocation_service`
3. `model_platform_service.py` 仅聚合导出，不承载调用细节

## 扩展建议（后续）

1. 新增能力（例如 `image/ocr/document_parse`）优先在 `invocation_service` 增加入口，再按复杂度拆到 `capabilities/*_invoker.py`
2. 厂商默认策略仅在 `runtime_profile_service.py` 的策略矩阵维护
3. 响应协议变种统一在 `response_normalizers.py` 增加归一化函数
4. 接入发现差异仅在 `provider_integration_service.py` 扩展

## 兼容性说明

当前拆分保持以下不变：

- API 层调用方式不变
- `services.model_platform_service` 的对外导出不变
- 业务逻辑与路由行为不变（仅模块迁移和职责分离）
