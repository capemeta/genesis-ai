# 用户体系与个人资料设计方案

> 更新说明（2026-03-28）：
> 当前项目已经调整为“账号全局唯一 + 用户归属单租户”的实现口径。
> 本文中关于“用户名/邮箱/手机号租户内唯一、登录时选择租户”的设计讨论已不再作为当前开发基线。
> 如后续需要支持“一人多租户”，建议升级为 `accounts + tenant_memberships` 模型，而不是继续扩展当前 `users` 单表。

## 1. 文档目的

本文档用于统一当前项目的用户体系设计，作为后续后端表结构、接口、前端 `settings/profile` 页面以及未来身份扩展能力的开发依据。

设计目标：

- 满足当前通用 RAG 知识库 / 智能体平台的用户管理需求
- 保持多租户语义清晰，避免后续唯一约束和组织挂载问题
- 当前不实现 SSO / OAuth 登录，但要预留未来兼容能力
- 个人资料页字段设计要兼顾当前可用和未来扩展
- 账号安全字段与偏好字段分层清晰，避免语义混乱

---

## 2. 当前设计评估结论

当前 `users` 表已经具备基本可用性，但整体更像“账号登录表”，还不够“平台用户主档”。

当前主要问题：

- `username`、`email` 使用全局唯一，不符合多租户平台的长期设计
- `organization_id` 没有外键约束，存在悬挂数据风险
- `is_verified` 语义不正确，不能再通过“email 非空”推断“已验证”
- 只有 `status`，缺少失败次数、锁定截止时间、密码修改时间等安全闭环字段
- 只有 `del_flag`，不利于审计、恢复、定期清理和部分唯一索引设计
- `settings` 方向是对的，但应仅承载偏好类数据，不应承载安全关键字段

结论：

- `users` 主表需要升级为“平台用户主档”
- 未来 SSO / OAuth 不应直接塞字段到 `users`，应通过独立扩展表承载
- 前端 `settings/profile` 当前字段够最小可用，但还不够平台级体验

---

## 3. 总体设计原则

### 3.1 主表职责边界

`users` 主表负责以下内容：

- 用户身份主信息
- 平台内账号状态
- 安全相关核心字段
- 组织归属
- 基础个人资料
- 偏好配置入口（`settings`）

`users` 主表不直接承载以下内容：

- 第三方身份绑定明细
- OAuth token
- SSO subject / access token / refresh token
- MFA 设备明细
- 历史登录事件明细

这些内容应放入独立扩展表。

### 3.2 多租户原则

对于本平台，用户唯一性应以“租户内唯一”为主，而不是“全局唯一”。

建议：

- `username`：租户内唯一
- `email`：租户内唯一
- `phone`：如果未来支持短信登录或短信找回密码，则租户内唯一

### 3.3 字段分层原则

字段分三类：

- 核心身份字段：放主列
- 安全状态字段：放主列
- 个性化偏好：放 `settings JSONB`

### 3.4 当前阶段原则

项目当前处于开发阶段，不考虑历史兼容成本，优先采用统一、可维护、长期稳定的结构设计。

## 4. 用户主表设计

## 4.1 建议保留字段

以下字段保留：

- `id`
- `tenant_id`
- `organization_id`
- `username`
- `nickname`
- `password_hash`
- `email`
- `phone`
- `avatar_url`
- `settings`
- `status`
- `last_login_at`
- `last_login_ip`
- `created_by_id`
- `created_by_name`
- `updated_by_id`
- `updated_by_name`
- `created_at`
- `updated_at`

## 4.2 建议新增字段

### 4.2.1 安全与生命周期字段

- `email_verified_at TIMESTAMPTZ`
- `phone_verified_at TIMESTAMPTZ`
- `failed_login_count INTEGER NOT NULL DEFAULT 0`
- `locked_until TIMESTAMPTZ`
- `password_changed_at TIMESTAMPTZ`
- `last_active_at TIMESTAMPTZ`
- `activated_at TIMESTAMPTZ`
- `deleted_at TIMESTAMPTZ`

### 4.2.2 平台通用资料字段

- `job_title VARCHAR(100)`
- `employee_no VARCHAR(100)`
- `bio VARCHAR(500)`

说明：

- `job_title` 用于职位展示、通讯录、Agent persona 引用
- `employee_no` 更偏组织内部标识，普通用户通常只读
- `bio` 用于个人简介、对话助手的用户背景提示、资料展示

## 4.3 建议弱化或淘汰字段

建议逐步淘汰：

- `del_flag`

建议方案：

- 使用 `deleted_at` 作为统一软删除标识
- 如果短期内不方便整体替换，可以先保留 `del_flag`，但新逻辑统一依赖 `deleted_at`
- 最终应逐步收敛到仅使用 `deleted_at`

## 5. 推荐的 `users` 表结构

以下为建议结构示意，后续开发可据此改造 `docker/postgresql/init-schema.sql`。

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    organization_id UUID REFERENCES organizations(id) ON DELETE SET NULL,

    username VARCHAR(100) NOT NULL,
    nickname VARCHAR(255),
    password_hash VARCHAR(255) NOT NULL,

    email VARCHAR(255),
    phone VARCHAR(20),
    avatar_url VARCHAR(512),

    job_title VARCHAR(100),
    employee_no VARCHAR(100),
    bio VARCHAR(500),

    settings JSONB NOT NULL DEFAULT '{}'::jsonb,

    status VARCHAR(20) NOT NULL DEFAULT 'active'
        CHECK (status IN ('active', 'disabled', 'locked')),

    email_verified_at TIMESTAMPTZ,
    phone_verified_at TIMESTAMPTZ,

    failed_login_count INTEGER NOT NULL DEFAULT 0,
    locked_until TIMESTAMPTZ,
    password_changed_at TIMESTAMPTZ,

    last_login_at TIMESTAMPTZ,
    last_login_ip VARCHAR(45),
    last_active_at TIMESTAMPTZ,
    activated_at TIMESTAMPTZ,
    deleted_at TIMESTAMPTZ,

    created_by_id UUID,
    created_by_name VARCHAR(255),
    updated_by_id UUID,
    updated_by_name VARCHAR(255),
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## 6. 唯一约束与索引设计

## 6.1 用户名唯一约束

建议用户名在租户内唯一，且只对未删除用户生效。

```sql
CREATE UNIQUE INDEX uq_users_tenant_username_active
ON users (tenant_id, lower(username))
WHERE deleted_at IS NULL;
```

## 6.2 邮箱唯一约束

建议邮箱在租户内唯一，允许为空，且只对未删除用户生效。

```sql
CREATE UNIQUE INDEX uq_users_tenant_email_active
ON users (tenant_id, lower(email))
WHERE email IS NOT NULL AND deleted_at IS NULL;
```

## 6.3 手机号唯一约束

如果后续支持短信验证码登录、短信找回密码，建议手机号也在租户内唯一。

```sql
CREATE UNIQUE INDEX uq_users_tenant_phone_active
ON users (tenant_id, phone)
WHERE phone IS NOT NULL AND deleted_at IS NULL;
```

## 6.4 常用索引

```sql
CREATE INDEX idx_users_tenant_status
ON users (tenant_id, status);

CREATE INDEX idx_users_tenant_org
ON users (tenant_id, organization_id);

CREATE INDEX idx_users_last_active_at
ON users (tenant_id, last_active_at DESC);

CREATE INDEX idx_users_locked_until
ON users (tenant_id, locked_until);
```

说明：

- `lower(username)` 和 `lower(email)` 用于避免大小写导致的重复数据问题
- 使用部分唯一索引可以自然兼容软删除用户重新注册或重新创建

## 7. `settings JSONB` 统一协议

`settings` 只用于保存偏好配置，不用于保存安全字段。

推荐初始协议如下：

```json
{
  "language": "zh",
  "timezone": "Asia/Shanghai",
  "theme": "system",
  "date_format": "YYYY-MM-DD",
  "time_format": "24h",
  "default_kb_id": null,
  "default_agent_id": null,
  "dashboard_layout": "default"
}
```

字段说明：

- `language`：界面语言
- `timezone`：用户时区
- `theme`：主题，建议支持 `light`、`dark`、`system`
- `date_format`：日期格式
- `time_format`：时间格式，建议支持 `12h`、`24h`
- `default_kb_id`：默认知识库
- `default_agent_id`：默认智能体
- `dashboard_layout`：控制台布局偏好

注意：

- `email_verified_at`、`locked_until`、`failed_login_count` 等安全字段禁止放入 `settings`
- `settings` 内部 key 要统一维护，不允许不同页面随意发明语义重复字段

## 8. 未来 SSO / OAuth 兼容设计

## 8.1 设计结论

当前阶段不做 SSO / OAuth 登录接入，但数据库设计必须允许未来平滑扩展。

原则：

- 不在 `users` 主表中塞第三方身份字段
- 使用独立扩展表 `user_identities`
- 主表和第三方身份关系为一对多

## 8.2 推荐表结构

```sql
CREATE TABLE user_identities (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,

    provider VARCHAR(50) NOT NULL,
    provider_user_id VARCHAR(255) NOT NULL,
    provider_email VARCHAR(255),
    provider_data JSONB NOT NULL DEFAULT '{}'::jsonb,

    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX uq_user_identities_provider_subject
ON user_identities (tenant_id, provider, provider_user_id);

CREATE INDEX idx_user_identities_user_id
ON user_identities (user_id);
```

## 8.3 字段说明

- `provider`：身份提供方，例如 `feishu`、`dingtalk`、`wecom`、`google`、`microsoft`
- `provider_user_id`：第三方平台唯一 subject
- `provider_email`：第三方身份返回的邮箱
- `provider_data`：扩展信息快照，例如租户、部门、头像、原始 claims

## 8.4 为什么不提前塞进 `users`

因为第三方身份信息具备以下特点：

- 来源多
- 字段差异大
- 绑定关系可能一人多身份
- 后续可能解绑、换绑、重绑

因此必须隔离到扩展表，避免污染 `users` 主表。

## 12. 明确约束

后续开发必须遵守以下约束：

- 不允许再把第三方身份字段直接堆入 `users`
- 不允许再用全局唯一约束替代租户内唯一语义
- 不允许把安全关键字段塞入 `settings`
- 不允许个人资料页直接修改平台管理字段
- 若后续要调整 `users`、`user_identities`、个人资料接口返回协议，必须前后端一起统一修改

---

## 13. 最终结论

本项目的用户体系应采用以下方案：

- `users` 作为平台用户主档
- `settings JSONB` 只承载偏好配置
- `user_identities` 预留未来 SSO / OAuth / IdP 接入
- `settings/profile` 升级为“基础资料 + 偏好设置 + 账号安全”三段式页面

该方案能够同时满足：

- 当前阶段快速开发
- 多租户平台长期演进
- 通用 RAG / 智能体平台能力扩展
- 未来接入企业身份系统的兼容性

后续数据库、接口和前端页面开发统一以本文档为准。
