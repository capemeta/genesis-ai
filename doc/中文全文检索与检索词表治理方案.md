# 中文全文检索与检索词表治理方案

## 1. 目标

当前系统已经具备：

- 向量检索
- PostgreSQL 全文检索
- query analysis（同义词、规则过滤、术语上下文）

但中文全文检索仍有三个典型问题：

1. `simple` 配置对中文天然较弱
2. 业务短语、专业术语、英文缩写容易被错过
3. 不同知识库对“应如何切词、哪些词要优先识别”的诉求并不相同

因此需要明确两层能力：

1. 中文全文检索增强
2. 知识库级检索词表治理

---

## 2. 当前实现分层

### 2.1 第一阶段：无扩展版中文全文检索增强

当前已实现：

- 索引写入阶段：
  - 文本归一化
  - 中文连续片段展开
  - 双字 `N-Gram` 展开
  - 英文 / 数字 / 混合术语保留
- 查询阶段：
  - `strict query`
  - `loose query`
  - 短语匹配兜底
- 融合阶段：
  - scope 级短语加权
  - QA / 表格的查询意图加权

这条路径的优点：

- 不依赖 PostgreSQL 中文扩展
- 部署成本低
- 能快速验证收益

局限：

- 对长中文句和复杂短语的理解仍有限
- 没有中文词典、停用词、专有名词切分能力

### 2.2 第二阶段：词表治理与术语优先

第二阶段不是先上重型引擎，而是先补“检索词表治理”。

目标：

- 让知识库自己表达“哪些词不能乱拆、哪些别名要统一、哪些泛词应该弱化”

---

## 3. 分词最佳实践判断

### 3.1 是否必须引入分词器

结论：长期看通常需要某种“分词能力”，但现阶段不一定要直接上数据库扩展。

建议路径：

1. 第一阶段
   - 应用层做归一化、术语切分、N-Gram 兜底
2. 第二阶段
   - 评估是否需要 PostgreSQL 中文扩展
   - 或切到 Elasticsearch / OpenSearch 中文分析器

### 3.2 PostgreSQL 自身是否有中文方案

PostgreSQL 原生：

- `simple`
- `english`

这些并不能真正解决中文分词问题。

因此如果坚持 PG 全文检索，通常有两条路：

1. 应用层自己展开索引文本和查询文本
2. 引入 PG 中文扩展

### 3.3 什么时候考虑引入中文词典 / 分词扩展

建议满足以下条件之一再推进：

1. 中文全文检索已经成为核心主路
2. 当前无扩展方案收益明显不够
3. 需要更精细的短语切分、停用词、行业术语维护

当前阶段结论：

- 继续保留 PG 本地链路
- 先把“词表治理 + 应用层增强”做扎实
- 不急着引入新的基础设施依赖

---

## 4. 是否需要知识库级别的分词维护

需要，但不建议直接对外叫“分词维护”。

更适合的产品概念是：

- 检索术语
- 检索别名
- 优先识别词
- 忽略词 / 低价值词

也就是“检索词表治理”。

### 4.1 为什么需要知识库级治理

因为不同知识库差异很大：

- 财务知识库：强调指标名、缩写、英文财务术语
- 医疗知识库：强调病种、药物名、检查项
- QA 知识库：强调问句口语化表达
- 表格知识库：强调列名、维度名、指标名、枚举值

这些都不适合只靠全局统一词典。

### 4.2 知识库级治理应维护什么

建议最小集合：

1. `term`
   - 标准词
2. `aliases`
   - 别名、缩写、口语写法
3. `mode`
   - `retrieval`
   - `explanation`
   - `both`
4. `is_phrase`
   - 是否应按短语优先
5. `weight`
   - 检索增强权重
6. `enabled`
7. `retrieval_stopwords`
   - 忽略词 / 低价值词
   - 用于压制“介绍、说明、相关内容”这类高频噪声词

### 4.3 不建议做什么

不建议在现阶段做成：

- 完整人工分词词典系统
- 复杂 DSL 规则引擎
- 每个知识库都手工维护大量切词规则

原因：

- 维护成本高
- 用户理解成本高
- 很容易配置失控

## 5. 专业术语是否默认加入检索分词

结论：应该默认参与检索增强，但不能无差别全部强塞进索引。

### 5.1 推荐原则

区分两类术语：

1. 检索术语
   - 用于 query rewrite
   - 用于 query 扩展
   - 用于短语优先匹配
2. 解释术语
   - 用于聊天提示注入
   - 用于回答理解增强

### 5.2 默认策略

建议默认：

- 同义词中心中的标准词 / 别名：进入检索增强
- glossary 中的专业术语：默认进入解释增强
- glossary 术语如果显式标记为 `retrieval` 或 `both`，再进入检索增强

### 5.3 为什么不能默认全量加入索引

如果无差别全加，会带来：

- 索引污染
- 泛化过度
- 某些只适合解释的术语反而干扰检索

---

## 6. 与现有能力的边界

### 6.1 与 synonym 的边界

- `synonym`
  - 标准词归一化
  - 更偏检索
- `retrieval lexicon`
  - 检索短语、别名、停用词、优先识别词
  - 更偏知识库级治理

### 6.2 与 glossary 的边界

- `glossary`
  - 主要服务回答解释
- `retrieval lexicon`
  - 主要服务检索召回和词项展开

### 6.3 与 metadata_fields 的边界

- `metadata_fields`
  - 负责过滤字段和值识别
- `retrieval lexicon`
  - 负责全文检索词项、短语和别名治理

### 6.4 与表格 schema 的边界

- 表格知识库中的：
  - 列名
  - 列别名
  - schema 枚举值
- 应优先作为运行时检索词表来源自动注入
- 不建议要求用户在：
  - `table.schema.columns`
  - `query_analysis.retrieval_lexicon`
  两处重复维护

## 7. 后续实现建议

### 7.1 第一批

1. 基于现有 `synonym / glossary` 做检索增强分层
2. 在检索测试页展示：
   - 命中的检索术语
   - 展开的短语
   - 被忽略的低价值词

### 7.2 第二批

1. 新增知识库级“检索词表”配置
2. 支持：
   - 短语优先
   - 别名扩展
   - 忽略词
3. 表格知识库优先结合：
   - 列名
   - 列别名
   - schema 枚举值

当前进展补充：

- 知识库级检索词表第一版已接入 `query analysis` 主链
- 已支持 `retrieval_stopwords`
- 已支持按知识库类型自动合并轻量默认模板：
  - `general`
  - `qa`
  - `table`
- 已支持把默认模板命中写入调试信息，便于判断模板是否有效
- 检索测试页已可展示“已忽略词项”
- 检索测试页已可展示：
  - 优先短语
  - 优先术语
  - 默认模板命中数
- 表格知识库已支持把：
  - 列名
  - 列别名
  - schema 枚举值
  自动注入运行时检索词表
- PostgreSQL 全文检索第二阶段已补第一版“术语 / 短语优先”：
  - 命中的优先短语会触发更高短语加权
  - 命中的优先术语会触发额外轻量加权
- 当前已继续补强为“多优先项参与”：
  - 不只命中单个最长短语 / 术语
  - 最多前 3 个优先短语 / 优先术语都会参与全文加权
- 当前已把这些统一沉淀到 `query analysis -> lexical search` 主链：
  - `priority_terms`
  - `priority_phrases`
  - 默认模板命中摘要
- 表格知识库已支持把表头、维度字段和指标字段补入 `row_group` 检索投影，作为中文全文检索的表格 contextual retrieval 第一版
- 当前 `row_group` 的含义已固定为：
  - 围绕单行构造的增强上下文投影
  - 不是跨多行真实聚合

### 7.3 第三批

视效果评估再决定：

1. PostgreSQL 中文扩展
2. Elasticsearch / OpenSearch 中文分析器
3. 更复杂的中文词典维护

---

## 8. 当前结论

1. 现阶段不急着直接上 PG 中文分词扩展。
2. 现阶段最优路径是“应用层增强 + 检索词表治理”。
3. 知识库级检索词表治理是有必要的，但应轻量化。
4. 专业术语应默认参与检索增强与解释增强分层，而不是全部无差别强塞进索引。
5. 后续如果中文全文检索成为核心主路，再评估中文词典或外部搜索引擎。
