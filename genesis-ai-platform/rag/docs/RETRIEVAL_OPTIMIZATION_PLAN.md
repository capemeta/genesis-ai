# 检索优化改造方案

## 文档目的

基于当前实现评审文档，给出一版可落地的检索优化方案。

目标不是推倒重来，而是：

1. 先修正当前实现里“配置语义和真实行为不一致”的问题
2. 再提升检索质量与性能
3. 最后再推进更长期的架构升级

---

## 一、总体判断

当前检索系统已经具备继续演进的基础，不建议重写。

建议采用“三阶段改造”：

1. 第一阶段：收口现有行为
2. 第二阶段：增强检索质量和性能
3. 第三阶段：升级到更成熟的检索架构

---

## 二、改造目标

本轮优化建议围绕 5 个目标展开：

1. 配置语义真实
2. 检索结果更稳定
3. 中文全文能力更强
4. 大规模下性能更稳
5. 后续评测和调参有闭环

---

## 三、第一阶段：先修“现有漏洞和假开关”

这一阶段优先级最高，因为这些问题会直接误导前端、测试和后续开发。

## 3.1 修复 `group_by_content_group` 未生效

### 当前问题

配置里有 `group_by_content_group`，但 `_fuse_hits()` 现在基本固定按：

1. `doc_summary:{kb_doc_id}`
2. `content_group_id`
3. `chunk_id`

来聚合，没有真正根据配置切换。

### 改造建议

把聚合逻辑显式拆成两种模式：

1. `group_by_content_group = true`
   - 优先按 `content_group_id`
   - 无 `content_group_id` 时回退 `chunk_id`
2. `group_by_content_group = false`
   - 一律按 `chunk_id`
   - `doc_summary` 仍保留文档级聚合键

### 预期收益

1. 参数语义真实
2. 前端调试页看到的结果与配置一致
3. 更方便比较“按业务组聚合”和“按细粒度块聚合”的效果

---

## 3.2 明确 `use_knowledge_graph` 的处理方式

### 当前问题

`use_knowledge_graph` 目前只是预留在 `extension_branch_context`，并没有实际参与检索。

### 改造建议

二选一，不要暧昧状态长期保留：

1. 如果短期不做 KG 检索
   - 前端/接口层不要对外宣传已支持
   - 后端保留内部预留字段即可
2. 如果近期要做 KG 检索
   - 明确接入点
   - 约定 KG 分支输入输出格式
   - 至少打通“query analysis 输出 -> KG 召回 -> 融合”的最小闭环

### 我的建议

短期先收口为“未实现，不暴露为可用能力”。

原因：

1. 当前主链路还没完全收稳
2. 过早把 KG 混进来会放大排查难度

---

## 3.3 修复 `rerank_model` 名不副实

### 当前问题

现在的 `rerank_model` 并没有真正调用 rerank 模型，而是做启发式 bigram overlap 重排。

### 改造建议

两步走：

1. 短期：
   - 把内部命名和调试信息改清楚
   - 明确当前是 `heuristic_rerank`
2. 中期：
   - 对接真实 rerank 模型
   - 抽象成统一 rerank 端口

### 推荐落地方式

新增一个更清晰的运行时字段：

- `rerank_strategy: none | heuristic | model`

然后：

1. `none`：不重排
2. `heuristic`：走当前实现
3. `model`：接真实 rerank 服务

### 预期收益

1. 配置语义真实
2. 后续替换真实 rerank 时不需要推翻现有接口

---

## 四、第二阶段：提升检索质量

## 4.1 收口 `doc_summary` 的实现方式

### 当前问题

`doc_summary` 当前不是完全没索引，而是：

1. 已经作为 `search_unit` 进入正式 lexical index
2. 同时又保留 `_search_doc_summary_hits()` 作为 Python 补召回

这不是最佳实践，原因：

1. 正式索引与补召回并存，行为复杂
2. 数据量大时 Python 补召回会下降
3. 补召回评分体系和主检索链路不完全一致

### 改造建议

把 `doc_summary` 收口成单一、可解释的策略。

推荐方式：

1. 保留 `doc_summary` 作为 `search_unit`
2. 保留其正式 lexical index
3. 重新评估 Python 补召回是否还要保留
4. 视效果决定是否建立低权重向量索引

### 是否建议给 `doc_summary` 做向量化

建议分两步：

1. 先只做 lexical 正式索引
2. 再用评测数据决定是否增加向量版本

原因：

1. 文档摘要通常更适合做文档级粗召回
2. 如果过早向量化，可能会和 chunk 级召回相互干扰

### 推荐定位

`doc_summary` 更适合作为：

1. 文档级召回补充
2. 长文档全局语义入口
3. “先找到相关文档，再回到 chunk 命中”的辅路

---

## 4.2 引入查询 embedding 缓存

### 当前问题

每次查询都要调用 embedding 模型，重复 query 没有短期缓存。

### 改造建议

参考现有文档向量缓存机制，为 query embedding 增加一层短 TTL 缓存。

建议缓存键：

1. `tenant_model_id`
2. 模型运行时签名
3. 归一化 query 文本

建议 TTL：

1. 10 分钟到 1 小时

### 预期收益

1. 降低高频问句重复成本
2. 降低模型平台压力
3. 提升检索响应速度

---

## 4.3 加强中文全文检索能力

### 当前问题

当前 lexical index 用 `to_tsvector('simple', ...)`，对中文天然不强。

现在主要依赖：

1. 预处理后的全文文本
2. `ILIKE`
3. priority phrase / priority term 的额外加分

这能用，但不是最佳实践。

### 改造路径

有两种路线。

#### 路线 A：继续使用 PostgreSQL

方向：

1. 引入更适合中文的分词能力
2. 明确中英文混合文本的分词策略
3. 保留现有结构，减少迁移成本

适合：

1. 想保持系统简单
2. 检索规模还没有大到必须拆外部搜索引擎

#### 路线 B：把全文检索迁到专业引擎

方向：

1. Qdrant / Milvus 同时承担向量检索与全文检索后端
2. PostgreSQL 保留结构化过滤与事务存储

适合：

1. 中文全文质量要求高
2. 数据规模持续增长
3. 后续想做更复杂的检索分析

### 我的建议

短期走路线 A，先在当前架构内增强中文能力。

原因：

1. 当前系统已经有 `lexical_backend` 抽象
2. 但现阶段更大的问题是现有行为还没完全收口
3. 先做本地增强，风险更低

---

## 4.4 优化融合打分体系

### 当前问题

当前融合分里有很多经验参数：

1. `vector_weight`
2. `scope_weight`
3. `coverage_bonus`
4. `backend_bonus`
5. `query_intent_bonus`
6. `metadata_bonus`

这些值现在主要靠经验，不是基于评测集校准。

### 改造建议

分三步：

1. 保留当前公式结构
2. 建离线评测集
3. 逐类知识库调参

### 推荐做法

至少建立三类评测集：

1. 通用文档问答集
2. QA 知识库命中集
3. 表格检索问句集

评测指标建议至少包括：

1. Top1 命中率
2. Top3 命中率
3. MRR
4. 是否命中正确 `content_group`
5. 是否命中正确 chunk

### 预期收益

1. 调参不再靠感觉
2. 每次改召回逻辑能知道是提升还是退化

---

## 五、第二阶段：提升结构化检索能力

## 5.1 表格检索继续增强

### 当前现状

表格检索已经是当前实现里最成熟的一部分，但还可以继续提高。

### 建议

1. 保留 `row` 作为主向量召回
2. 保留 `row_group` 作为全文辅助召回
3. 增加实验开关，评估是否给 `row_group` 增加低权重向量召回

### 为什么不建议立刻改

因为 `row_group` 一旦向量化，可能会扩大噪声。

所以建议：

1. 先通过评测验证
2. 再决定是否开放

---

## 5.2 QA 检索增强

### 当前现状

QA 检索已经支持：

1. `question_only`
2. `question_answer`
3. 同问 lexical 结构化加分

### 建议继续增强

1. 把“相似问题命中”和“问题原文命中”区分更清楚
2. 评估 answer 向量召回在不同知识库下的收益
3. 进一步提高完全同问稳定性

### 推荐方向

让 `question` 与 `answer` 的职责更清晰：

1. `question` 负责“找到问题”
2. `answer` 负责“补完整答案上下文”

而不要过早把二者做成完全对等的主召回入口。

---

## 六、第二阶段：性能优化

## 6.1 优化候选过滤 SQL

### 当前现状

当前过滤顺序是对的，但后续数据量增大后，下面这些地方会变重：

1. 文档层候选集构建
2. 表格主事实表过滤
3. QA 主事实表过滤

### 建议

1. 补充必要索引
2. 评估主事实表过滤字段的联合索引
3. 对高频过滤模式做专门优化

### 特别建议

对于表格知识库，建议重点检查：

1. `KBTableRow.kb_id`
2. `KBTableRow.kb_doc_id`
3. 高频 `row_data` JSONB 过滤字段

---

## 6.2 明确索引重建策略

### 当前现状

训练阶段会删旧索引再重建。

### 风险

如果数据量继续增大，整批重建会更重。

### 建议

中期考虑：

1. 增量重建
2. 基于内容哈希跳过无变化的 search_unit
3. 把 lexical / vector index 更新做成更细粒度任务

---

## 七、第三阶段：更长期的架构升级

## 7.1 统一检索策略编排层

当前 `HybridRetrievalService` 已经承担了很多职责：

1. query analysis
2. filter resolving
3. embedding
4. vector search
5. lexical search
6. fusion
7. hydration
8. rerank

这在当前阶段没问题，但后续会越来越重。

### 长期建议

把检索主链拆成几个更稳定的组件：

1. QueryPlanner
2. CandidateResolver
3. RecallExecutor
4. FusionRanker
5. ResultHydrator

这样后面要接：

1. KG
2. RAPTOR
3. 多后端 AB 测试

都会容易很多。

---

## 7.2 真正引入模型级 rerank

长期建议把 rerank 彻底从启发式升级到模型级。

推荐形态：

1. 召回宽一些
2. 用 rerank model 对 TopN 候选做二阶段精排

这样可以降低手工融合权重的复杂度。

---

## 7.3 文档级召回与 chunk 级召回分层

长期建议把检索分成两层：

1. 文档级粗召回
2. chunk 级精召回

推荐组合：

1. 文档摘要 / 文档标题 / 文档级关键词做文档级召回
2. chunk / row / question / answer 做细粒度召回

这样对长文档和大知识库更友好。

---

## 八、推荐的实施顺序

## 第 1 周：先收口

1. 修复 `group_by_content_group`
2. 明确 `use_knowledge_graph` 对外状态
3. 把 `rerank_model` 改成真实语义

## 第 2 周：补强主链路

1. 给 query embedding 加缓存
2. 把 `doc_summary` 变成正式索引召回
3. 补检索调试信息，区分正式索引命中与辅助命中

## 第 3 周：评测和调参

1. 建评测集
2. 校准融合权重
3. 评估表格 `row_group` 是否要做低权重向量召回

## 第 4 周及以后：中长期升级

1. 中文全文方案增强
2. 真实 rerank 模型接入
3. 检索链路组件化

---

## 九、哪些改造最值得马上做

如果只选 3 件事，我最建议优先做：

1. 修 `group_by_content_group`
2. 给 query embedding 加缓存
3. 把 `doc_summary` 正式索引化

原因：

1. 这三项都能明显提升系统真实性和稳定性
2. 改动收益大，风险相对可控
3. 不需要立刻引入重型外部依赖

---

## 十、最终建议

当前系统最适合的策略不是“大重构”，而是：

1. 先把现有检索主链路收口成一套行为一致、配置真实的系统
2. 再补 query cache、doc_summary 正式索引、中文 lexical 能力
3. 最后再引入真实 rerank、KG、文档级召回分层

一句话总结：

**先把当前这套混合检索做扎实，再做更高级的检索能力扩展。**
