## 数据库设计约定

### 1. 审计与所有权字段约定 (Audit & Ownership)

为了保证系统的可追溯性和企业级安全性，所有表的设计遵循以下分层原则：

- **业务实体表 (Business Entities)**：如 `tenants`, `knowledge_bases`, `documents`, `tasks` 等。
  - 必须包含：`owner_id` (业务负责人), `created_by_id`, `created_by_name` (冗余以加速显示), `updated_by_id`, `updated_by_name`, `created_at`, `updated_at`。
- **高频/海量数据表 (High-volume Data)**：如 `chunks`, `embeddings`。
  - 仅包含：`created_at`。
  - **理由**：一个文档生成数千个切片，冗余操作人名称会导致存储爆炸。审计信息通过关联的 `document_id` 向上溯源。
- **关系/中间表 (Intermediate Tables)**：如 `user_roles`, `resource_tags`。
  - 仅包含：`created_at`。
  - **理由**：关系通常只有“建立”和“解除”，不存在“修改”动作，无需 `updated_by`。
- **只读日志表 (Read-only Logs)**：如 `audit_logs`, `query_logs`。
  - 仅包含：`user_id`, `created_at`。
  - **理由**：日志为客观事实的追加记录，不可篡改。

### 2. 文件夹与标签系统深度设计 (Folders & Tags)

为了实现极高的灵活性并兼顾“资产管理”与“业务闭环”，系统采用了**“物理-逻辑-应用”三层解耦**架构，并引入了高性能的 **ltree** 树形设计。

- **核心理念**：物理去重（Asset Vault）+ 逻辑多维映射（Logical Mapping）+ 标签元数据三级全覆盖。
- **详细设计方案**：请参考独立文档 [文件夹与标签系统深度设计](./文件夹与标签系统深度设计.md)。

### 3. 特色检索逻辑 (Advanced Query Logic)

- **标签别名**：通过存储别名解决术语不一致导致的分类隔离问题。
- **Self-Query**：支持通过 LLM 从自然语言中提取结构化过滤条件，转化为数据库硬过滤。

### 5. 主键 ID 策略 (Primary Key)

为了兼顾安全与性能，系统采用混合主键方案：

- **UUID (v4/v7)**：用于核心业务实体（如 `users`, `tenants`, `knowledge_bases`, `documents` 等）。
  - **优势**：防止 ID 遍历，支持分布式扩展。
- **BIGINT (BIGSERIAL)**：用于海量流水数据（如 `chunks`, `embeddings`, `messages`, `logs` 等）。
  - **优势**：空间利用率高（8字节），索引 B-Tree 查找极其高效，天然有序。

### 6. 高性能设计：去物理外键化 (Logical Foreign Keys)

为了确保系统在海量切片数据下的写入性能和未来的扩展性：

- **无硬约束**：数据库层面**不设置** `FOREIGN KEY` 约束。
- **强制索引**：所有逻辑上的关联字段（如 `tenant_id`, `kb_id`）必须建立 `INDEX`，以保证查询性能。
- **应用层闭环**：数据的一致性和级联清理（Delete Cascade）由后端业务代码维护，利用数据库事务保证原子性。

**ltree 扩展使用说明**（强烈推荐启用）：

```sql
CREATE EXTENSION IF NOT EXISTS ltree;

-- 创建租户根部门时自动生成 path，例如：
-- tenant_id=1 的根部门 path = 't1'
-- 其子部门研发部 path = 't1.rd'
-- 研发部下的前端组 path = 't1.rd.frontend'
```

**常用查询示例**：

- 获取某部门及其所有子部门：
  ```sql
  SELECT * FROM organizations 
  WHERE tenant_id = 1 AND path ~ 't1.rd.*'::lquery;
  ```
- 获取某部门的所有上级：
  ```sql
  SELECT * FROM organizations 
  WHERE tenant_id = 1 AND 't1.rd.frontend' @> path;
  ```

##### 结构化分块设计（V1）

- `content`：检索文本，是由 `content_blocks` 派生出的最终文本视图，可融合已启用的图片 OCR/视觉理解结果。
- `content_blocks`：正文事实来源，支持单个 chunk 内的“文字 -> 图片 -> 文字”混排，不要求图片单独成为 chunk。
- 图片分析结果建议挂在 `content_blocks[].analysis` 上，便于后续支持按图片选择 `none/ocr/vision/ocr_vision`。
- `source_refs` 继续作为统一定位协议；后续若要扩展到块内更细粒度定位，可在 block 级继续细化，不必推翻现结构。

#### 7. resource_permissions （通用资源权限/共享表）—— 实现极致细粒度控制

这是权限设计的核心，支持将任意资源（知识库、应用、工具）共享给任意主体（用户、角色、部门）。


| 字段名                                                          | 类型           | 约束/索引                  | 描述                                                 |
| ------------------------------------------------------------ | ------------ | ---------------------- | -------------------------------------------------- |

| **resource_type**                                            | VARCHAR(50)  | INDEX                  | 资源类型：'knowledge_base', 'agent', 'workflow', 'tool' |
| **resource_id**                                              | UUID         | INDEX                  | 对应的资源 ID。                                          |
