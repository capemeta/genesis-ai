# 模型中心统一配置与特殊模型路由设计方案

## 1. 文档目标

本文档用于明确 Genesis AI 模型中心在“统一配置”与“特殊模型兼容”之间的正式设计基线。

本文档重点回答以下问题：

1. 模型中心应如何做到“大多数厂商只配置一个 `base_url` 就能工作”
2. 当厂商内部不同能力、不同模型的协议与地址不一致时，如何优雅处理
3. LiteLLM 在整套设计中的合理定位是什么
4. 前端配置如何做到“默认简单，高级可控”

本文档生效后，后续模型平台的前端交互、后端路由、模型同步、手动添加模型、能力调用接口都以本方案为准。

---

## 2. 一句话结论

模型中心的正式设计原则应当是：

**默认按厂商统一配置，特殊情况按能力覆盖，再特殊情况按模型覆盖。**

也就是：

- 普通场景：只配置厂商级 `base_url`
- 特殊能力：允许配置能力级覆盖
- 特殊模型：允许配置模型级调用路由

这样既能保证大多数场景足够简单，也能兼容 Tongyi-Qianwen 这类“同厂商内多协议、多端点、多请求结构”的复杂情况。

---

## 3. 当前问题归纳

结合当前仓库实现，模型中心已经具备较好的基础抽象，但已经暴露出几类结构性问题。

### 3.1 当前优点

- 已经有 `provider_definition / tenant_provider / platform_model / tenant_model` 这组基础对象
- 已经有 `model_adapter_bindings` 这类“适配器覆盖”的预留表
- 已经支持厂商模型同步
- 已经支持手动添加模型
- 已经具备按能力选择 `capability_base_urls` 的雏形

### 3.2 当前核心问题

当前设计仍然偏向：

- 一个厂商，对应一种主协议
- 一个厂商，对应一组能力地址
- 模型只是“名称 + 能力标签”

但实际厂商生态并不是这样。

以 Tongyi-Qianwen 为例：

- Chat 可能走 `compatible-mode/v1`
- Embedding 也可能走 `compatible-mode/v1/embeddings`
- 某些 Rerank 模型可能走 `compatible-api/v1/reranks`
- 某些 Rerank 模型又走原生 `api/v1/services/rerank/...`
- 多模态 Rerank 的请求结构与文本 Rerank 又不同

这说明：

**最终调用协议，不能只挂在 provider 上，必须允许下沉到 capability 级甚至 model 级。**

---

## 4. 正式设计原则

### 4.1 设计总原则

模型中心应遵循以下四条原则：

1. `provider` 负责默认值，不负责所有特例
2. `capability` 负责中层覆盖，但不是默认必填
3. `model` 负责最终调用路由
4. 平台对业务层始终暴露统一接口，不把厂商差异泄漏出去

### 4.2 配置优先级

运行时解析调用配置时，按以下优先级从高到低覆盖：

1. 模型级覆盖
2. 能力级覆盖
3. 租户 provider 默认配置
4. 内置 provider definition 默认配置

这套优先级可以保证：

- 普通厂商零额外心智负担
- 特殊能力可覆盖
- 特殊模型不破坏全局统一配置

## 5. 三层配置模型

### 5.1 第一层：厂商级默认配置

这是模型中心的主路径，用于覆盖 80% 以上场景。

用户默认只需要配置：

- `api_key`
- `base_url`
- 可选 `api_version`
- 可选 `region`

系统自动完成：

- 默认能力可用性判断
- 默认调用协议推断
- 模型发现
- 模型同步
- 默认路由生成

这一层的目标是：

**让大多数厂商只配置一次，就能完成接入。**

### 5.2 第二层：能力级高级覆盖

当同一厂商下，不同能力入口不同，但仍然能按能力维度稳定区分时，允许使用能力级覆盖。

典型例子：

- `chat` 走一套地址
- `embedding` 走另一套地址
- `rerank` 走第三套地址
- `tts`、`asr` 走各自官方音频入口

能力级覆盖建议支持以下字段：

- `base_url`
- `endpoint_path`
- `adapter_type`
- `request_schema`
- `extra_headers`
- `request_defaults`

注意：

**能力级覆盖应是高级配置，不应成为所有厂商默认必填项。**

#### 5.2.1 什么是“能力级高级覆盖”

能力级高级覆盖的含义是：

**在同一个厂商实例下，针对某一种能力单独覆写它的调用方式。**

它解决的问题不是“这个厂商能不能接入”，而是：

- 这个厂商整体能接入
- 但某个能力的网关、协议、请求结构或返回结构和默认路径不同
- 因此需要对这个能力单独指定运行时路由

例如 Tongyi-Qianwen：

- `chat` 默认可能走 `compatible-mode/v1`
- `embedding` 也许继续走 OpenAI-compatible
- `rerank` 却可能既有 `compatible-api/v1/reranks`，也有原生 `api/v1/services/rerank/...`

这时就不能只靠厂商级 `base_url` 解决，需要对 `rerank` 这个能力单独覆盖。

#### 5.2.2 能力级高级覆盖和模型级覆盖的边界

建议按下面的规则判断该用哪一层：

- 如果“同一能力下大多数模型都走同一套协议”，优先用能力级高级覆盖
- 如果“同一能力下只有个别模型是特殊协议”，优先用模型级覆盖

换句话说：

- `capability_overrides` 适合处理“能力共性”
- `model_runtime_config / *_override` 适合处理“模型特例”

#### 5.2.3 能力级高级覆盖字段说明

前端当前的能力级高级覆盖，建议按下面的语义理解：

- `base_url`
  表示该能力调用时使用的基础地址
- `endpoint_path`
  表示该能力实际拼接的接口路径
- `adapter_type`
  表示最终交给哪一类适配器执行，例如 `litellm`、`native`、`openai_sdk`、`custom`
- `request_schema`
  表示“统一请求 DTO 应该如何转换成上游请求体”
- `response_schema`
  表示“上游返回结果应该如何转换回平台统一响应”
- `implementation_key`
  表示底层具体实现键，用于选择某个协议族里的细分实现
- `supports_multimodal_input`
  表示该能力是否支持图文音视频混合输入

其中最容易混淆的是下面三项：

- `adapter_type`
  解决“谁来调用”
- `request_schema`
  解决“怎么组请求”
- `response_schema`
  解决“怎么解响应”

#### 5.2.4 Request Schema 是什么

`Request Schema` 不是数据库表结构，也不是前端表单结构。

它表示的是：

**平台统一请求对象，如何被翻译成上游厂商实际请求体。**

例如平台统一的 `RerankRequest` 可能长这样：

```json
{
  "query": "什么是文本排序模型",
  "documents": [
    "文本排序模型广泛用于搜索引擎和推荐系统中",
    "量子计算是计算科学的一个前沿领域"
  ],
  "top_n": 2
}
```

如果 `request_schema = openai_rerank`，则适配器可能组装成：

```json
{
  "model": "qwen3-rerank",
  "query": "什么是文本排序模型",
  "documents": [
    "文本排序模型广泛用于搜索引擎和推荐系统中",
    "量子计算是计算科学的一个前沿领域"
  ],
  "top_n": 2
}
```

如果 `request_schema = dashscope_text_rerank_v1`，则适配器可能组装成：

```json
{
  "model": "gte-rerank-v2",
  "input": {
    "query": "什么是文本排序模型",
    "documents": [
      "文本排序模型广泛用于搜索引擎和推荐系统中",
      "量子计算是计算科学的一个前沿领域"
    ]
  },
  "parameters": {
    "top_n": 2,
    "return_documents": true
  }
}
```

也就是说：

**Request Schema 决定“平台统一请求”翻译成哪一种上游协议。**

#### 5.2.5 Response Schema 是什么

`Response Schema` 表示：

**上游厂商返回的数据，应该按哪一种格式解析并归一化为平台统一响应。**

例如平台统一的 `RerankResponse` 可以约定为：

```json
{
  "model": "xxx",
  "results": [
    {
      "index": 0,
      "score": 0.98,
      "document": {
        "text": "..."
      }
    }
  ],
  "usage": {},
  "raw_response": {}
}
```

而上游实际返回可能分别是：

1. OpenAI-compatible 风格
2. DashScope 原生文本 rerank 风格
3. DashScope 原生多模态 rerank 风格

这时：

- `response_schema = openai_rerank`
  表示用 OpenAI 风格解析器归一化结果
- `response_schema = dashscope_text_rerank_v1`
  表示用 DashScope 文本 rerank 解析器归一化结果
- `response_schema = dashscope_multimodal_rerank_v1`
  表示用 DashScope 多模态 rerank 解析器归一化结果

如果前端没有单独填写 `response_schema`，建议默认等于 `request_schema`。

#### 5.2.6 整个运行体系怎样配合代码

能力级高级覆盖不是“只配个 JSON 就自动魔法生效”，它必须和代码里的协议适配体系配合。

正式的运行链路应该是：

1. 前端保存 `capability_overrides`
2. 后端运行时解析出 `ResolvedEndpointProfile`
3. 根据 `adapter_type + request_schema + response_schema + implementation_key` 选择适配器
4. `Request Builder` 负责把统一请求转成上游请求
5. 调用上游接口
6. `Response Normalizer` 负责把上游响应转回平台统一响应

所以要特别明确：

**前端配置只是“选择策略”，真正统一入口的是后端适配器代码。**

前端字段和后端代码的关系可以理解成：

- 前端：声明这次应该走哪种协议族
- 后端：真正实现这套协议族的请求构建和响应归一

如果某个 `request_schema` 或 `response_schema` 只是配了名字，但后端没有对应实现，那它仍然不能真正工作。

#### 5.2.7 一个完整案例：Tongyi 的 Rerank 能力级覆盖

场景：

- 厂商：`Tongyi-Qianwen`
- 厂商默认 `base_url`：`https://dashscope.aliyuncs.com`
- Chat 和 Embedding 保持默认
- Rerank 能力整体先按 OpenAI-compatible 方式接到 `qwen3-rerank`

那么能力级高级覆盖可以配置为：

```json
{
  "rerank": {
    "base_url": "https://dashscope.aliyuncs.com",
    "endpoint_path": "/compatible-api/v1/reranks",
    "adapter_type": "native",
    "request_schema": "openai_rerank",
    "response_schema": "openai_rerank",
    "implementation_key": "openai_compatible_rerank"
  }
}
```

此时运行效果是：

- 普通聊天模型仍继承厂商默认配置
- embedding 仍继承厂商默认配置
- rerank 能力整体改走 `/compatible-api/v1/reranks`
- 后端选择 `openai_rerank` 这一套 builder / normalizer

如果后面只有 `gte-rerank-v2` 这个模型是特殊的，不走兼容接口，而走原生接口：

那就不要再改能力级高级覆盖，而应该在“手动添加模型”或模型编辑里，对这个模型单独配置模型级覆盖。

#### 5.2.8 一个完整案例：Tongyi 的模型级覆盖

例如手动添加 `gte-rerank-v2` 时：

- `adapter_override_type = native`
- `request_schema_override = dashscope_text_rerank_v1`
- `endpoint_path_override = /api/v1/services/rerank/text-rerank/text-rerank`
- `implementation_key_override = dashscope_text_rerank_v1`

这表示：

- `rerank` 能力默认继续按兼容协议走
- 但 `gte-rerank-v2` 这个模型是特例
- 它在最终运行时覆盖掉能力级配置，走自己的原生协议

这正是“三层配置模型”的意义：

- 厂商级负责默认
- 能力级负责共性差异
- 模型级负责个别特例

#### 5.2.9 多模态模型的归类原则

**多模态对话模型，主类型通常仍然归为 `chat`，而不是单独改成 `vision`。**

也就是说：

- GPT-4o、Gemini 多模态、Claude 视觉版、Qwen-VL 这类“既能聊天也能看图”的模型
- 在模型中心里，通常仍属于 `chat`
- 同时再声明它具备 `vision` 能力，或标记 `supports_multimodal_input = true`

建议按下面的规则建模：

1. 如果模型的主要用途是“对话生成”，即使支持图片输入，主类型仍设为 `chat`
2. 如果模型是“专用视觉理解入口”，不承担常规聊天生成，再考虑主类型设为 `vision`
3. 多模态 `rerank` 仍属于 `rerank`，不要因为能收图片或视频就改成 `vision`
4. `asr / tts / image / video` 也仍然按各自主能力归类

推荐这样理解当前数据结构：

- `model_type`
  表示主能力，例如 `chat`、`embedding`、`rerank`
- `capabilities`
  表示附加能力，例如一个多模态对话模型可以是 `["chat", "vision"]`
- `supports_multimodal_input`
  表示该能力是否支持 `text / image / audio / video` 等混合输入

案例 A：多模态聊天模型

```json
{
  "model_type": "chat",
  "capabilities": ["chat", "vision"],
  "supports_multimodal_input": true
}
```

案例 B：专用视觉理解模型

```json
{
  "model_type": "vision",
  "capabilities": ["vision"],
  "supports_multimodal_input": true
}
```

案例 C：多模态 Rerank

```json
{
  "model_type": "rerank",
  "capabilities": ["rerank"],
  "supports_multimodal_input": true
}
```

对 Genesis AI 当前项目，建议采用下面的约定：

1. `chat` 继续作为大模型对话的主类型
2. `vision` 保留，但主要用于“专用视觉能力”或“需要独立视觉路由”的模型
3. 多模态聊天模型优先建成 `chat + vision`
4. 前端“手动添加模型”时，如果模型既能聊天又能看图，优先选 `chat`
5. 前端应支持“主类型 + 附加能力”的表达方式，而不是只允许单一能力

### 5.4 默认适配策略的收敛方式

对于默认适配器选择，不建议按整个协议族一刀切。

更稳妥的方式是：

1. 大多数厂商继续继承 `provider_definition.adapter_type`
2. 仅对已知存在兼容差异的“厂商 + 能力”组合，做定向默认推导
3. 如果未来验证某个厂商能力稳定，再回收这些特例

例如当前阶段：

- `Tongyi-Qianwen + embedding`
  默认更适合走 `native`
- 其余大多数 `openai_compatible` 厂商
  仍可继续保持原默认策略

这样可以避免：

- 为了解决单个厂商问题，把所有 `openai_compatible` embedding 全部改成 native
- 也能减少“先 LiteLLM 失败，再 fallback 成功”的噪声链路

### 5.3 第三层：模型级最终路由覆盖

当同一能力下，不同模型依然存在协议差异、地址差异或请求体差异时，必须允许模型级覆盖。

典型例子：

- `qwen3-rerank` 走 OpenAI 风格 rerank
- `gte-rerank-v2` 走 DashScope 原生文本 rerank
- `qwen3-vl-rerank` 走 DashScope 原生多模态 rerank

模型级覆盖应被定义为：

**最终调用路由的唯一权威来源。**

## 6. 推荐的数据建模方式

### 6.1 Provider Definition 的职责

`model_provider_definitions`

职责应限定为：

- 厂商标识
- 展示名称
- 默认协议偏好
- 默认 `base_url`
- 默认 discovery 策略
- 理论支持能力列表
- UI 展示元数据

这里建议区分两类能力：

- `declared_capabilities`
- `runtime_supported_capabilities`

说明：

- `declared_capabilities` 表示厂商理论上支持哪些能力
- `runtime_supported_capabilities` 表示当前平台版本实际上已经实现了哪些能力

这样可以避免出现“前端显示支持 rerank，但后端并未打通”的语义错位。

### 6.2 Tenant Provider 的职责

`tenant_model_providers`

职责应限定为：

- 租户级凭证与连接配置
- 默认 `base_url`
- 少量高级覆盖项
- 模型同步状态

建议保留但弱化当前的 `capability_base_urls`：

- 可以保留
- 但语义应调整为“能力级高级覆盖”
- 不应作为标准配置主路径

建议补充或演进为：

- `capability_overrides: JSONB`

推荐结构如下：

```json
{
  "embedding": {
    "base_url": "https://example.com/compatible-mode/v1",
    "adapter_type": "litellm",
    "request_schema": "openai_embedding"
  },
  "rerank": {
    "base_url": "https://example.com",
    "endpoint_path": "/compatible-api/v1/reranks",
    "adapter_type": "native",
    "request_schema": "openai_rerank"
  }
}
```

### 6.3 Platform Model 的职责

`platform_models`

当前如果仅保存：

- `raw_model_name`
- `display_name`
- `model_type`
- `capabilities`

是不够的。

建议补充“调用画像”字段，使平台模型不仅是“目录项”，也是“默认路由模板”。

建议新增字段：

- `implementation_key`
- `request_schema`
- `default_endpoint_path`
- `supports_multimodal_input`
- `supports_file_input`
- `supports_stream`
- `metadata_info.discovery_payload`

推荐语义：

- `implementation_key`：使用哪一种实现逻辑
- `request_schema`：请求体采用哪种协议模板
- `default_endpoint_path`：默认 endpoint 路径
- `supports_multimodal_input`：是否支持文本/图片/视频混合输入

### 6.4 Tenant Model 的职责

`tenant_models`

租户模型不应只承担“平台模型绑定关系”，还应允许对单个模型做最终调用覆盖。

建议新增或强化以下字段：

- `adapter_override_type`
- `implementation_key_override`
- `request_schema_override`
- `endpoint_path_override`
- `model_runtime_config`

推荐 `model_runtime_config` 示例：

```json
{
  "base_url_override": "https://dashscope.aliyuncs.com",
  "endpoint_path_override": "/api/v1/services/rerank/text-rerank/text-rerank",
  "request_schema_override": "dashscope_text_rerank_v1",
  "supports_multimodal_input": false
}
```

### 6.5 Model Adapter Binding 的职责

`model_adapter_bindings`

当前这个表方向是对的，建议正式承担“能力/模型路由覆盖策略”的职责。

建议用途：

- 某厂商的 `chat` 默认走 LiteLLM
- 某厂商的 `embedding` 走 LiteLLM OpenAI-compatible
- 某厂商的 `rerank` 默认走 Native Adapter
- 某个租户实例覆盖为自定义实现

推荐优先级匹配范围：

1. `tenant_model_id`
2. `tenant_provider_id + capability_type`
3. `provider_definition_id + capability_type`
4. 全局 `capability_type`

如果暂时不想新增 `tenant_model_id` 字段，也可以先通过 `tenant_model.model_runtime_config` 承担模型级覆盖，后续再统一收口。

---

## 7. 推荐的运行时解析模型

### 7.1 统一的解析结果对象

运行时不要直接拿数据库字段就开始拼请求，应先统一解析出一个中间对象：

`ResolvedEndpointProfile`

推荐结构：

```json
{
  "capability_type": "rerank",
  "adapter_type": "native",
  "implementation_key": "dashscope_text_rerank",
  "base_url": "https://dashscope.aliyuncs.com",
  "endpoint_path": "/api/v1/services/rerank/text-rerank/text-rerank",
  "request_schema": "dashscope_text_rerank_v1",
  "supports_multimodal_input": false,
  "extra_headers": {},
  "request_defaults": {}
}
```

### 7.2 运行时解析流程

建议统一流程如下：

1. 加载 `provider_definition`
2. 加载 `tenant_provider`
3. 加载 `platform_model`
4. 加载 `tenant_model`
5. 加载适用的 `adapter_binding`
6. 生成 `ResolvedEndpointProfile`
7. 根据 `request_schema` 组装请求
8. 根据 `adapter_type` 分发到 LiteLLM / Native / Custom
9. 将上游响应归一成平台统一响应

### 7.3 这样做的好处

- 配置层与执行层分离
- 不会在业务调用代码里堆大量厂商 `if/else`
- 同一个能力可以平滑接入多个协议
- 手动添加模型不会变成“只加名字，不可调用”

## 9. LiteLLM 的正式定位

### 9.1 基本原则

LiteLLM 不是“所有能力强行统一的唯一入口”，而应是：

**能稳定走 LiteLLM 的能力优先走 LiteLLM，LiteLLM 不支持或不稳定的能力走 Native Adapter。**

### 9.2 推荐使用策略

- `chat`：优先 LiteLLM
- `embedding`：若厂商可稳定兼容 OpenAI embedding，则优先 LiteLLM
- `rerank`：优先 Native Adapter，LiteLLM 仅在明确支持的 provider 上使用
- `asr`：若 LiteLLM 对该 provider 已支持标准音频转写，则可走 LiteLLM
- `tts`：若 LiteLLM 对该 provider 已支持标准语音合成，则可走 LiteLLM
- 多模态特殊能力：优先 Native Adapter

### 9.3 对 Tongyi-Qianwen 的结论

Tongyi-Qianwen 当前更适合采用“双轨方案”：

- `chat` / `embedding`：优先走 OpenAI-compatible + LiteLLM
- `rerank`：走 Native Adapter 或 Custom Adapter
- 多模态 Rerank：走独立 Native Adapter

原因不是 Tongyi 不支持，而是：

- 同一厂商内不同 rerank 模型协议不统一
- LiteLLM 当前版本对 DashScope provider 的 endpoint 支持并不覆盖所有能力

因此正式设计不应把 Tongyi 简化成“一个 provider + 一种协议”。

## 10. 统一入口下的协议适配与响应归一

### 10.1 统一入口不等于底层协议统一

模型中心必须明确一个核心原则：

**平台统一入口，并不意味着所有上游模型都必须使用同一种协议。**

统一入口真正表示的是：

- 上层业务调用方式统一
- 平台内部负责吸收上游差异
- 最终返回统一响应格式

因此，当不同模型虽然同属 `rerank` 能力，但它们的：

- endpoint 不同
- 请求结构不同
- 返回结构不同

平台内部就必须存在一层协议适配与归一化逻辑。

这个结论不是设计缺陷，而是模型平台的正常职责。

### 10.2 平台内部必须拆成两层

建议平台在每个复杂能力下，都显式拆成两类组件：

1. `Request Builder`
2. `Response Normalizer`

两层职责如下：

#### Request Builder

负责将平台统一请求 DTO 转换为上游厂商协议请求。

例如：

- `openai_rerank_builder`
- `dashscope_text_rerank_builder`
- `dashscope_multimodal_rerank_builder`

#### Response Normalizer

负责将厂商原始响应转换为平台统一响应格式。

例如：

- `openai_rerank_normalizer`
- `dashscope_text_rerank_normalizer`
- `dashscope_multimodal_rerank_normalizer`

### 10.3 为什么不能只靠 URL 配置解决

如果只有 endpoint 不同，而请求与响应结构完全一样，那么配置 URL 的确足够。

但在 Tongyi 这类场景里，问题不只在 URL：

- 有的模型请求体是 OpenAI 风格
- 有的模型请求体是 DashScope 原生风格
- 有的模型输入是纯文本文档数组
- 有的模型输入是文本、图片、视频混合结构
- 有的模型返回字段名、得分结构、文档回传结构并不一致

因此，仅靠：

- `base_url`
- `endpoint_path`

无法实现真正的统一入口调用。

必须再加：

- `request_schema`
- `response_schema`
- 对应的 Builder / Normalizer

### 10.4 推荐的协议族实现方式

不要采用“一个模型对应一套代码”的方式，否则维护成本会迅速失控。

正确做法是：

**按协议族实现，而不是按模型名实现。**

例如 Tongyi Rerank 至少可以分成三类协议族：

1. `openai_rerank`
2. `dashscope_text_rerank_v1`
3. `dashscope_multimodal_rerank_v1`

这样新增模型时，只需把模型映射到某个协议族，而不是重复开发新代码。

### 10.5 推荐的统一能力处理链路

以 `rerank` 为例，推荐运行时处理顺序为：

1. 上层业务调用统一 `rerank(request)`
2. 根据模型配置解析出 `ResolvedEndpointProfile`
3. 根据 `request_schema` 选择对应的 `Request Builder`
4. 组装上游请求
5. 根据 `adapter_type` 发起调用
6. 根据 `response_schema` 选择对应的 `Response Normalizer`
7. 归一为平台统一响应
8. 返回给调用方

这个链路可以保证：

- 业务侧永远使用统一接口
- 平台内部可兼容多种协议
- 特殊模型只影响自己的协议适配，不污染全局调用链

### 10.6 推荐的统一 Rerank 响应格式

建议平台内部固定统一响应结构：

```json
{
  "model": "xxx",
  "results": [
    {
      "index": 0,
      "score": 0.95,
      "document": {
        "text": "..."
      }
    }
  ],
  "usage": {},
  "raw_response": {}
}
```

字段建议如下：

- `model`：最终使用的模型名
- `results[].index`：原始候选文档索引
- `results[].score`：统一归一后的分数
- `results[].document`：标准化后的文档内容
- `usage`：统一资源消耗信息
- `raw_response`：保留上游原始响应，便于排查问题

### 10.7 推荐的统一文档输入结构

为了兼容未来多模态 rerank，建议统一 `documents` 输入从第一版开始就支持：

- `string`
- `{ "text": "..." }`
- `{ "image": "..." }`
- `{ "video": "..." }`

平台内部由 Builder 决定：

- 哪些协议只接受纯文本
- 哪些协议支持多模态输入
- 不支持的协议在请求前直接报错

这样可以避免未来再次推翻 `rerank` 接口。

### 10.8 哪些情况下必须写自定义适配器

满足以下任一条件时，建议不要强行统一到底层通用实现，而应编写 Native / Custom Adapter：

1. endpoint 不同
2. 请求结构不同
3. 返回结构不同
4. 输入类型不同
5. LiteLLM 当前版本未稳定支持

Tongyi 的部分 Rerank 模型明显满足以上条件，因此推荐路线是：

- 对上层统一 `rerank` 接口
- 对平台底层编写协议族级适配器

### 10.9 正式设计结论

统一入口的目标应该是：

- 对业务方统一
- 对前端统一
- 对平台外部语义统一

而不是要求：

- 所有上游厂商都完全同协议
- 所有模型都零自定义适配

因此，对特殊模型进行 Builder / Normalizer 开发是模型平台的正常职责，应当被纳入正式架构，而不是视为临时补丁。

## 11. Tongyi-Qianwen 的推荐落地方式

### 10.1 默认接入方式

Tongyi-Qianwen 的默认接入策略推荐为：

- Provider 默认 `base_url`：`https://dashscope.aliyuncs.com/compatible-mode/v1`
- `chat` 默认：`litellm + openai_compatible`
- `embedding` 默认：`litellm + openai_compatible`

这样可以让大多数用户只配置一次 `base_url` 即可完成：

- 文本对话
- 多模态对话
- 向量化

### 10.2 特殊能力处理

当接入 rerank 时：

- 不再默认继承 provider 主协议
- 而是走能力级或模型级覆盖

例如：

- `qwen3-rerank`
  - `adapter_type = native`
  - `request_schema = openai_rerank`
  - `endpoint_path = /compatible-api/v1/reranks`

- `gte-rerank-v2`
  - `adapter_type = native`
  - `request_schema = dashscope_text_rerank_v1`
  - `endpoint_path = /api/v1/services/rerank/text-rerank/text-rerank`

- `qwen3-vl-rerank`
  - `adapter_type = native`
  - `request_schema = dashscope_multimodal_rerank_v1`
  - `endpoint_path = /api/v1/services/rerank/text-rerank/text-rerank`
  - `supports_multimodal_input = true`

### 10.3 手动添加模型的正式定位

手动添加模型应保留，但定位应从“补录模型名”升级为：

**为特殊模型创建高级路由配置。**

这意味着手动添加模型时，除基础信息外，应允许高级用户填写：

- `adapter_type`
- `implementation_key`
- `request_schema`
- `endpoint_path`
- `extra_config`

## 12. 前端设计建议

### 11.1 总目标

前端必须同时满足两类需求：

1. 普通用户快速完成厂商接入
2. 高级用户优雅配置特殊能力和特殊模型

因此推荐采用“基础模式 + 高级模式”设计。

### 11.2 基础模式

默认只展示：

- 厂商名称
- API Key
- Base URL
- API Version
- Region
- 测试连接
- 同步模型

基础模式不展示：

- Embedding URL
- Rerank URL
- 各种特殊 endpoint path

原因：

- 大多数用户不需要这些
- 这些内容会显著增加理解成本

### 11.3 高级模式

高级模式折叠展开，按需显示：

- 能力级覆盖配置
- 自定义请求头
- 自定义默认参数
- 请求协议覆盖
- 模型级路由覆盖入口

### 11.4 手动添加模型弹窗

推荐分成两部分：

基础字段：

- 模型名
- 显示名
- 能力类型
- 是否启用

高级字段：

- 调用方式
- 协议模板
- endpoint path
- 是否多模态
- 额外参数

这样可以保证：

- 普通模型录入足够轻
- 特殊模型也有能力被优雅配置

## 13. 对现有代码的改造建议

### 12.1 第一阶段：先不推翻现有结构

建议先在现有结构基础上增强，而不是大改表结构。

第一阶段建议：

1. 保留现有 `provider_definition / tenant_provider / platform_model / tenant_model`
2. 保留 `capability_base_urls`
3. 新增 `capability_overrides`
4. 在 `tenant_model` 增加 `model_runtime_config`
5. 运行时新增 `ResolvedEndpointProfile` 解析层

### 12.2 第二阶段：补齐统一调用能力

建议按顺序补：

1. `rerank` 统一入口
2. `asr` 统一入口
3. `tts` 统一入口
4. 多模态 rerank 统一请求结构

### 12.3 第三阶段：正式收口适配器路由

将分散在 service 内部的厂商特判逐步收敛为：

- discovery strategy
- adapter binding
- request schema builder
- response normalizer

最终实现：

- 发现模型与调用模型彻底分离
- provider 特判变成策略注册
- 特殊模型不再依赖零散 `if/else`

## 14. 推荐实施顺序

### 13.1 近期必须完成

1. 明确 `provider`、`capability`、`model` 三层职责
2. 给 `rerank` 增加统一 API 入口
3. 给 `tenant_model` 增加模型级运行时覆盖字段
4. 前端将能力级 URL 配置移入“高级配置”

### 13.2 中期建议完成

1. 引入 `ResolvedEndpointProfile`
2. 将 Tongyi rerank 按不同模型拆分为不同 `implementation_key`
3. 补齐 ASR / TTS 的统一调用接口
4. 调整手动添加模型为“手动添加模型路由”

### 13.3 后续优化

1. 增加 discovery strategy 注册机制
2. 增加 request schema 注册机制
3. 增加 runtime supported capability 自动探测能力
4. 增加模型路由可视化展示

## 15. 最终结论

模型中心的最佳方案不是：

- 所有厂商都配置一堆能力 URL
- 所有能力都强行走 LiteLLM
- 所有特例都靠服务层 `if/else`

而是：

**默认统一，逐级覆盖，运行时解析，接口归一。**

正式表达为：

- 厂商级统一配置负责“简单”
- 能力级高级覆盖负责“适配”
- 模型级最终路由负责“兜底”
- 上层统一接口负责“稳定”

这样设计后，模型中心就能同时满足：

- 普通厂商一键接入
- 特殊厂商优雅兼容
- 特殊模型可手工精细配置
- 前端简单可用
- 后端结构长期可维护

---

## 16. 建议作为后续开发的正式规则

建议将以下规则作为模型中心后续开发约束：

1. 默认情况下，厂商仅需配置一个 `base_url`
2. 能力级地址覆盖仅作为高级配置出现
3. 特殊模型必须允许单独配置调用路由
4. 统一调用接口优先于统一底层实现
5. LiteLLM 是优先适配器，但不是唯一适配器
6. 同厂商不同模型协议不一致时，以模型级配置为准

本文档作为当前阶段模型中心统一配置与特殊模型路由设计的正式基线。
