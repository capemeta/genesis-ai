# 认证系统详细说明

## 🔥 双模式认证支持

### 设计理念

**后端支持双模式，前端选择单模式**

为了支持多端（Web、移动端、API），后端同时支持两种认证方式：

1. **Cookie 模式**（Web 端推荐）
   - 使用 HttpOnly Cookie 存储 token
   - 防止 XSS 攻击（JavaScript 无法读取）
   - SameSite=Lax 防止 CSRF 攻击
   - 浏览器自动携带，前端无需处理

2. **Bearer Token 模式**（移动端/API 推荐）
   - 使用 Authorization Header 传递 token
   - 前端存储在 localStorage/SecureStorage
   - 更灵活，适合移动端和第三方 API

### 当前使用模式

| 客户端 | 认证模式 | 存储方式 | 安全性 |
|--------|---------|---------|--------|
| **Web 前端（当前）** | Cookie 模式 | HttpOnly Cookie | ⭐⭐⭐⭐⭐ 最安全 |
| 移动端 App（未来） | Bearer Token 模式 | SecureStorage | ⭐⭐⭐⭐ 安全 |
| 第三方 API（未来） | Bearer Token 模式 | 服务端存储 | ⭐⭐⭐⭐ 安全 |

### 实现机制

#### 后端实现（同时支持两种模式）

**登录接口**：
```python
# 同时设置 Cookie 和返回 token
response.set_cookie(
    key="access_token",
    value=session_data["access_token"],
    httponly=True,      # 防止 XSS
    samesite="lax",     # 防止 CSRF
    secure=is_https,    # HTTPS 环境启用
)

return Token(**session_data)  # 同时返回 token 到响应体
```

**认证中间件**：
```python
# 优先从 Cookie 读取，其次从 Header 读取
session_id = request.cookies.get("access_token")
if not session_id:
    auth_header = request.headers.get("Authorization")
    if auth_header and auth_header.startswith("Bearer "):
        session_id = auth_header[7:]
```

**登出接口**：
```python
# 同时清除 Cookie 和撤销 session
response.delete_cookie(key="access_token")
await session_service.revoke_session(session_id)
```

#### Web 前端实现（Cookie 模式）

**登录**：
```typescript
// 调用登录 API，后端自动设置 Cookie
const response = await login(loginData)

// 不存储 token 到 localStorage
// 浏览器会自动携带 Cookie
```

**请求**：
```typescript
// Axios 配置
const axiosInstance = axios.create({
  withCredentials: true,  // 自动携带 Cookie
})

// 不需要添加 Authorization Header
// 浏览器会自动携带 Cookie
```

**登出**：
```typescript
// 调用登出 API，后端自动清除 Cookie
await logout()

// 不需要清除 localStorage
```

#### 移动端实现（Bearer Token 模式，未来）

**登录**：
```typescript
// 调用登录 API
const response = await login(loginData)

// 存储 token 到 SecureStorage
await SecureStorage.setItem('access_token', response.access_token)
await SecureStorage.setItem('refresh_token', response.refresh_token)
```

**请求**：
```typescript
// 从 SecureStorage 读取 token
const token = await SecureStorage.getItem('access_token')

// 添加到 Authorization Header
headers: {
  'Authorization': `Bearer ${token}`
}
```

**登出**：
```typescript
// 调用登出 API
await logout()

// 清除 SecureStorage
await SecureStorage.removeItem('access_token')
await SecureStorage.removeItem('refresh_token')
```

### 安全性对比

| 特性 | Cookie 模式 | Bearer Token 模式 |
|------|------------|------------------|
| **XSS 防护** | ✅ HttpOnly 完全防护 | ⚠️ 需要防范 XSS |
| **CSRF 防护** | ✅ SameSite 防护 | ✅ 不受 CSRF 影响 |
| **中间人攻击** | ✅ HTTPS + Secure | ✅ HTTPS 保护 |
| **适用场景** | Web 浏览器 | 移动端、API |
| **实现复杂度** | 简单（浏览器自动处理） | 中等（需要手动管理） |
| **跨域支持** | 需要 CORS 配置 | 简单 |

### 切换模式

如果需要切换认证模式：

**Web 前端切换到 Bearer Token 模式**：
1. 登录时存储 token 到 localStorage
2. 请求时添加 Authorization Header
3. 登出时清除 localStorage

**移动端使用 Bearer Token 模式**：
1. 使用 SecureStorage 存储 token
2. 请求时添加 Authorization Header
3. 登出时清除 SecureStorage

## 系统架构

### 核心设计理念

**纯 Redis Session 方案**：
- Session ID 作为 token（256 位随机字符串）
- 所有 session 数据存储在 Redis 中
- 不使用 JWT，避免 token 无法撤销的问题
- 支持即时撤销、会话管理

### 技术栈

- **后端框架**: FastAPI
- **Session 存储**: Redis（使用连接池）
- **密码哈希**: bcrypt (passlib)
- **随机生成**: secrets.token_urlsafe()
- **数据库**: PostgreSQL (用户数据)

## 前端认证集成

### 🔥 当前使用：Cookie 认证模式

Web 前端使用 **Cookie 模式**，这是最安全的 Web 认证方式。

#### 核心特性

1. **HttpOnly Cookie**
   - 后端设置 HttpOnly Cookie
   - JavaScript 无法读取，防止 XSS 攻击
   - 浏览器自动携带，前端无需处理

2. **SameSite 保护**
   - SameSite=Lax 防止 CSRF 攻击
   - 允许顶级导航携带 Cookie
   - 阻止跨站请求携带 Cookie

3. **自动过期管理**
   - 浏览器自动处理 Cookie 过期
   - 不记住我：Access 30分钟，Refresh 3天
   - 记住我：Access 2小时，Refresh 30天

4. **无需前端存储**
   - 不使用 localStorage（避免 XSS 风险）
   - 不使用 sessionStorage
   - 完全依赖浏览器 Cookie 机制

#### 工作流程

**登录流程**：
```typescript
// 1. 用户输入凭证
const loginData = {
  username: 'admin',
  password: 'password',
  remember_me: true,
}

// 2. 调用登录 API
const response = await login(loginData)

// 3. 后端自动设置 HttpOnly Cookie
// Set-Cookie: access_token=xxx; HttpOnly; SameSite=Lax; Path=/

// 4. 前端不存储 token（Cookie 模式）
// 浏览器会自动管理 Cookie

// 5. 获取用户信息
const user = await getCurrentUser()

// 6. 更新认证状态
genesisAuth.login(user)
```

**请求流程**：
```typescript
// Axios 配置
const axiosInstance = axios.create({
  withCredentials: true,  // 🔥 关键：自动携带 Cookie
})

// 发起请求
await axiosInstance.get('/api/v1/users')

// 浏览器自动携带 Cookie：
// Cookie: access_token=xxx

// 后端从 Cookie 读取 token 并验证
```

**登出流程**：
```typescript
// 1. 调用登出 API
await logout()

// 2. 后端清除 Cookie
// Set-Cookie: access_token=; Max-Age=0

// 3. 后端撤销 session
await session_service.revoke_session(session_id)

// 4. 前端清除认证状态
genesisAuth.logout()
```

#### 与 localStorage 模式的对比

| 特性 | Cookie 模式（当前） | localStorage 模式（旧） |
|------|-------------------|----------------------|
| **XSS 防护** | ✅ HttpOnly 完全防护 | ❌ JavaScript 可读取 |
| **CSRF 防护** | ✅ SameSite 防护 | ✅ 需手动发送 |
| **实现复杂度** | ✅ 简单（浏览器自动） | ⚠️ 需手动管理 |
| **安全性** | ⭐⭐⭐⭐⭐ 最高 | ⭐⭐⭐ 中等 |
| **适用场景** | Web 浏览器 | 移动端、API |

### 核心特性（通用）

前端认证系统基于 Refine 框架，与后端 Session 机制无缝集成。

#### 1. "记住我"功能（前后端协同）

**设计理念**：
- 前端传递 `remember_me` 参数给后端
- 后端根据参数设置不同过期时间的 Cookie
- 通过 Cookie 过期时间控制会话长度

**实现机制**：
- **勾选"记住我"**：
  - 前端：传递 `remember_me: true` 给后端
  - 后端：返回长期 token（Access: 2小时，Refresh: 30天）
  - 后端：设置长期 Cookie（Max-Age: 30天）
  
- **不勾选"记住我"**：
  - 前端：传递 `remember_me: false` 给后端
  - 后端：返回短期 token（Access: 30分钟，Refresh: 3天）
  - 后端：设置短期 Cookie（Max-Age: 3天）

**工作流程**：
1. 用户登录时选择是否"记住我"
2. 前端调用登录接口，传递 `remember_me` 参数
3. 后端根据参数设置不同过期时间的 Cookie
4. 浏览器自动管理 Cookie 过期
5. Cookie 过期后，用户需要重新登录

**优势**：
- ✅ **安全可控**：通过 Cookie 过期时间控制，无法被前端篡改
- ✅ **滑动过期**：持续使用时自动续期（Token Rotation）
  - 不记住我：每3天内访问一次，可以无限期使用
  - 记住我：每30天内访问一次，可以无限期使用
  - 只有连续不访问（3天或30天），才会过期
- ✅ **浏览器原生支持**：利用浏览器的 Cookie 机制
- ✅ **多端统一**：支持 Web/App（都使用过期时间控制）

**关于"无限期使用"的说明**：
- 这是**滑动过期**的特性，不是 bug
- 符合业界主流做法（GitHub, Slack, Notion 等都是这样）
- 每次刷新 token 时，新 token 有完整的有效期（不是剩余时间）
- 安全性通过其他措施保障：
  - Token 轮换（旧 token 立即失效）
  - 重放检测（检测异常使用）
  - 会话管理（用户可查看和撤销所有设备）
  - 敏感操作二次认证（修改密码等）

**安全考虑**：
- Cookie 过期时间由后端控制，前端无法篡改
- HttpOnly Cookie 防止 XSS 攻击
- 生产环境必须使用 HTTPS
- 建议配置 Content Security Policy (CSP)

#### 2. 统一错误处理

**错误类型**：
- **401 未授权**：Cookie 无效或过期，跳转登录页
- **403 禁止访问**：权限不足提示
- **其他错误**：友好提示

**实现**：
```typescript
axiosInstance.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      // Cookie 无效，跳转登录
      window.location.href = '/sign-in'
    }
    return Promise.reject(error)
  }
)
```

#### 3. 验证码支持

**触发条件**：
- 登录失败 3 次后显示验证码
- 自动检测是否需要验证码
- 支持刷新验证码

const axiosInstance = axios.create({

#### 认证状态管理（genesis-auth-store.ts）

**Cookie 模式实现**：
```typescript
export const useGenesisAuthStore = create<GenesisAuthState>((set) => ({
  user: null,
  isAuthenticated: false,

  login: (user) => {
    // 🔥 不存储 token，依赖 Cookie
    set({ user, isAuthenticated: true })
  },

  logout: () => {
    // 🔥 不清除 token，后端已清除 Cookie
    set({ user: null, isAuthenticated: false })
  },

  checkAuth: async () => {
    // 🔥 通过 API 调用检查登录状态
    try {
      const user = await getCurrentUser()
      set({ user, isAuthenticated: true })
      return true
    } catch {
      set({ user: null, isAuthenticated: false })
      return false
    }
  },
}))
```

#### 认证初始化（use-auth-init.ts）

**Cookie 模式实现**：
```typescript
export function useAuthInit() {
  const { setUser, setLoading, logout } = useGenesisAuthStore()

  useEffect(() => {
    const initAuth = async () => {
      setLoading(true)
      try {
        // 🔥 通过 API 调用检查登录状态
        // 浏览器会自动携带 Cookie
        const user = await getCurrentUser()
        setUser(user)
      } catch {
        // Cookie 无效或过期
        logout()
      } finally {
        setLoading(false)
      }
    }
    initAuth()
  }, [])
}
```

### 前端与后端交互（Cookie 模式）

#### 登录流程

```
用户输入凭证 + 选择"记住我"
    ↓
前端：调用 POST /auth/login（传递 remember_me 参数）
    ↓
后端：验证凭证，根据 remember_me 设置过期时间
    ↓
后端：设置 HttpOnly Cookie（不同的过期时间）
    Set-Cookie: access_token=xxx; HttpOnly; SameSite=Lax; Max-Age=7200
    Set-Cookie: refresh_token=yyy; HttpOnly; SameSite=Lax; Max-Age=2592000; Path=/api/v1/auth/refresh
    ↓
后端：返回 token 到响应体（兼容其他客户端）
    ↓
前端：不存储 token（Cookie 模式）
    ↓
前端：调用 GET /users/me 获取用户信息
    （浏览器自动携带 Cookie）
    ↓
前端：更新认证状态
    ↓
跳转到首页
```

**注意**：
- 前端传递 `remember_me` 参数给后端
- 后端根据参数设置不同过期时间的 Cookie：
  - 不记住我：Access 30分钟，Refresh 3天
  - 记住我：Access 2小时，Refresh 30天
- 前端不存储 token，完全依赖 Cookie
- 浏览器自动管理 Cookie 过期

#### 登出流程

```
用户点击登出
    ↓
前端：调用 POST /auth/logout
    （浏览器自动携带 Cookie）
    ↓
后端：从 Cookie 读取 access_token
    ↓
后端：验证 session 所有权
    ↓
后端：撤销 access session 和 refresh session
    ↓
后端：清除 Cookie
    Set-Cookie: access_token=; Max-Age=0
    Set-Cookie: refresh_token=; Max-Age=0; Path=/api/v1/auth/refresh
    ↓
前端：清除认证状态
    ↓
跳转到登录页
```

**安全措施**：
- 后端验证 session 属于当前用户
- 同时撤销 access 和 refresh session
- 即使 access session 过期，也能撤销 refresh session
- 后端清除 Cookie，浏览器自动删除
- 前端清除认证状态

#### Token 刷新流程（后端自动处理）

```
Access Token 过期（30分钟或2小时）
    ↓
前端：发起 API 请求
    （浏览器自动携带 Cookie，包含过期的 access_token）
    ↓
后端：检测 access_token 过期
    ↓
后端：从 Cookie 读取 refresh_token
    ↓
后端：验证 refresh_token
    ↓
后端：创建新的 session 对
    ↓
后端：更新 Cookie（新的 access_token 和 refresh_token）
    Set-Cookie: access_token=new_xxx; HttpOnly; SameSite=Lax; Max-Age=7200
    Set-Cookie: refresh_token=new_yyy; HttpOnly; SameSite=Lax; Max-Age=2592000; Path=/api/v1/auth/refresh
    ↓
后端：返回 API 响应
    ↓
前端：无感知（浏览器自动更新 Cookie）
```

**注意**：
- 🔥 **前端无需手动刷新 token**
- 后端自动检测 token 过期并刷新
- 浏览器自动更新 Cookie
- 用户完全无感知
- 旧的 refresh token 立即失效（轮换机制）

### 前端最佳实践（Cookie 模式）

#### 1. 使用 HTTPS

生产环境必须使用 HTTPS，Cookie 的 `Secure` 标志才会生效。

```python
# 后端自动检测
is_secure = request.url.scheme == "https"
response.set_cookie(
    key="access_token",
    secure=is_secure,  # HTTPS 环境下启用
)
```

#### 2. 配置 CORS

Cookie 模式需要正确配置 CORS：

```python
# 后端 CORS 配置
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://your-domain.com"],  # 不能使用 *
    allow_credentials=True,  # 🔥 关键：允许携带 Cookie
    allow_methods=["*"],
    allow_headers=["*"],
)
```

```typescript
// 前端 Axios 配置
const axiosInstance = axios.create({
  withCredentials: true,  // 🔥 关键：携带 Cookie
})
```

#### 3. 配置 CSP

在 `index.html` 中配置 Content Security Policy：

```html
<meta http-equiv="Content-Security-Policy" 
      content="default-src 'self'; 
               script-src 'self' 'unsafe-inline'; 
               style-src 'self' 'unsafe-inline';">
```

#### 4. 提示用户风险

在"记住我"旁边添加说明：
- "30天内自动登录"（记住我）
- "3天内自动登录"（不记住我）
- "请勿在公共设备上勾选"

#### 5. 定期清理

提供"登出所有设备"功能，让用户主动清理所有 session。

### 前端常见问题（Cookie 模式）

#### 1. Cookie 未携带

**原因**：
- 未设置 `withCredentials: true`
- CORS 配置错误
- Cookie 域名不匹配

**解决方案**：
```typescript
// Axios 配置
const axiosInstance = axios.create({
  withCredentials: true,  // 必须设置
})
```

```python
# 后端 CORS 配置
allow_credentials=True  # 必须设置
allow_origins=["https://your-domain.com"]  # 不能使用 *
```

#### 2. 跨域 Cookie 问题

**问题**：开发环境前后端不同域名，Cookie 无法携带

**解决方案**：
- 使用代理（推荐）
- 或者前后端使用相同域名的不同端口

```typescript
// vite.config.ts
export default defineConfig({
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:8200',
        changeOrigin: true,
      },
    },
  },
})
```

#### 3. 新标签页登录状态

**Cookie 模式的优势**：
- Cookie 自动在所有标签页共享
- 用户在任何标签页登录，其他标签页立即可用
- 无需额外的跨标签页通信

#### 4. 关闭浏览器后的行为

**说明**：
- Cookie 的 `Max-Age` 控制过期时间
- 不记住我：3天内有效（滑动过期）
- 记住我：30天内有效（滑动过期）
- 关闭浏览器不会清除 Cookie（除非是 Session Cookie）
- 这是业界主流做法（GitHub, Slack, Notion 等）

#### 5. XSS 攻击防护

**Cookie 模式的优势**：
- HttpOnly Cookie 完全防止 XSS 窃取 token
- JavaScript 无法读取 Cookie
- 即使网站存在 XSS 漏洞，token 也无法被窃取

**额外防护措施**：
- 使用 CSP 限制脚本来源
- 不使用 `dangerouslySetInnerHTML`
- 不执行用户输入的代码
- 使用 HTTPS
- 定期更新依赖

## 退出登录流程

### 流程概述

1. **验证当前用户**（通过 get_current_user）
2. **提取 Access Session ID**（从 Authorization Header）
3. **读取 Access Session**（获取关联的 Refresh Session ID）
4. **撤销 Access Session**（从 Redis 删除）
5. **撤销 Refresh Session**（从 Redis 删除，防止继续使用）
6. **记录审计日志**

### 🔒 安全改进（已实现）

**问题**：如果 Access Session 已过期，无法读取关联的 Refresh Session ID

**解决方案（三层回退机制）**：
1. **方法 1**：使用 `read_session()` 读取（可能已过期返回 None）
2. **方法 2**：从 Redis 直接读取原始数据（即使已过期）
3. **方法 3**：如果仍无法找到，撤销用户所有 session（兜底方案）

**Session 所有权验证**：
- 验证 session 的 `user_id` 与当前用户一致
- 防止用户使用他人的 session 登出
- 验证失败返回 403 Forbidden

**安全保障**：
- 确保即使 access session 过期，也能正确撤销 refresh session
- 防止用户登出后，攻击者仍可使用 refresh token
- 即使撤销失败，前端也会清除 token（双重保障）

### 登出所有设备

**功能**：撤销用户的所有 Session

**使用场景**：
- 密码被盗用
- 设备丢失
- 安全审计

## Token 刷新流程

### 流程概述

1. **验证 Refresh Session**
2. **检查 Token 重放**（已使用的 token 会被标记）
3. **提取用户信息**（user_id, tenant_id, scope, token_family_id）
4. **标记旧 Token 为已使用**（存储 7 天）
5. **撤销旧 Session**（Access + Refresh）
6. **创建新 Session 对**（继承 token_family_id，**使用原来的过期时间配置**）
7. **返回新 Token**（新的 Access Token + 新的 Refresh Token）

### 🔥 关键特性：滑动过期（Sliding Expiration）

**核心机制**：
- 每次刷新时，**旧的 Refresh Token 失效**
- 创建**新的 Refresh Token**，有**完整的有效期**（3天或30天）
- 新的 Access Token 也有完整的有效期（30分钟或2小时）

**实际效果**：
```
不记住我（Refresh Token: 3天）：
第1天 09:00 - 用户登录
  → Refresh Token 有效期到第4天 09:00

第3天 08:00 - Access Token 过期，自动刷新
  → 旧 Refresh Token 失效
  → 新 Refresh Token 有效期到第6天 08:00 ✅

第5天 08:00 - 再次刷新
  → 新 Refresh Token 有效期到第8天 08:00 ✅

只要持续使用（每3天内至少访问一次），可以无限期登录！
只有连续3天不访问，才会过期需要重新登录。

记住我（Refresh Token: 30天）：
同理，只要每30天内至少访问一次，可以无限期登录！
```

**这就是"滑动过期"的含义**：
- ✅ 持续使用：过期时间不断向后推移（滑动）
- ❌ 长期不用：最终会过期（安全性）

**与"绝对过期"的对比**：
```
绝对过期（如果实现的话）：
第1天 09:00 - 用户登录
  → Refresh Token 有效期到第8天 09:00
  → 绝对过期时间：第31天 09:00（固定，不会改变）

第3天 08:00 - 刷新
  → 新 Refresh Token 有效期到第14天 08:00
  → 绝对过期时间：第31天 09:00（仍然不变）

第31天 09:00 - 到达绝对过期时间
  → 即使持续使用，也必须重新登录 ❌

当前实现：没有绝对过期，只有滑动过期
```

### 🔒 Refresh Token 轮换（已实现）

**为什么需要轮换？**

如果 refresh token 泄露，攻击者可以无限次刷新获取新的 access token。通过轮换，每次刷新后旧的 refresh token 立即失效，限制了攻击窗口。

**实现机制**：
- 每次刷新后，旧的 refresh token 被标记为已使用
- 标记存储在 Redis 中（key: `auth:refresh:revoked:{token}`）
- 有效期与 refresh token 相同（3 天或 30 天）

**重要说明**：
- ✅ 轮换机制**同时实现了滑动过期**
- ✅ 新的 refresh token 有**完整的有效期**（不是剩余时间）
- ✅ 这意味着持续使用可以无限期登录
- ⚠️ 这是**设计选择**，不是 bug！

**如果需要限制最长登录时间**：
- 可以实现"绝对过期"机制（见上文对比）
- 或者定期强制重新认证（如每90天）
- 当前实现：只有滑动过期，符合业界主流做法

### 🔒 重放检测（已实现）

**检测机制**：
1. 刷新时检查 token 是否已被标记为已使用
2. 如果已使用，说明发生了重放攻击
3. 撤销整个 token 家族（所有相关 session）
4. 撤销该用户的所有 session（更激进的安全措施）

**Token 家族（Token Family）**：
- 每个 session 对创建时生成唯一的 `token_family_id`
- 刷新时继承相同的 `token_family_id`
- 检测到重放时，撤销整个家族的所有 token

### 🔒 IP/UA 变化监控（已实现）

**设计理念**：监控但不阻止

**为什么不验证 IP 一致性？**

严格的 IP 验证会严重影响用户体验：
- **移动场景**：WiFi ↔ 4G 切换导致 IP 变化
- **VPN 使用**：用户开启/关闭 VPN
- **动态 IP**：运营商分配的 IP 可能变化
- **代理服务器**：企业网络使用代理
- **跨地域**：用户旅行或出差

**当前实现**：
- 记录 IP 和 User-Agent 变化
- 写入日志（警告级别）
- **不阻止**请求继续执行
- 便于安全审计和异常检测

**日志示例**：
```
WARNING: IP changed during token refresh: 
  original=192.168.1.100, current=10.0.0.50, 
  user_id=xxx, token_family_id=yyy
```

**推荐的安全策略**（参考微信）：
1. **设备指纹**：使用浏览器指纹识别设备
2. **异常登录通知**：检测到异常时发送通知
3. **风险评分**：综合多个因素评估风险
4. **敏感操作验证**：修改密码、绑定邮箱等需要额外验证
5. **设备管理**：用户可查看和管理登录设备

**未来增强**（可选）：
- 实现设备指纹识别
- 异常登录邮件/短信通知
- 设备管理界面
- 风险评分系统

### 🔒 并发控制（已实现）

**问题**：多个请求同时使用同一个 refresh token

**解决方案**：
- 使用 Redis 分布式锁（`auth:refresh:lock:{token}`）
- 锁的有效期：5 秒
- 只有获得锁的请求才能刷新
- 其他请求返回错误："Refresh token is being used by another request"

**实现细节**：
- 使用 Redis `SETNX` 命令（原子操作）
- 使用 Lua 脚本释放锁（确保只删除自己的锁）
- 锁自动过期，防止死锁

### 🔒 Rate Limiting（已实现）

**刷新接口限流**：
- 每分钟最多 10 次刷新请求（按 IP）
- 防止暴力破解 refresh token
- 超过限制返回 429 Too Many Requests

**登录接口限流**：
- 5 次失败后锁定 15 分钟
- 3 次失败后需要验证码
- 基于用户名 + IP 双重限制

### 🔒 并发 Session 限制（已实现）

**限制**：
- 每个用户最多 10 个活跃 session
- 超过限制时删除最旧的 session（FIFO）
- 防止无限创建 session

**使用场景**：
- 防止恶意用户创建大量 session
- 限制同时登录设备数量
- 节省 Redis 存储空间

## 安全机制

### 1. Session ID 生成

**方法**：`secrets.token_urlsafe(32)`
- 生成 32 字节（256 位）随机字符串
- URL 安全（Base64 编码）
- 碰撞概率：2^-256（实际上不可能）

### 2. 密码安全

**密码强度要求**：
- 至少 8 位字符
- 至少一个大写字母 (A-Z)
- 至少一个小写字母 (a-z)
- 至少一个数字 (0-9)
- 至少一个特殊字符 (!@#$%^&*等)

**密码哈希**：
- 使用 bcrypt 算法
- 自动加盐
- 计算成本可调（防止暴力破解）

### 3. 登录保护

**验证码**（失败 3 次后触发）：
- 图形验证码（4-6 位字母数字）
- 5 分钟有效期
- 不区分大小写
- 防止机器人攻击

**登录尝试限制**：
- 3 次失败：需要验证码
- 5 次失败：锁定 15 分钟
- 限制基于：用户名 + IP（双重限制）

### 4. Session 安全

**即时撤销**：
- 所有 session 存储在 Redis 中
- 可以立即撤销任何 session
- 无需等待 token 过期

**Token 轮换**：
- 每次刷新后，旧的 refresh token 失效
- 防止 token 泄露后被重复使用

**重放检测**：
- 检测旧 token 被再次使用
- 撤销整个 token 家族
- 撤销用户所有 session

**并发控制**：
- 使用分布式锁防止并发刷新
- 防止竞态条件

### 5. Redis 安全

**连接池**：
- 使用 Redis 连接池（最大 50 个连接）
- 所有客户端共享连接池
- 提高性能，减少连接开销

**数据结构**：
- Session 数据：`auth:session:{session_id}`
- 用户 Session 列表：`auth:user:sessions:{user_id}`（过期时间：3天）
- 已撤销 Token：`auth:refresh:revoked:{token}`（过期时间：3天）
- 刷新锁：`auth:refresh:lock:{token}`（过期时间：5秒）

**过期时间修复**：
- `USER_SESSIONS_PREFIX` 使用 refresh token 的过期时间（3天）
- 之前错误使用 access token 的过期时间（30分钟），导致用户 session 列表过早清除

### 6. 审计日志

**记录内容**：
- 登录成功/失败
- 登出
- Token 刷新
- 重放攻击检测
- IP/UA 变化
- Session 所有权验证失败
- 所有敏感操作

**日志格式**：
```python
logger.info(
    f"User login: user_id={user.id}, "
    f"username={user.username}, "
    f"tenant_id={user.tenant_id}, "
    f"ip={client_ip}, "
    f"timestamp={datetime.utcnow().isoformat()}"
)
```

**日志级别**：
- `INFO`：正常操作（登录、登出、刷新）
- `WARNING`：异常但不阻止（IP 变化、UA 变化）
- `ERROR`：错误和攻击（重放检测、所有权验证失败）

## 总结

### 核心特性

1. ✅ **纯 Redis Session**：简单、安全、易于管理
2. ✅ **双模式认证**：Cookie 模式（Web）+ Bearer Token 模式（移动端/API）
3. ✅ **当前使用 Cookie 模式**：HttpOnly + SameSite，最安全的 Web 认证方式
4. ✅ **前后端协同"记住我"**：后端控制过期时间，前端传递参数
5. ✅ **滑动过期**：持续使用时自动续期（Token Rotation）
6. ✅ **Refresh Token 轮换**：防止 token 泄露后被重复使用
7. ✅ **重放检测**：检测旧 token 被再次使用，撤销整个 token 家族
8. ✅ **并发控制**：使用分布式锁防止并发刷新
9. ✅ **Session 所有权验证**：登出时验证 session 属于当前用户
10. ✅ **登出时正确撤销**：三层回退机制，确保 refresh session 被撤销
11. ✅ **IP/UA 监控**：记录变化但不阻止（用户体验优先）
12. ✅ **Rate Limiting**：防止暴力破解和滥用
13. ✅ **并发 Session 限制**：每用户最多 10 个活跃 session
14. ✅ **即时撤销**：支持立即撤销任何 session
15. ✅ **会话管理**：查看和管理所有活跃会话
16. ✅ **登录保护**：验证码、登录尝试限制
17. ✅ **审计日志**：记录所有认证事件
18. ✅ **Redis 连接池**：共享连接，提高性能
19. ✅ **用户信息缓存**：减少数据库查询，提升性能

### 安全优势

**Cookie 模式（当前）**：
- **HttpOnly 防止 XSS**：JavaScript 无法读取 Cookie
- **SameSite 防止 CSRF**：阻止跨站请求携带 Cookie
- **服务端完全控制**：所有 session 数据在服务端
- **即时撤销**：无需等待 token 过期
- **强随机性**：256 位随机 session ID
- **密码安全**：bcrypt 哈希 + 强度验证
- **登录保护**：验证码 + 尝试限制
- **重放检测**：防止 token 重复使用
- **审计追溯**：完整的操作日志
- **用户体验优先**：监控但不过度限制

**Bearer Token 模式（未来）**：
- **灵活性**：适合移动端和第三方 API
- **跨平台**：支持所有客户端
- **无 CORS 限制**：简化跨域配置
- **服务端控制**：session 数据仍在服务端

### 性能考虑

- **Redis 连接池**：复用连接，提高性能
- **分布式锁**：防止并发问题
- **Redis 持久化**：防止数据丢失
- **支持水平扩展**：Redis 集群
- **用户信息缓存**：减少数据库查询
- **浏览器自动处理**：Cookie 模式下无需前端逻辑

### 已实现的安全修复

1. ✅ **登出时正确撤销 Refresh Session**
   - 三层回退机制（read_session → 直接读取 Redis → 撤销所有 session）
   - 防止用户登出后，攻击者仍可使用 refresh token

2. ✅ **Session 所有权验证**
   - 验证 session 的 user_id 与当前用户一致
   - 防止用户使用他人的 session 登出

3. ✅ **Refresh Token 重放检测**
   - 标记已使用的 refresh token
   - 检测到重放时，撤销整个 token 家族
   - 撤销用户所有 session（更激进的安全措施）

4. ✅ **Token 预刷新**
   - 前端检测 token 即将过期（5分钟内）
   - 提前刷新，用户无感知

5. ✅ **IP/UA 变化监控**
   - 记录 IP 和 User-Agent 变化
   - 写入日志但不阻止请求
   - 便于安全审计

6. ✅ **Rate Limiting**
   - 刷新接口：每分钟最多 10 次
   - 登录接口：5 次失败锁定 15 分钟

7. ✅ **并发 Session 限制**
   - 每用户最多 10 个活跃 session
   - 超过限制删除最旧的 session

8. ✅ **Redis 连接池修复**
   - 使用 Redis 连接池
   - 所有客户端共享连接池
   - 提高性能和稳定性

9. ✅ **USER_SESSIONS_PREFIX 过期时间修复**
   - 使用 refresh token 的过期时间（3天）
   - 之前错误使用 access token 的过期时间（30分钟）

10. ✅ **API URL 统一**
    - 前端使用 `REAL_API_URL` 常量
    - 避免硬编码 API 地址
