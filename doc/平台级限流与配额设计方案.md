# 平台级限流与配额设计方案

## 当前阶段说明

这份文档描述的是平台最终形态的治理蓝图，覆盖：

- 并发控制
- 频率限流
- 周期配额
- 成本预算
- 队列与优先级

但**当前项目阶段不建议一次性全部落地**。结合现有模型中心与调用链，当前应先聚焦一个最小可用目标：

1. **只做并发控制，不做 RPM / TPM / token 配额 / 成本预算**
2. **只做模型级总体并发控制，先不做 tenant / provider / kb / api_key 多层叠加**
3. **统一在模型调用入口生效，避免前端配了但运行时没接上**

也就是说，当前阶段推荐的最小实现是：

- 保留现有全局并发保护
- 允许为某个 `tenant_model` 单独配置 `concurrency_limit`
- 所有 `chat / embedding / rerank / asr / tts` 都在统一调用入口应用该限制

后续如果要扩展，再在这个基础上逐层加：

- tenant 级
- provider / model 资源池级
- HTTP 入口级
- token / cost / 月度配额级

## 一、背景

当前平台是一个面向未来扩展的通用 AI 平台，不只是知识库系统，后续还会持续承载：

- 多租户知识库
- 知识库增强任务（自动问题、自动摘要、自动标签等）
- 知识图谱 / Raptor 等重型构建任务
- 平台内聊天测试
- 对外开放的租户 API
- Agent 运行与工具调用
- 模型平台统一调用入口

这类平台的资源治理不能只做单点限流，而必须同时覆盖：

- 多租户公平性
- 上游模型 / API 资源保护
- 平台自身 CPU / 内存 / Worker / Redis / DB 保护
- 套餐计费与周期配额
- 交互流量与后台流量隔离

因此，平台需要同时设计两套能力：

1. **限流与并发控制**
2. **套餐配额与周期预算控制**

这两者相关，但不能混为一谈。

## 二、核心设计原则

### 2.1 配额与并发分离

- **限流 / 配额**：控制一段时间内最多能用多少
- **并发控制**：控制同一时刻最多能跑多少

示例：

- 每分钟最多调用 100 次 API：属于限流
- 同时最多跑 10 个大模型请求：属于并发控制
- 每月最多消耗 500 万 token：属于周期配额

### 2.2 交互流量与后台流量隔离

必须把下面两类流量拆开治理：

- **交互类流量**：聊天测试、租户对外 chat API、Agent 在线会话
- **后台类流量**：自动摘要、自动标签、知识图谱构建、Raptor 重建、批量增强

否则后台任务很容易把实时交互拖死。

### 2.3 多层策略叠加，最小上限生效

一次请求往往同时受多个维度控制：

- 全局
- 租户
- 知识库
- Agent
- 接口
- API Key
- Provider
- 具体模型

最终采用“**命中的所有规则中最严格的那个**”。

### 2.4 不是只看请求数，还要看 token / 成本 / 队列深度

LLM 场景下，只按请求次数限制不够，必须同时考虑：

- 请求次数
- token 数
- 调用成本
- 队列积压
- 并发占用时长

### 2.5 重任务要支持排队、去重、合并

对于知识图谱、Raptor、自动标签、自动摘要等后台任务：

- 可以排队
- 要限制队列深度
- 同类重复任务要支持去重或合并

而不是所有请求都直接放行到队列里。

## 三、统一名词

为了避免后续实现阶段语义混乱，建议统一以下术语。

### 3.1 限流类术语

- **Rate Limit**：单位时间内的频率限制
- **Concurrency Limit**：同一时刻的并发上限
- **Quota**：周期内累计可使用额度
- **Budget**：预算，通常对应 token / 次数 / 成本上限
- **Admission Control**：准入控制，决定请求能否开始
- **Backpressure**：背压，请求过多时排队、拒绝、降级或延后

### 3.2 平台统一工作负载类型

建议把所有流量归到以下几类，而不是按页面功能散落配置：

- `interactive_chat`
- `interactive_agent`
- `batch_llm_enhance`
- `batch_kg_build`
- `batch_raptor_build`
- `embedding_index_build`
- `external_open_api`
- `internal_system`

### 3.3 平台统一来源字段

建议所有模型调用、任务提交、开放 API 请求，都带统一 `request_source`。

推荐值：

- `ui_chat_test`
- `tenant_open_api_chat`
- `tenant_open_api_agent`
- `kb_auto_summary`
- `kb_auto_questions`
- `kb_auto_tags`
- `kg_extract`
- `raptor_build`
- `agent_llm_step`
- `agent_tool_call`
- `model_platform_api`
- `runtime_internal`

该字段后续可以直接用于：

- 策略匹配
- 监控统计
- 成本分摊
- 优先级控制

## 四、总体分层设计

建议限流与配额按两条轴设计：

### 4.1 业务归属轴

- 全局
- 租户
- 知识库
- Agent
- 用户 / API Key / 客户端

### 4.2 资源消耗轴

- Provider
- 模型
- 能力类型（chat / embedding / rerank / vision）
- 任务类型
- 接口类型

最终形成“多层叠加”的治理模型。

## 五、必须支持的控制层级

## 5.1 全局层

全局层用于保护整个平台，防止任何单一来源拖垮系统。

建议支持：

- `platform.global.requests_per_second`
- `platform.global.requests_per_minute`
- `platform.global.llm_concurrency`
- `platform.global.embedding_concurrency`
- `platform.global.agent_run_concurrency`
- `platform.global.kg_concurrency`
- `platform.global.raptor_concurrency`
- `platform.global.background_job_queue_depth`

作用：

- 平台自保
- 防止配置错误或流量突刺
- 防止上游异常时大量请求堆积

## 5.2 Provider / 模型资源层

这一层是保护“大模型本身并发有限”最关键的一层，必须独立存在。

建议至少支持以下粒度：

- `provider`
- `provider + capability`
- `provider + model`
- `provider + model + capability`

示例：

- `openai.chat.global_concurrency`
- `openai.embedding.global_concurrency`
- `openai:gpt-4.1.chat.concurrency`
- `openai:text-embedding-3-large.embedding.concurrency`
- `vllm:qwen2.5-72b.chat.concurrency`

建议该层同时支持：

- 并发限制
- RPM（每分钟请求数）
- TPM（每分钟 token 数）
- 每日 / 每月成本预算

原因：

- 不同模型吞吐差异很大
- chat 与 embedding 的资源模型不同
- 仅限制请求数无法覆盖长 prompt / 长上下文场景

## 5.3 租户层

租户层用于做多租户公平和套餐治理。

建议支持：

- `tenant.global.requests_per_minute`
- `tenant.global.llm_concurrency`
- `tenant.global.embedding_concurrency`
- `tenant.global.tokens_per_minute`
- `tenant.daily_token_budget`
- `tenant.monthly_token_budget`
- `tenant.monthly_cost_budget`

同时建议拆分为：

- `tenant.interactive.*`
- `tenant.batch.*`

例如：

- 交互类聊天每分钟 200 次
- 后台增强任务每分钟只允许提交 20 次
- 交互类并发 20
- 后台类并发 5

## 5.4 知识库层

知识库层用于防止某个 KB 的构建任务拖垮同租户其他 KB。

建议支持：

- `tenant + kb + interactive_chat_concurrency`
- `tenant + kb + batch_enhance_concurrency`
- `tenant + kb + kg_build_concurrency`
- `tenant + kb + raptor_build_concurrency`
- `tenant + kb + queue_depth`

建议默认策略：

- 知识库聊天测试：按 `interactive_chat`
- 自动摘要 / 自动问题 / 自动标签：按 `batch_llm_enhance`
- 知识图谱 / Raptor：按 `heavy_batch`

## 5.5 Agent 层

Agent 不是一次调用，而是一串调用，因此必须单独治理。

建议支持：

- `tenant + agent_definition + run_concurrency`
- `tenant + agent_definition + requests_per_minute`
- `tenant + agent_run + max_steps`
- `tenant + agent_run + max_llm_calls`
- `tenant + agent_run + max_tool_calls`
- `tenant + agent_run + max_total_tokens`
- `tenant + agent_tool_concurrency`

说明：

- 要限制 Agent 入口运行数
- 还要限制 Agent 内部 fan-out
- 否则一个 Agent run 可以放大成很多次模型调用和工具调用

## 5.6 接口 / API Key 层

该层用于开放 API 防滥用与细粒度套餐控制。

建议支持：

- `tenant + endpoint`
- `tenant + api_key`
- `tenant + client_app`
- `tenant + ip`（可选）

建议按接口类型拆分：

- `chat api`
- `agent invoke api`
- `kb search api`
- `batch submit api`
- `admin api`
- `model platform api`

## 六、套餐与周期配额设计

平台需要支持付费用户在固定周期内按不同口径计费或限制。

建议将套餐能力拆成两层：

1. **套餐定义层**
2. **租户订阅生效层**

## 6.1 套餐定义支持的计量维度

建议支持以下配额口径：

### 6.1.1 按 API 请求次数

适用于：

- 对外开放 API
- 平台页面中的聊天测试
- Agent 调用接口

典型指标：

- 每日 API 调用次数
- 每月 API 调用次数
- 按 endpoint 分类计数

示例：

- 每月 10 万次 chat API
- 每月 1 万次 agent invoke API

### 6.1.2 按大模型调用次数

适用于：

- 只要调用了 LLM 就扣一次
- 不区分 HTTP API 请求是否为同一接口

典型指标：

- 每日 LLM 调用次数
- 每月 LLM 调用次数
- 按模型类型分别统计

示例：

- 每月最多 2 万次 chat 模型调用
- 每月最多 5 万次 embedding 调用

### 6.1.3 按 token 数

这是最推荐的主配额口径。

建议拆分：

- 输入 token
- 输出 token
- 总 token
- 按模型组统计 token

典型指标：

- 每分钟 token 限流
- 每日 token 配额
- 每月 token 配额

示例：

- 每月 500 万输入 token
- 每月 100 万输出 token
- 每月总 token 600 万

### 6.1.4 按成本预算

适用于：

- 多模型价格不一致
- 不同模型单价差异大
- 未来做真正商业化结算

建议支持：

- 日预算
- 月预算
- 总预算阈值告警

示例：

- 每月模型调用预算 500 元
- 超过 80% 触发告警
- 超过 100% 自动停用高成本能力

### 6.1.5 按任务次数

适用于后台任务。

建议支持：

- 自动摘要次数
- 自动标签次数
- 自动问题次数
- 知识图谱重建次数
- Raptor 重建次数
- Agent run 次数

示例：

- 每月允许 100 次知识图谱重建
- 每月允许 1000 次自动摘要任务

## 6.2 套餐周期类型

建议支持以下周期：

- `minute`
- `hour`
- `day`
- `week`
- `month`
- `billing_cycle`

说明：

- `minute/hour` 通常用于频率限流
- `day/month/billing_cycle` 通常用于套餐配额

其中 `billing_cycle` 建议支持：

- 按自然月
- 按订阅起始日滚动月

## 6.3 套餐能力分层

建议套餐定义支持以下维度：

- 免费版
- 专业版
- 企业版
- 私有化版

并允许每个租户做覆盖配置：

- 标准套餐默认值
- 销售特批值
- 临时提额值
- 灰度试用值

## 七、统一策略模型建议

建议把“限流、并发、配额、预算”统一抽象为策略表，而不是散落在配置文件中。

推荐逻辑模型如下：

```text
Policy
- id
- policy_type                  # rate_limit / concurrency / quota / budget / queue_depth
- subject_type                 # platform / tenant / kb / agent / endpoint / api_key / provider / model
- subject_id                   # 具体对象 ID，平台级可为空或固定 global
- workload_type                # interactive_chat / batch_llm_enhance / batch_kg_build ...
- capability_type              # chat / embedding / rerank / vision / tool_call
- request_source               # ui_chat_test / kb_auto_summary / ...
- endpoint_key                 # /api/v1/chat/completions 等
- provider_key                 # openai / azure_openai / ollama / vllm
- model_key                    # gpt-4.1 / qwen2.5-72b / text-embedding-3-large
- metric_type                  # requests / llm_calls / tokens_in / tokens_out / tokens_total / cost / running_jobs
- window_type                  # second / minute / hour / day / month / billing_cycle
- limit_value
- burst_value
- priority                     # 规则优先级
- action_on_exceed             # reject / queue / delay / degrade
- queue_name                   # 命中排队时进入哪个队列
- is_enabled
- effective_start_at
- effective_end_at
```

## 八、统一运行时上下文建议

每次请求、任务、模型调用都先构建统一 `LimitContext`。

推荐字段：

```text
LimitContext
- tenant_id
- user_id
- api_key_id
- kb_id
- agent_definition_id
- agent_run_id
- endpoint_key
- request_source
- workload_type
- provider_key
- model_key
- capability_type
- estimated_input_tokens
- estimated_output_tokens
- priority
- current_queue_name
```

平台根据该上下文自动匹配应命中的全部策略。

## 九、计数口径建议

为避免后续“到底扣哪一次”的争议，必须定义统一计量口径。

## 9.1 API 请求次数

建议在 HTTP 入口处计数，按“成功进入业务层”记一次。

可选统计口径：

- 所有请求次数
- 成功请求次数
- 被限流拒绝次数
- 上游失败请求次数

## 9.2 大模型调用次数

建议在统一模型调用入口计数，按“实际发起上游模型请求”记一次。

说明：

- 一个 Agent run 可以消耗多次 LLM 调用
- 一个 chat API 也可能只触发一次或多次内部调用

因此“大模型调用次数”不能等于“API 请求次数”。

## 9.3 token 数

建议按统一模型调用日志中的 usage 记录。

至少保留：

- `input_tokens`
- `output_tokens`
- `total_tokens`

对于调用前预估限制，建议同时维护：

- `estimated_input_tokens`
- `estimated_total_tokens`

这样可以先做准入，再在调用完成后做真实回写与补差。

## 9.4 成本金额

建议基于：

- 模型单价表
- 实际 token usage
- 调用类型

统一换算成成本字段：

- `cost_amount`
- `currency`

后续支持：

- 多模型定价
- 不同租户加价系数
- 包含免费额度后的超额计费

## 十、准入流程建议

建议请求进入平台后，按以下顺序做控制。

### 10.1 HTTP 入口限流

位置：

- FastAPI middleware
- 或 API 网关层

作用：

- 防止恶意打流量
- 做 `tenant + endpoint + api_key + ip` 级请求频率限制

### 10.2 业务准入控制

位置：

- Service 层
- 提交任务前
- 发起模型调用前

作用：

- 校验租户套餐是否还有额度
- 校验知识库级任务并发
- 校验 Agent 是否超过运行上限
- 校验后台任务是否超过排队深度

### 10.3 资源准入控制

位置：

- 统一模型调用执行器
- 外部 API 调用执行器

作用：

- 校验全局并发
- 校验模型并发
- 校验 provider RPM / TPM
- 校验系统资源池可用性

### 10.4 结束回写

在调用结束后统一回写：

- 实际 token
- 实际成本
- 成功 / 失败状态
- 并发释放
- 周期配额累计值

## 十一、推荐的策略执行顺序

一次请求建议按以下顺序检查：

1. 是否命中 HTTP 入口频率限制
2. 是否命中套餐周期配额上限
3. 是否命中业务域并发限制
4. 是否命中 Provider / 模型资源限制
5. 是否命中队列深度限制
6. 决定直接执行、排队、延迟还是拒绝

建议优先级：

- 先做轻量快速检查
- 再做昂贵检查
- 对重任务优先返回“排队中”，对交互任务优先返回“超限拒绝”

## 十二、优先级与队列建议

平台建议至少区分三档优先级：

- `P1`：交互类实时请求
- `P2`：普通后台任务
- `P3`：重型构建任务

推荐映射：

- 页面聊天测试：`P1`
- 租户 chat API：`P1`
- Agent 在线会话：`P1`
- 自动摘要 / 自动标签 / 自动问题：`P2`
- 知识图谱构建 / Raptor 重建：`P3`

建议策略：

- `P1`：短队列，超限快速失败
- `P2`：可排队，限制队列长度
- `P3`：低并发、强去重、支持延迟启动

## 十三、后台任务的特殊策略

对知识图谱、Raptor、自动增强等任务，建议加入以下规则。

### 13.1 同类型任务去重

示例：

- 同一个 KB 已有 `kg_build` 在运行或排队时，不再重复创建同类全量任务
- 同一个 KB 的 `auto_tag` 若已有全量任务，可合并为一次

### 13.2 队列深度限制

建议支持：

- 全局队列深度
- 租户队列深度
- 知识库队列深度

### 13.3 KB 级互斥或低并发

建议默认：

- 同一知识库同时最多 1 个 `kg_build`
- 同一知识库同时最多 1 个 `raptor_build`
- 同一知识库同时最多 N 个增强任务

## 十四、Agent 的专项治理

Agent 需要区别于普通 chat。

建议额外控制：

- 单次 run 最大步数
- 单次 run 最大工具调用数
- 单次 run 最大 LLM 调用数
- 单次 run 最大总 token
- 单个 Agent 定义并发运行数
- 单租户 Agent 总运行数

建议对 Agent 的成本统计至少细分为：

- planning token
- tool selection token
- memory summarize token
- final response token
- external tool API 次数

### 15.2 套餐规则表


## 十六、Redis Key 建议

运行时的高频计数和并发控制建议放 Redis。

示例：

```text
rl:{policy_id}:{window_start_ts}
cc:{subject_type}:{subject_key}
quota:{tenant_id}:{metric_type}:{window_start_ts}
queue_depth:{workload_type}:{scope_key}
dedupe:{task_type}:{tenant_id}:{kb_id}
```

建议说明：

- `rl:` 用于频率限流
- `cc:` 用于并发占用
- `quota:` 用于周期额度快速判断
- `queue_depth:` 用于排队上限
- `dedupe:` 用于重任务去重

## 十七、策略动作建议

超过限制后，不同场景建议采取不同动作。

### 17.1 reject

适用于：

- 实时交互请求
- 明确超出套餐配额
- 明确超过模型资源上限且不应排队

### 17.2 queue

适用于：

- 后台增强任务
- 图谱构建
- Raptor 构建
- 批量 embedding

### 17.3 delay

适用于：

- 短时间抖动后的重试
- 低优先级后台任务

### 17.4 degrade

适用于：

- 超预算后切换到低成本模型
- 暂时关闭高成本增强功能
- 只保留核心交互能力

## 十八、告警与可观测性建议

限流与配额系统必须自带观测能力。

建议至少监控：

- 各层级命中次数
- 各层级拒绝次数
- 排队次数与平均等待时长
- provider / model 并发占用
- token 使用量
- 成本消耗量
- 配额消耗进度
- 重任务去重命中次数

建议至少做以下告警：

- 租户月预算达到 80%
- 租户月预算达到 100%
- 某模型并发占用长期接近上限
- 某类后台任务队列积压过深
- 某租户短时突刺异常

## 十九、落地顺序建议

### 19.1 第一期：先解决平台稳定性

## 二十二、推荐的首版最小实现范围

如果首版不想做得太重，建议先落下面这些能力：

### 22.1 当前推荐的最小落地范围

如果当前目标只是“先把某个模型的总体并发控住”，建议首版只做这 4 件事：

1. 全局并发控制继续保留
2. 模型级总体并发控制
3. 所有模型能力统一在 `services/model_platform_service.py` 生效
4. 前端模型中心支持配置 `concurrency_limit`

推荐配置结构：

```json
{
  "concurrency_limit": 4
}
```

说明：

- 这个值表示“该租户下这个模型在平台内的总体并发上限”
- 不区分 chat / embedding / rerank 子能力时，可以直接使用平铺结构
- 如果后续要细分能力，再演进为：

```json
{
  "default": { "concurrency_limit": 4 },
  "embedding": { "concurrency_limit": 8 }
}
```

当前阶段不建议马上做：

- RPM / TPM
- token 配额
- 成本预算
- tenant / provider / api_key 多层叠加
- HTTP 入口频率限流
- 套餐计费

### 22.2 当前阶段的最佳实践建议

当前只做模型级总体并发时，建议遵循下面几条：

1. **统一入口生效**
   不要在某些业务代码里单独限流，必须统一在模型调用服务层处理。

2. **全局保护 + 模型保护叠加**
   模型级并发不能替代全局并发，必须两层同时存在。

3. **先支持 fail_fast / wait / wait_timeout**
   这三种模式已经足够覆盖当前阶段的交互类和后台类需求。

4. **前端只暴露最少字段**
   首版只给用户一个“模型并发上限”，不要一开始就暴露 RPM、TPM、burst、预算这些复杂概念。

5. **日志里保留 request_source 和 capability_type**
   后续你要扩展 tenant / provider / 月配额时，才能平滑演进。

### 22.3 当前阶段的结论

对于当前项目，首版最合理的范围就是：

- 先把“某个模型总体并发不要超”做好
- 用 `tenant_model.rate_limit_config.concurrency_limit` 承载配置
- 统一在模型调用入口应用
- 前端模型中心可编辑

这套做法足够轻，也不会把后续更完整的治理路线堵死。
