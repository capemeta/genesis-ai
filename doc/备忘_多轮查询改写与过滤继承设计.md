# 多轮查询改写与过滤继承设计备忘

相关参考：

- [ragflow_查询改写](./ragflow/ragflow_查询改写.md)

## 1. 当前链路

当前聊天检索与回答大致分为四层：

1. 用户问题进入聊天会话
2. 检索侧执行 query analysis
   - 同义词改写
   - 检索词表扩展
   - 规则提取标签/元数据
   - LLM 提取候选与统一过滤表达式
3. Hybrid retrieval 检索 + rerank
4. 把检索结果拼成回答阶段的系统上下文，再交给聊天 LLM

当前已经有的多轮相关能力：

- 多轮过滤继承决策器：判断上一轮过滤是否延续
- query analysis：负责单轮问句的检索分析

当前已经新增独立的“查询改写器”阶段，但它与表达式 LLM 分开：

- 查询改写器：产出 `standalone_query`
- 表达式 / 候选 LLM：产出 `candidates` 与 `filter_expression`

## 2. 两种主要方案

### 方案 A：服务端存状态 + 多轮过滤继承

核心思想：

- 服务端保存上一轮的 `resolved_filters`
- 下一轮先判断当前问题是否继承上一轮范围
- 继承后再进入本轮 query analysis 与检索

适合处理：

- “还是刚才那个范围”
- “那赣州的呢”
- “继续查这个地区的”
- “同一事项下还有哪些材料”

优点：

- 理论上准确率上限更高
- 对省略问法支持强
- 能利用上一轮已经确认过的结构化范围

风险：

- 最怕误继承
- 一旦把上一轮地区、机构、标签错误带入本轮，可能直接把正确答案过滤掉
- 通用知识库里，主题跳转频繁，继承判断做不好会比较脆

### 方案 B：多轮查询改写 + scope hints

核心思想：

- 不优先继承硬过滤
- 先把当前问题改写成更完整的 standalone query
- 改写阶段顺带输出结构化范围 hints

示例：

- 当前问题：`那赣州的呢`
- 改写输出：
  - `standalone_query = 赣州地区的分支机构登记（备案）申请书规则是什么`
  - `scope_hints = { "metadata": { "地区": ["赣州"] } }`

优点：

- 更稳、更保守
- 不容易因为历史状态错误把当前范围锁死
- 对向量检索、全文检索、rerank 都自然有效
- 更适合标签类型还不够强结构化的通用知识库

风险：

- 改写后的问句只是补全语义，不等于天然形成强过滤
- 遇到强业务边界时，单靠问句改写不一定足够

## 3. 哪种效果更好

如果只看长期准确率上限：

- 方案 A 往往更高

如果看第一阶段的稳定性、可控性、落地风险：

- 方案 B 更合适

结论：

- 第一阶段更建议先做方案 B
- 第二阶段再引入方案 A

## 4. 最佳实践

最佳实践不是二选一，而是分层组合：

1. 多轮查询改写器
   - 输出 `standalone_query`
   - 输出 `scope_hints`
   - 负责补全省略表达、指代、延续主题
2. 过滤继承器
   - 只处理少量强约束字段
   - 例如：地区、部门、机构、事项
3. query analysis
   - 对 standalone query 继续做同义词改写、词表扩展、标签/元数据/表达式提取
4. 检索执行
   - 显式过滤优先
   - 继承得到的强约束进入硬过滤
   - 不确定的 hints 只作为弱信号，不直接锁死
5. 最终排序
   - 以 rerank / LLM 判断为主

## 5. 对标签与元数据的建议

### 不建议

- 所有标签默认多轮继承
- 所有历史命中标签都继续作为硬过滤

### 建议

- 只有少量“可继承强约束”才进入继承判断
- 普通标签更多作为 hints 或候选信号
- 元数据比纯标签更适合表达多轮强约束

推荐优先考虑继承的字段：

- 地区
- 部门
- 机构
- 事项
- 适用主体

## 6. 为什么不依赖客户端传上一轮命中标签

不建议把这件事交给前端。

原因：

- 客户端未必能拿到完整可信状态
- 容易受展示逻辑影响
- 多端同步复杂
- 很难区分“上一轮真实生效过滤”和“前端缓存的命中结果”

更合适的做法：

- 服务端存上一轮 `resolved_filters`
- 可选再存上一轮检索摘要

## 7. 聊天回答阶段上下文最佳实践

回答 LLM 不应只看到正文片段，也应适度看到结构化头信息，但必须克制：

- 文档标签：只传命中的/高价值标签名
- 文件夹标签：只传高层业务域标签
- 文档元数据：只传白名单字段
  - 地区
  - 部门
  - 适用主体
  - 生效时间

不建议：

- 把整包文档 metadata JSON 直接塞给回答模型
- 把所有文件夹标签、所有文档标签无差别灌进去

## 8. 文档摘要背景

当前聊天回答阶段已经支持把命中文档摘要作为补充背景，但前提是配置开启：

- 知识库 `persistent_context.enable_doc_summary_as_context = true`

行为：

- 若文档自身没有单独配置 persistent context
- 且知识库开启了“文档摘要作为背景”
- 则命中文档的 `kb_doc.summary` 会作为补充上下文传给回答模型

## 9. 推荐实施顺序

第一阶段：

1. 完善聊天回答阶段的结构化上下文
2. 新增多轮 standalone query 改写器
3. 让改写器输出 `standalone_query + scope_hints`

第二阶段：

1. 再补服务端过滤继承
2. 只先支持少数字段级强约束继承
3. 诊断清晰展示：
   - 改写结果
   - 继承结果
   - 本轮最终过滤

## 10. 当前建议

当前阶段先不开发多轮继承主链路时，建议优先走：

- 多轮查询改写
- scope hints
- query analysis 继续做表达式与过滤判定

这样能先提升效果，同时保持系统稳定。

## 10.1 当前已落地的查询改写流程

当前实现顺序是：

1. `raw_query`
2. 可选查询改写 LLM
   - 输出 `standalone_query`
   - 可选输出 `folder_routing_hints`
3. 同义词改写
4. 检索词表 / glossary / 过滤候选 / 表达式提取
5. 检索

其中：

- 查询改写开关：`enable_query_rewrite`
- 同义词改写开关：`enable_synonym_rewrite`

### 为什么查询改写在同义词改写之前

最佳实践上，应先补全多轮省略语义，再做术语标准化。

原因：

- 多轮改写需要优先理解原始问句和上文承接关系
- 如果先做同义词替换，可能会在省略问法里过早改写局部词面
- 先得到 `standalone_query`，再做同义词归一化，更利于全文检索和规则抽取

### 什么时候会真正调用查询改写 LLM

为了避免无收益改写，当前实现是保守启用：

- 聊天场景：有最近几轮上下文时，可以调用
- 检索测试：默认无历史上下文；只有存在有效多级目录时，才会调用
- 没有多轮上下文且只有根目录时：直接跳过

### 目录建议何时启用

只有当前知识库存在有效多级目录时，才会在查询改写 prompt 中启用目录建议。

如果只有根目录：

- 不输出目录建议
- prompt 会显式告诉模型“当前没有可用的多级目录范围”

### 目录建议当前如何使用

当前第一阶段先做这些：

- 进入诊断
- 作为后续 soft routing 的预留输出
- 不直接进入硬过滤

后续如果需要接 soft routing，推荐方式仍然是：

- 主候选目录优先召回
- 次候选目录补充召回
- 全库保底兜底

## 11. 聊天 / API 过滤能力边界

当前聊天里其实已经支持显式选择部分范围：

- 文件夹
- 文档
- 文件夹标签
- 文档标签
- 文档元数据
- 统一过滤表达式

这说明检索 API 的设计不应该只围绕“标签”展开，而应围绕“结构化范围能力”展开。

### 最佳实践：简单字段 + 高级表达式 双层设计

推荐分成两层：

#### 第一层：显式简单过滤字段

面向高频、清晰、可直接由前端控件表达的条件：

- `folder_ids`
- `kb_doc_ids`
- `document_ids`
- `folder_tag_ids`
- `tag_ids`
- `document_metadata`
- `search_unit_metadata`

这层的特点：

- 前端容易配置
- 语义清晰
- 诊断清楚
- 适合聊天作用域、检索测试、API 调用方直接指定

#### 第二层：统一过滤表达式

用于表达复杂组合关系：

- 括号
- AND / OR / NOT
- 多字段组合
- `in / not_in / exists / not_exists`
- 标签与元数据混合约束

这层的特点：

- 功能强
- 复杂度高
- 不应成为普通用户的默认入口

### 设计原则

最佳实践不是“所有维度都既做简单字段又做表达式”，而是：

- 高频能力优先走显式字段
- 复杂组合能力统一收敛到表达式

这里要明确分层：

- **查询理解层**：可以理解文件夹名称、目录路径、别名、自然语言提法
- **API 契约层**：尽量使用稳定 ID 和结构化字段

也就是说：

- 理解可以用名称
- 过滤落地尽量用 ID

## 12. 文件夹名称是否进入 API

### 结论

文件夹名称应该进入 query analysis / 查询改写 / 候选推断，但**不建议**作为 API 的长期显式过滤主字段。

更合适的做法：

- API / 前端显式过滤层只传 `folder_ids`
- 文件夹名称在 query analysis 中先解析成 `folder_id` 候选
- 高置信时再落成 `folder_ids` 或写入统一表达式

### 为什么不建议直接暴露 `folder_names`

原因：

- 名称可能重名
- 目录可能改名
- API 使用方不一定知道完整规范目录名
- 名称作为 API 主字段时稳定性差，不如 ID

所以最佳实践是：

- **内部支持按名称理解**
- **外部显式过滤统一用 ID**
- **文件夹名称属于查询理解层，不属于长期稳定 API 契约层**

## 13. 文件夹名称是否支持高级表达式

### 结论

不建议把“文件夹名称”本身做成高级表达式字段。

推荐表达方式：

- 表达式里仍然使用 `folder`
- 值用 `folder_id`

例如：

- `(folder in [A, B]) OR (tag in [X])`

而不是：

- `folder_name = "南康专区"`

### 原因

表达式应尽量基于稳定标识，而不是展示名称。

所以：

- 文件夹名称适合做“查询理解输入”
- `folder_id` 适合做“最终过滤表达”

## 14. 建议支持的过滤维度

### 显式简单过滤层

推荐长期稳定支持：

- 文件夹：`folder_ids`
- 知识库文档：`kb_doc_ids`
- 业务文档：`document_ids`（仅在确实需要按底层业务文档 ID 指定时使用）
- 文件夹标签：`folder_tag_ids`
- 文档标签：`tag_ids`
- 文档元数据：`document_metadata`
- 搜索单元元数据：`search_unit_metadata`

### 统一过滤表达式层

推荐支持：

- `folder`
- `tag`
- `folder_tag`
- `document_metadata`
- `search_unit_metadata`
- `kb_doc`
- `document`

这已经足够覆盖绝大多数复杂需求。

## 15. 是否会太复杂

如果把所有东西都开放成显式字段，会复杂。

如果把所有东西都逼到表达式，也会复杂。

最佳实践是：

- 显式字段覆盖 80% 的常见需求
- 表达式覆盖剩余 20% 的复杂组合

这样复杂度是可控的。

### 推荐边界

普通用户 / 普通前端表单：

- 只接触显式字段

高级调试 / 专业调用方：

- 使用统一过滤表达式

query analysis / LLM：

- 可以理解名称、别名、路径
- 最终输出仍收敛为 ID / metadata path / expression

## 16. 目录层级与 pageindex 类思路的关系

如果后续考虑引入 pageindex 这类“最佳查询路径”能力，建议建立在上面的设计之上。

### 它应该放在哪一层

结论：

- **不要塞进表达式 LLM**
- **更适合放进查询改写 LLM（尤其是多轮对话改写）**

原因：

- 表达式 LLM 的职责是提取标签、元数据、统一过滤表达式
- pageindex / 目录路径建议的职责是判断“当前问题更像要去哪条目录路径检索”
- 这更接近 standalone query 改写与检索导航，而不是过滤表达式生成

推荐职责分层：

1. 查询改写 LLM
   - 输出 `standalone_query`
   - 输出 `scope_hints`
   - 输出 `folder_routing_hints`
     - `primary_folder_candidates`
     - `secondary_folder_candidates`
     - `reason`
     - `confidence`
2. 表达式 LLM
   - 输出 `candidates`
   - 输出 `filter_expression`
3. 后端检索执行
   - 显式过滤优先
   - 表达式负责硬过滤
   - 当前实现里，`folder_routing_hints` 只进入诊断展示
   - 如果未来接入执行层，推荐只用于 soft routing / 候选优先级

也就是说：

- **查询改写 LLM 负责“问题补全 + 路径建议”**
- **表达式 LLM 负责“过滤理解 + 结构化表达”**

### 推荐做法（未来增强方向，不是当前已实现）

1. 先把文件夹名称解析为目录候选
2. 再让 LLM 或策略层给出“目录路径建议”
3. 该建议如果未来接入执行层，只做：
   - soft routing
   - 候选优先级
   - 诊断展示

### 不推荐做法

- 让 LLM 直接输出唯一目录路径，并强行只走这一条

### 结论

pageindex 类思路有价值，但只适合作为第二阶段增强层。

当前阶段最佳实践仍是：

- 文件夹名称先做目录候选
- API 显式过滤只用 `folder_ids`
- 复杂目录关系通过统一表达式处理

### 后续如果接入多轮查询改写

最佳实践是把目录路径建议并入“多轮查询改写 LLM”输出，而不是并入表达式 LLM。

推荐输出结构：

```json
{
  "standalone_query": "南康地区分支机构登记（备案）申请书规则是什么",
  "scope_hints": {
    "metadata": {
      "地区": ["南康"]
    }
  },
  "folder_routing_hints": {
    "primary_folder_candidates": [
      { "folder_id": "uuid-1", "confidence": 0.91, "reason": "命中完整目录语义" }
    ],
    "secondary_folder_candidates": [
      { "folder_id": "uuid-2", "confidence": 0.72, "reason": "命中上层目录语义" }
    ]
  }
}
```

使用原则（未来增强时推荐，不代表当前已经落地）：

- `standalone_query`：进入后续 query analysis 与检索
- `scope_hints`：作为后续过滤/表达式分析输入
- `folder_routing_hints`：推荐只用于 soft routing，不直接变成硬过滤

## 17. Soft Routing 推荐方案（未来增强方向）

如果后续目录路径建议接入检索执行层，当前最推荐的实现不是“只搜推荐目录”，而是**按召回预算做分层 soft routing**。

注意：本节描述的是推荐增强方向，当前代码尚未落地这套执行逻辑。

### 推荐用法

当存在目录候选时：

- 主候选目录：优先召回
- 次候选目录：补充召回
- 全库：保底兜底

### 示例

原本全库向量召回：

- `topK = 50`

接入 soft routing 后，可以调整成：

- 主候选目录下召回：30
- 次候选目录下召回：15
- 全库兜底召回：15

### 这样做的好处

- 候选目录更容易进入候选池
- 非候选目录仍然保留机会
- 不会因为目录建议失误而一刀切漏掉正确答案
- 比目录硬过滤更稳
- 比完全不利用目录层级更有效

### 使用原则

- 只影响召回预算和候选优先级
- 不直接变成硬过滤
- 显式 API / 前端过滤仍然优先
- 最终排序仍交给 rerank / 回答模型判断

### 置信度退化策略

如果目录候选置信度较低，建议退化为：

- 不做目录预算倾斜
- 仅保留全库召回

如果目录候选置信度较高，也建议：

- 仍保留少量全库兜底召回

这样能在“利用目录层级价值”和“避免误判漏召回”之间取得更好的平衡。
