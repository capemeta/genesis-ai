# 认证错误处理架构

## 架构概览

本项目采用**职责分离**的错误处理架构，将认证逻辑和 UI 逻辑分离到不同层次。

```
用户操作
  ↓
TanStack Query (useQuery/useMutation)
  ↓
axios 请求
  ↓
后端 API
  ↓
401 错误
  ↓
【第一层】axios 响应拦截器 (axios-instance.ts)
  - 尝试自动刷新 token
  - 刷新成功 → 重试原请求（用户无感知）
  - 刷新失败 → 清除认证状态，重新抛出错误
  ↓
【第二层】QueryCache 错误处理 (main.tsx)
  - 显示用户友好的提示："登录已过期，请重新登录"
  - 跳转到登录页（带 redirect 参数）
```

## 职责分离

### 第一层：axios 拦截器（底层认证逻辑）

**文件**：`src/lib/api/axios-instance.ts`

**职责**：
- ✅ 捕获 401 错误
- ✅ 自动刷新 token（使用 Cookie 中的 refresh_token）
- ✅ 使用刷新锁防止并发刷新
- ✅ 刷新成功后重试原请求
- ✅ 刷新失败后清除认证状态
- ❌ **不负责** UI 层逻辑（不显示提示、不跳转页面）

**代码示例**：
```typescript
axiosInstance.interceptors.response.use(
  (response) => response,
  async (error: AxiosError) => {
    if (error.response?.status === 401) {
      // 1. 尝试刷新 token
      try {
        await refreshToken()
        // 2. 刷新成功，重试原请求
        return axiosInstance(originalRequest)
      } catch (refreshError) {
        // 3. 刷新失败，清除认证状态
        useAuthStore.getState().logout()
        // 4. 重新抛出错误，让 QueryCache 处理 UI 层逻辑
        return Promise.reject(refreshError)
      }
    }
    return Promise.reject(error)
  }
)
```

### 第二层：QueryCache（UI 层错误处理）

**文件**：`src/main.tsx`

**职责**：
- ✅ 接收 axios 抛出的 401 错误
- ✅ 显示用户友好的提示
- ✅ 跳转到登录页
- ✅ 处理其他 HTTP 错误（403、500 等）
- ❌ **不负责** 认证逻辑（不刷新 token）

**代码示例**：
```typescript
queryCache: new QueryCache({
  onError: (error) => {
    if (error instanceof AxiosError) {
      if (error.response?.status === 401) {
        // axios 拦截器已尝试刷新 token
        // 如果到这里，说明刷新失败
        toast.error('登录已过期，请重新登录')
        router.navigate({ to: '/login', search: { redirect } })
      }
      if (error.response?.status === 403) {
        toast.error('无权限访问')
      }
      if (error.response?.status === 500) {
        toast.error('服务器错误，请稍后重试')
      }
    }
  },
})
```

## 工作流程

### 场景 1：Token 过期但 Refresh Token 有效

```
1. 用户访问租户列表页
2. axios 发送请求 → 401 错误
3. axios 拦截器自动刷新 token → 成功
4. axios 重试原请求 → 成功
5. 用户看到租户列表（无感知）
```

**用户体验**：✅ 无感知，页面正常加载

### 场景 2：Refresh Token 也过期

```
1. 用户访问租户列表页
2. axios 发送请求 → 401 错误
3. axios 拦截器尝试刷新 token → 失败（401）
4. axios 清除认证状态，重新抛出错误
5. QueryCache 捕获错误
6. 显示提示："登录已过期，请重新登录"
7. 跳转到登录页
```

**用户体验**：✅ 友好提示 + 自动跳转

### 场景 3：登录接口本身返回 401

```
1. 用户输入错误的用户名/密码
2. 登录接口返回 401
3. axios 拦截器检测到是认证接口，不尝试刷新
4. 直接重新抛出错误
5. 登录页面显示错误提示："用户名或密码错误"
```

**用户体验**：✅ 不会触发自动刷新（避免循环）

## 优势

### 1. 职责分离
- **axios**：专注底层认证逻辑
- **QueryCache**：专注 UI 层错误处理
- 代码更清晰，易于维护

### 2. 自动重试
- Token 过期时自动刷新并重试
- 用户无感知，体验更好

### 3. 统一处理
- 所有 401 错误都经过相同流程
- 避免重复代码

### 4. 防止并发刷新
- 使用刷新锁机制
- 多个请求同时 401 时，只刷新一次

### 5. 灵活扩展
- 可以轻松添加其他 HTTP 错误处理
- 可以自定义不同错误的提示和行为

## 与其他方案对比

### ❌ 方案 A：只在 axios 处理（不推荐）

```typescript
// axios 拦截器
if (error.response?.status === 401) {
  await refreshToken()
  toast.error('登录已过期')  // ❌ 混合了 UI 逻辑
  window.location.href = '/login'  // ❌ 混合了 UI 逻辑
}
```

**问题**：
- axios 层混合了 UI 逻辑
- 难以测试
- 不符合单一职责原则

### ❌ 方案 B：只在 QueryCache 处理（不推荐）

```typescript
// QueryCache
if (error.response?.status === 401) {
  await refreshToken()  // ❌ QueryCache 不应该处理认证逻辑
  toast.error('登录已过期')
  router.navigate({ to: '/login' })
}
```

**问题**：
- QueryCache 混合了认证逻辑
- 无法自动重试请求
- 用户体验差（每次都要重新登录）

### ✅ 方案 C：职责分离（推荐，当前方案）

```typescript
// axios 拦截器：认证逻辑
if (error.response?.status === 401) {
  await refreshToken()
  return retryRequest()  // 自动重试
}

// QueryCache：UI 逻辑
if (error.response?.status === 401) {
  toast.error('登录已过期')
  router.navigate({ to: '/login' })
}
```

**优势**：
- 职责清晰
- 自动重试
- 易于测试和维护

## 最佳实践

1. ✅ **所有 API 调用都使用 `axiosInstance`**
   ```typescript
   import axiosInstance from './axios-instance'
   
   // ✅ 正确
   const response = await axiosInstance.get('/api/v1/users')
   
   // ❌ 错误
   const response = await axios.get('http://localhost:8200/api/v1/users')
   ```

2. ✅ **不要在组件中手动处理 401 错误**
   ```typescript
   // ❌ 错误
   try {
     await fetchUsers()
   } catch (error) {
     if (error.response?.status === 401) {
       navigate('/login')  // 不需要手动处理
     }
   }
   
   // ✅ 正确
   const { data } = useQuery({
     queryKey: ['users'],
     queryFn: fetchUsers,
     // 401 错误会自动处理
   })
   ```

3. ✅ **使用 TanStack Query 的错误处理**
   ```typescript
   const { data, error } = useQuery({
     queryKey: ['users'],
     queryFn: fetchUsers,
     onError: (error) => {
       // 只处理业务逻辑错误，不处理 401
       if (error.response?.status === 400) {
         toast.error('请求参数错误')
       }
     },
   })
   ```

## 总结

本项目采用**职责分离**的错误处理架构：

- **axios 拦截器**：负责底层认证逻辑（自动刷新 token）
- **QueryCache**：负责 UI 层错误处理（显示提示、跳转页面）

这种架构清晰、易维护，提供了最佳的用户体验。
