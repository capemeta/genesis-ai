# 权限控制使用指南

## 概述

CRUD 工厂完全支持 API 级别的权限控制，使用非常简单！

## 🎯 核心特性

- ✅ **声明式权限**：注册时直接指定权限要求
- ✅ **细粒度控制**：每个操作（list/get/create/update/delete）可以单独配置权限
- ✅ **灵活组合**：支持"或"逻辑，满足任一权限即可
- ✅ **零额外代码**：权限检查自动集成，无需手动编写
- ✅ **兼容现有系统**：使用项目现有的 `require_permissions` 机制

## 🚀 快速开始

### 场景 1：不需要权限（只需登录）

```python
# 只需要用户登录即可访问
crud_factory.register(
    model=Tag,
    prefix="/tags",
    tags=["tags"]
)
# 所有操作只需要登录用户，不检查额外权限
```

### 场景 2：所有操作使用相同权限

```python
# 所有操作都需要 kb:read 或 admin 权限
crud_factory.register(
    model=KnowledgeBase,
    prefix="/knowledge-bases",
    tags=["knowledge-bases"],
    list_permissions=["kb:read", "admin"],
    get_permissions=["kb:read", "admin"],
    create_permissions=["kb:read", "admin"],
    update_permissions=["kb:read", "admin"],
    delete_permissions=["kb:read", "admin"]
)
```

### 场景 3：不同操作使用不同权限（推荐）

```python
# 读操作需要 kb:read，写操作需要 kb:write，删除需要 kb:delete
crud_factory.register(
    model=KnowledgeBase,
    prefix="/knowledge-bases",
    tags=["knowledge-bases"],
    list_permissions=["kb:read", "admin"],      # 列表：读权限
    get_permissions=["kb:read", "admin"],       # 获取：读权限
    create_permissions=["kb:write", "admin"],   # 创建：写权限
    update_permissions=["kb:write", "admin"],   # 更新：写权限
    delete_permissions=["kb:delete", "admin"]   # 删除：删除权限
)
```

### 场景 4：只读资源

```python
# 审计日志：只能读取，不能修改
crud_factory.register(
    model=AuditLog,
    prefix="/audit-logs",
    tags=["audit-logs"],
    list_permissions=["audit:read", "admin"],
    get_permissions=["audit:read", "admin"],
    enable_create=False,  # 禁用创建
    enable_update=False,  # 禁用更新
    enable_delete=False   # 禁用删除
)
```

### 场景 5：管理员专属资源

```python
# 系统配置：只有管理员可以访问
crud_factory.register(
    model=SystemConfig,
    prefix="/system-configs",
    tags=["system-configs"],
    list_permissions=["admin"],
    get_permissions=["admin"],
    create_permissions=["admin"],
    update_permissions=["admin"],
    delete_permissions=["admin"]
)
```

## 📋 权限配置参数

```python
crud_factory.register(
    model=YourModel,
    prefix="/your-resources",
    
    # 各操作的权限配置
    list_permissions=["permission1", "permission2"],    # 列表权限
    get_permissions=["permission1", "permission2"],     # 获取权限
    create_permissions=["permission1", "permission2"],  # 创建权限
    update_permissions=["permission1", "permission2"],  # 更新权限
    delete_permissions=["permission1", "permission2"],  # 删除权限
)
```

**权限逻辑**：
- 如果 `*_permissions` 为 `None`（默认）：只需要登录用户
- 如果 `*_permissions` 为列表：用户需要拥有列表中**任意一个**权限（OR 逻辑）
- 示例：`["kb:read", "admin"]` 表示用户拥有 `kb:read` **或** `admin` 权限即可

## 🎨 实际案例

### 案例 1：知识库管理

```python
# api/crud_registry.py
from core.crud_factory import crud_factory
from models.knowledge_base import KnowledgeBase

def register_all_crud():
    # 知识库：读写分离权限
    crud_factory.register(
        model=KnowledgeBase,
        prefix="/knowledge-bases",
        tags=["knowledge-bases"],
        list_permissions=["kb:read", "kb:write", "admin"],
        get_permissions=["kb:read", "kb:write", "admin"],
        create_permissions=["kb:write", "admin"],
        update_permissions=["kb:write", "admin"],
        delete_permissions=["kb:delete", "admin"]
    )
```

**效果**：
- 普通用户（kb:read）：可以查看知识库列表和详情
- 编辑用户（kb:write）：可以创建、修改知识库
- 管理员（admin）：拥有所有权限
- 删除操作需要特殊的 kb:delete 权限

**效果**：

## 🔧 权限系统集成

### 现有权限系统

项目使用 `core.security.require_permissions` 进行权限检查：

```python
# core/security.py
def require_permissions(*permissions: str):
    """
    权限检查装饰器
    
    用户需要拥有 permissions 中的任意一个权限
    """
    async def permission_checker(current_user: User = Depends(get_current_active_user)):
        # 检查用户是否拥有所需权限
        user_permissions = await get_user_permissions(current_user)
        
        if not any(perm in user_permissions for perm in permissions):
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail="You don't have permission to perform this action"
            )
        
        return current_user
    
    return Depends(permission_checker)
```

### CRUD 工厂如何使用

```python
# core/base_router.py
def get_user_dependency(permissions: Optional[List[str]] = None):
    """根据权限要求返回对应的用户依赖"""
    if permissions:
        # 如果指定了权限，使用 require_permissions
        from core.security import require_permissions
        return require_permissions(*permissions)
    else:
        # 否则只需要登录用户
        return get_current_user

# 在路由中使用
@router.get("/")
async def list_resources(
    current_user: User = Depends(get_user_dependency(list_permissions)),
    ...
):
    pass
```

**完全兼容现有系统，无需修改任何权限检查逻辑！**

## 🔍 权限检查流程

```
用户请求
    │
    ▼
FastAPI 路由
    │
    ▼
权限依赖检查
    │
    ├─ 如果配置了 *_permissions
    │   │
    │   ▼
    │  require_permissions(*permissions)
    │   │
    │   ├─ 获取用户权限列表
    │   ├─ 检查是否拥有任一所需权限
    │   ├─ 有权限 → 继续
    │   └─ 无权限 → 403 Forbidden
    │
    └─ 如果未配置 *_permissions
        │
        ▼
       get_current_active_user
        │
        ├─ 检查用户是否登录
        ├─ 已登录 → 继续
        └─ 未登录 → 401 Unauthorized
```

## 📝 常见问题

### Q1: 如何让某个操作不需要权限？

```python
# 方式 1：不指定权限参数（默认只需登录）
crud_factory.register(
    model=Tag,
    prefix="/tags",
    # 不指定 *_permissions，只需要登录用户
)

# 方式 2：明确指定为 None
crud_factory.register(
    model=Tag,
    prefix="/tags",
    list_permissions=None,  # 只需登录
    get_permissions=None,   # 只需登录
)
```

### Q2: 如何配置"与"逻辑（需要同时拥有多个权限）？

```python
# CRUD 工厂默认使用"或"逻辑
# 如果需要"与"逻辑，需要自定义 Service 或路由

# 方式 1：自定义路由
router = crud_factory.get_router("KnowledgeBase")

@router.post("/{kb_id}/publish")
async def publish_kb(
    kb_id: UUID,
    # 需要同时拥有两个权限
    current_user: User = Depends(require_permissions("kb:write")),
    _: User = Depends(require_permissions("kb:publish")),
):
    pass

# 方式 2：创建组合权限
# 在权限系统中定义 "kb:publish" 权限，只授予同时拥有 kb:write 的用户
```

### Q3: 如何查看用户拥有哪些权限？

```python
# 在路由中
@router.get("/me/permissions")
async def get_my_permissions(
    current_user: User = Depends(get_current_active_user)
):
    permissions = await get_user_permissions(current_user)
    return {"data": permissions}
```

## 🎉 总结

**权限控制非常简单**：

1. **不需要权限**：不指定 `*_permissions` 参数
2. **需要权限**：指定 `*_permissions=["perm1", "perm2"]`
3. **不同操作不同权限**：分别指定各操作的权限
4. **完全兼容**：使用现有的 `require_permissions` 机制

**零额外代码，声明式配置，清晰易维护！**
