# ZIP 上传与包内文件（含图片）处理方案

## 一、背景与目标

- **Markdown + 本地图片**：用户希望上传带本地图片的 MD 时，能通过一个 ZIP 包一起上传，避免图片链接失效。
- **ZIP 通用化**：ZIP 不应只服务 Markdown，应支持「任意类型文件打成的包」上传。
- **包内图片**：ZIP 内的图片如何参与检索？是否需要「视觉识别」？

当前状态简要说明：

- 前端 `file-upload-dropzone` 已接受 `.zip`、`.jpg` 等；后端 `documents.py` 的 `allowed_extensions` **未包含** `.zip` 和图片格式。
- 上传是「一个文件 → 一个 Document」；解析任务按「一个 kb_document → 一份文件内容」走 ParserFactory/ChunkerFactory，**没有解压 ZIP、没有按包内多文件分别处理**。
- 已有 **OCR 解析器**（图片/扫描件→文字）、**Vision 解析器**（视觉大模型理解图表/复杂布局），可用于图片内容抽取。

---

## 二、整体思路

1. **ZIP 定位为「通用容器」**  
   支持所有当前允许的文档类型 + 图片打成的 ZIP；上传后**解压**，对**包内每个文件**按类型分别解析与分块，而不是把 ZIP 当作一个整体去解析。

2. **两种可选存储/关联模型**  
   - **方案 A（推荐）**：ZIP 上传后解压，**为包内每个文件各建一条 Document**，再关联到知识库；MD 与图片在「同一批上传」或「同一文件夹」维度关联，通过 `source_package`、`relative_path` 等元数据表达「来自同一 ZIP」。  
   - **方案 B**：保留 ZIP 为一条 Document，解压到临时/持久目录，解析时按「包」为单位，内部遍历条目逐个解析，结果写回与「ZIP 这条 Document」关联的 chunk；需要新增「包级」的解析/分块流程。

3. **图片处理策略（是否做视觉识别）**  
   - **仅 OCR**：只做文字识别，适合扫描文档、截图中的文字。  
   - **仅视觉描述（Vision）**：用视觉大模型生成「这张图里有什么」的文本描述，适合图表、示意图、无文字图片。  
   - **组合策略（推荐）**：  
     - 默认：图片走 **OCR**，得到文本参与检索；成本低、可解释性强。  
     - 可选：对部分或全部图片再走 **Vision**，把描述拼进上下文（例如作为该图片的 alt 文本或独立 chunk），便于「按图意」检索。  
   - 实现上：ZIP 内图片在解析阶段按「文件类型」路由到现有 `OCRParser` / `VisionParser` 即可，无需新解析器，只需在「ZIP 解压后的单文件解析」里统一按扩展名选策略。

## 三、方案要点

### 3.1 上传与存储

- **允许的扩展名**：在现有列表上增加 `.zip`，以及图片格式（如 `.jpg`, `.jpeg`, `.png`, `.webp`, `.bmp`），与前端一致。  
- **ZIP 上传后**：  
  - 校验 ZIP 结构（避免炸弹、路径穿越）。  
  - 解压到存储的某个目录（例如按 `document_id` 或 `package_id` 建子目录），保持相对路径。  
  - **方案 A**：为每个解压出的文件创建一条 `Document` 记录，`file_key` 指向解压后的路径（或对象存储中的等价路径）；可选字段如 `source_package_id`（若保留 ZIP 为一条 Document）、`relative_path`（包内相对路径）。  
  - **方案 B**：ZIP 只对应一条 Document，`file_key` 指向 ZIP 本身，再在对象存储或本地为该 ZIP 维护一个「解压目录」的路径或清单（如 manifest.json），解析时按清单逐文件处理。

### 3.2 解析与分块（按「单文件」复用现有能力）

- **ZIP 包内每个文件**（或方案 B 下清单中的每个条目）：  
  - 按**扩展名**选择解析策略（与现有 `ParserFactory.auto_select_strategy` 一致）：  
    - `.md` → 文本解析（Basic 等） + Markdown 分块；  
    - `.pdf` / 图片等 → 现有 OCR / MinerU / Vision 逻辑。  
  - 若希望 **MD 与同包内图片** 在检索时更紧密：  
    - 可在 MD 解析时注入「同目录下的图片列表」的元数据；  
    - 或对「MD + 同包图片」做一次增强：把图片的 OCR/Vision 结果作为「附加段落」或 alt 文本与 MD 内容一起分块（需要定义「同包/同目录」的规则，例如同一 `source_package_id` + 同目录）。

### 3.3 图片：OCR 与视觉识别

- **统一按「单文件」走现有解析器**：  
  - 图片 → `ParserFactory` 自动选 OCR 或 Vision（或由知识库/租户配置指定）。  
- **是否做视觉识别**：  
  - **要**：对图片调用现有 **Vision 解析器**（视觉大模型），得到描述文本，再参与分块与检索。  
  - **不做**：仅用 **OCR**，只识别图中文字。  
  - 建议：**默认 OCR**，在知识库或解析配置中增加「对图片启用 Vision 描述」的开关，按需开启。

### 3.4 Markdown 内图片引用

- 解压后 MD 中相对路径（如 `./images/fig1.png`）需能解析到包内对应文件：  
  - 解析 MD 时，**工作目录**或**基础路径**设为该 MD 所在解压目录，这样相对路径可定位到同包内的图片文件。  
  - 若要对「MD 中引用的图片」做视觉增强：可解析 MD 得到图片引用列表，只对这些引用到的图片做 Vision/OCR，并把结果注入到 MD 的 chunk 元数据或上下文中（避免对包内全部图片都做 Vision，节省成本）。

## 四、实现步骤建议

1. **后端上传**  
   - 在 `documents.py` 的 `allowed_extensions` 中增加 `.zip` 和图片格式。  
   - 若上传为 ZIP：  
     - 校验并解压到指定存储路径；  
     - 按选定方案（A 或 B）创建一条或多条 Document，并写入 `source_package_id` / `relative_path` 等（若采用方案 A）。

2. **知识库关联**  
   - 若为方案 A：用户关联到知识库时，可以「按包」关联（一次选多个 document_id，或按 `source_package_id` 批量关联）；前端可展示「来自同一 ZIP」的分组。  
   - 若为方案 B：关联的是 ZIP 这条 Document，解析任务识别到是 ZIP 后，按解压清单逐个文件解析。

3. **解析任务**  
   - 对「来自 ZIP 的单个文件」：与现有一致，按 `file_extension` 走 `ParserFactory.auto_select_strategy` 和对应 Chunker。  
   - 对 ZIP 内图片：在解析层统一走 OCR；若配置了「图片启用 Vision」，再对同一图片调用 Vision 解析器，将描述与 OCR 文本合并后参与分块。

4. **配置与产品**  
   - 知识库或全局配置中增加：  
     - 「ZIP 内图片处理方式」：仅 OCR / 仅 Vision / OCR + Vision；  
   - 前端上传页说明：支持上传 ZIP，包内 MD 与图片将按类型分别解析，图片会做文字识别（及可选的视觉描述）。

---

## 五、小结

| 问题 | 建议 |
|------|------|
| ZIP 是否只针对 Markdown？ | **否**，ZIP 作为通用容器，包内任意支持的文档+图片都按类型分别处理。 |
| 包内图片怎么处理？ | **按文件类型路由**：图片走现有 **OCR**（文字识别）；可选开启 **视觉大模型** 做「图意描述」，与 OCR 结果一起参与检索。 |
| 是否要做视觉识别？ | **可选**。默认 OCR 即可；需要「按图意检索」时再开启 Vision，避免所有图片都走大模型、控制成本。 |

这样既满足「MD + 本地图片一起上传」的需求，又保持「ZIP 通用、图片可 OCR 可视觉」的扩展性。
