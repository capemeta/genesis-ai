# 检索存储与混合检索最佳实践方案

## 1. 目标与约束

当前项目需要同时支持：

- PostgreSQL 全文检索
- PostgreSQL 向量检索（pgvector）
- 混合检索（全文 + 向量）
- 多类型知识库统一承载
  - 通用文档
  - QA 知识库
  - 表格型知识库
  - 网页同步知识库
- 后续可能接入：
  - Qdrant / 其他向量数据库
  - Elasticsearch 等全文检索引擎
  - 知识图谱

并且当前项目处于开发阶段，允许推翻旧设计，因此本方案以长期可演进的最佳实践为目标，而不是兼容现有表结构的最小改动。

## 2. 先回答两个关键结论

### 2.1 `chunk_relations` 是否建议现在加入主方案

不建议把 `chunk_relations` 作为当前主检索链路的核心表。

原因：

1. 检索热路径不需要它
  当前主链路是“召回 -> 聚合 -> 回填上下文”。真正高频的是按检索单元做全文 / 向量召回，再聚合回业务单元。这个过程并不依赖任意图状关系。
2. 会带来额外写放大
  每次重新解析、重分块、重建索引时，不仅要写 `chunks` 和索引投影，还要同步维护关系边；对 QA、表格、网页同步这类频繁重建的场景不划算。
3. 会带来额外查询复杂度
  如果把它纳入主检索路径，查询往往会变成：
   检索命中 -> 查 relation -> 再查 chunk -> 再聚合。
   对 PG 热路径不友好。
4. 它解决的是“复杂跨块关系”，不是“混合检索主干问题”
  当前最核心的问题不是关系表达不足，而是：
  - 内容主表与检索投影未分层
  - QA / 表格 / 网页缺少统一的检索投影抽象
  - PG 全文与 PGVector 未来还要兼容外部引擎

结论：

- 当前阶段：不把 `chunk_relations` 放进主方案
- 后续阶段：如果真的需要表达跨块强语义关系，再补一张“冷路径关系表”，但不应进入热检索主链路

### 2.2 最佳实践是否还是“chunks + embeddings + search_vector”

不是。

更合理的最佳实践是三层分离：

1. 内容主表层
  负责保存真正的内容与展示结构
2. 检索投影层
  负责把内容拆成适合检索的逻辑单元
3. 引擎实现层
  负责当前用 PG 做全文与向量、未来切到 Qdrant / ES 等

这三层分开后，才能同时满足：

- 当前用 PG
- 后续切外部引擎
- QA 做问句优先 / 问答联合
- 表格和网页也能统一复用

## 3. 推荐的最终架构

推荐架构如下：

1. `chunks`
  内容主表，保存 canonical content
2. `chunk_search_units`
  检索投影表，保存“可检索单元”
3. `pg_chunk_search_unit_vectors`
  当前 PGVector 的本地物理索引表
4. `pg_chunk_search_unit_lexical_indexes`
  当前 PostgreSQL 全文检索的本地物理索引表
5. 当前运行时后端默认值来自 `.env`
  用于声明系统当前启用的向量 / 全文后端默认实现
6. 后续按需新增：
  - `kg_entities`
  - `kg_relations`
  - `kg_chunk_links`

这个方案的核心思想是：

- `chunks` 是内容事实
- `chunk_search_units` 是检索视角下的投影
- `pg_*` 表是当前 PostgreSQL 本地引擎实现
- 当前项目运行时只读取 `.env` 中的默认后端配置

## 4. 为什么要增加 `chunk_search_units`

这是本方案里最关键的一层。

如果没有它，你会被迫让 `chunks` 同时承担两种职责：

1. 给前端和大模型展示的最终内容
2. 给检索引擎做召回的检索文本

这在简单文档场景勉强可行，但在以下场景会变得很别扭：

- QA：展示要完整问答，检索却要分 `question` / `answer`
- 表格：展示要完整行或完整分组，检索可能只想检索某些列
- 网页：展示要完整正文块，检索可能只想用正文、标题、摘要的不同组合
- 后续知识图谱：图谱召回与文本召回的单位并不完全一致

因此最佳实践不是继续堆 `chunk_role` 在 `chunks.metadata` 里，而是显式增加一个“检索投影表”。

这里要特别强调一个边界：

- `chunks` 主要负责 canonical content、父子层级、前端展示、LLM 上下文回填
- `chunk_search_units` 主要负责检索召回、权重计算、混合检索聚合

因此，像 QA 的 `question_only / question_answer`、问题权重 / 答案权重这类能力，应优先落在 `chunk_search_units` 层，而不是继续改造前端已经依赖的 chunk 树结构。

## 5. 表设计方案

### 5.1 `chunks`：内容主表

职责：

- 保存最终内容单元
- 保存结构化正文流
- 保存前端预览与 LLM 上下文所需信息
- 不直接承担多路检索职责

建议保留字段：

- `id`
- `tenant_id`
- `kb_id`
- `document_id`
- `kb_doc_id`
- `content`
- `original_content`
- `content_blocks`
- `structure_version`
- `token_count`
- `text_length`
- `summary`
- `chunk_type`
- `status`
- `is_active`
- `is_content_edited`
- `position`
- `path`
- `parent_id`
- `metadata`
- `created_at`

建议新增字段：

- `content_hash`
用于判断 canonical content 是否变化，便于增量重建索引
- `source_type`
例如：
  - `document`
  - `qa`
  - `table`
  - `web`
  - `graph_context`
- `display_enabled`
是否允许作为最终上下文返回给前端 / LLM
- `content_group_id`
用于把多个 chunk 聚合回同一个业务单元
示例：
  - QA 的同一条问答
  - 表格的同一行
  - 网页的同一页面版本
  - 通用文档中同一章节父子块组

说明：

- `chunks` 不再负责表达“question 索引”“answer 索引”这类检索角色
- 这些职责应下沉到 `chunk_search_units`

### 5.2 `chunk_search_units`：检索投影表

职责：

- 保存逻辑上的“可检索单元”
- 一个 `chunk` 可以派生多个 `search_unit`
- 混合检索的主入口应基于 `search_unit`，不是直接基于 `chunk`

推荐字段：

- `id BIGSERIAL PRIMARY KEY`
- `tenant_id UUID NOT NULL`
- `kb_id UUID NOT NULL`
- `chunk_id BIGINT NOT NULL`
- `kb_doc_id UUID NOT NULL`
- `document_id UUID NOT NULL`
- `content_group_id UUID`
- `search_scope VARCHAR(50) NOT NULL`
- `search_text TEXT NOT NULL`
- `search_text_hash VARCHAR(64) NOT NULL`
- `token_count INTEGER DEFAULT 0`
- `text_length INTEGER DEFAULT 0`
- `is_active BOOLEAN DEFAULT true`
- `is_primary BOOLEAN DEFAULT false`
- `priority SMALLINT DEFAULT 100`
- `metadata JSONB DEFAULT '{}'::jsonb`
- `created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP`
- `updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP`

关键字段说明：

- `chunk_id`
指向最终内容 chunk
- `content_group_id`
聚合回同一业务单元的关键键
- `search_scope`
检索语义域，建议标准化
可取值示例：
  - `default`
  - `question`
  - `answer`
  - `summary`
  - `title`
  - `row`
  - `row_fragment`
  - `page_body`
  - `page_title`
- `search_text`
真正送入全文 / 向量索引的文本
- `is_primary`
是否为该业务单元的主检索投影
- `priority`
聚合和召回调优时的默认优先级
- `metadata`
用于记录检索投影来源与附加语义，建议至少包括：
  - `source_chunk_role`
  - `source_chunk_id`
  - `source_parent_chunk_id`
  - `source_node_id`
  - `qa_row_id` / `table_row_id` / `version_id`
  - `should_vectorize`
  - `exclude_from_retrieval`

索引建议：

- `INDEX (kb_id, is_active, search_scope)`
- `INDEX (chunk_id)`
- `INDEX (content_group_id)`
- `GIN (metadata jsonb_path_ops)`

说明：

- 未来外部检索引擎无论是 PG、Qdrant、ES，最终都围绕 `chunk_search_units.id` 进行投影
- 这使得存储层与检索引擎实现解耦

### 5.3 `pg_chunk_search_unit_vectors`：当前 PGVector 本地实现表

职责：

- 保存当前 PostgreSQL/pgvector 本地物理向量索引
- 只服务于“本地 PGVector”这一种实现
- 不承担未来外部向量引擎的兼容职责

推荐字段：

- `id BIGSERIAL PRIMARY KEY`
- `tenant_id UUID NOT NULL`
- `kb_id UUID NOT NULL`
- `search_unit_id BIGINT NOT NULL`
- `model_id UUID`
- `model_name VARCHAR(255)`
- `vector_scope VARCHAR(50) NOT NULL`
- `embedding_dimension INTEGER NOT NULL`
- `embedding VECTOR NOT NULL`
- `content_hash VARCHAR(64) NOT NULL`
- `is_active BOOLEAN DEFAULT true`
- `metadata JSONB DEFAULT '{}'::jsonb`
- `created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP`
- `updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP`

关键说明：

- `search_unit_id` 才是主关联键，不再直接绑 `chunk_id`
- `vector_scope` 一般与 `search_scope` 对齐
- `embedding` 不要在表定义里硬编码为 1536 维
- 运行上要求：同一套本地 PG 索引中的向量维度必须一致；如果未来允许多维模型并存，应按模型或索引集分区/分表管理

唯一约束建议：

- `UNIQUE (search_unit_id, model_id, vector_scope)`

索引建议：

- `HNSW (embedding vector_cosine_ops)` 用于当前 PGVector
- `INDEX (kb_id, is_active)`

### 5.4 `pg_chunk_search_unit_lexical_indexes`：当前 PostgreSQL 全文本地实现表

职责：

- 保存当前 PostgreSQL 全文检索的本地索引
- 只服务于“本地 PostgreSQL FTS”这一种实现
- 不承担未来 ES/OpenSearch 的兼容职责

推荐字段：

- `id BIGSERIAL PRIMARY KEY`
- `tenant_id UUID NOT NULL`
- `kb_id UUID NOT NULL`
- `search_unit_id BIGINT NOT NULL`
- `lexical_scope VARCHAR(50) NOT NULL`
- `language_config VARCHAR(50)`
- `search_text TEXT NOT NULL`
- `search_vector TSVECTOR`
- `content_hash VARCHAR(64) NOT NULL`
- `is_active BOOLEAN DEFAULT true`
- `metadata JSONB DEFAULT '{}'::jsonb`
- `created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP`
- `updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP`

关键说明：

- 当前 PG 路径使用 `search_vector`
- `search_text` 明确保存，避免全文检索隐式依赖 `chunks.content`

唯一约束建议：

- `UNIQUE (search_unit_id, lexical_scope)`

索引建议：

- `GIN (search_vector)`
- `INDEX (kb_id, is_active)`

### 5.5 当前阶段的后端选择策略：只使用 `.env`

职责：

- 作为当前阶段唯一启用的后端配置来源
- 控制系统默认使用哪种向量后端、哪种全文后端
- 当前推荐只做“全局切换”，不引入知识库级覆盖

当前建议环境变量：

- `DEFAULT_VECTOR_SEARCH_BACKEND`
- `DEFAULT_LEXICAL_SEARCH_BACKEND`

默认值：

- `DEFAULT_VECTOR_SEARCH_BACKEND=pg_vector`
- `DEFAULT_LEXICAL_SEARCH_BACKEND=pg_fts`

当前最佳实践建议：

- 开发阶段先只保留这一套全局配置来源
- 检索执行层统一先读 `.env`
- 不要同时让 `.env` 与知识库级 profile 在主流程里并存，避免配置来源过多

## 6. 针对不同知识库类型的投影方式

### 6.1 通用文档知识库

`chunks`

- 章节块
- 段落块
- 叶子块

`chunk_search_units`

- 一般一个叶子块对应一个 `default` 检索单元
- 可选补一个 `summary` 检索单元
- 可选补一个 `doc_summary` 检索单元（来自 `kb_doc.summary`，辅路）

`doc_summary` 设计原则：

- 来源：`knowledge_base_documents.summary`
- 定位：辅助召回，不是主召回
- 默认：关闭（需显式开启）
- 权重：低于 `default` 与结构化主投影

建议开关：

- `retrieval_config.query_analysis.enable_doc_summary_retrieval`（知识库级）
- `chat_session.config.persistent_context.enable_doc_summary_retrieval`（会话级，可选）

建议第一阶段：

- 仅做 lexical（`should_vectorize=false`）
- 命中后只做轻量加分，不直接顶替主结果

### 6.2 QA 知识库

`chunks`

- 保留现有 QA 父子 chunk 结构，用于前端展示、分块预览与最终上下文回填
- 短答案：
  - 一个 `qa_row` 叶子块
- 长答案：
  - 一个 `qa_row` 父块
  - 多个现有答案子块

说明：

- 不建议为了 `question_only / question_answer` 再额外新增一套平行“展示 chunk”
- 特别是不建议仅为了检索目的，把 `qa_question` 设计成新的展示节点并插入现有树结构
- 原因是这会直接增加前端层级复杂度，并破坏当前基于父子块的 UI 直觉
- 最佳实践是：展示结构继续由 `chunks` 承担，检索语义改在 `chunk_search_units` 层表达

`chunk_search_units`

- `question`
  - 来源：从 `qa_row` 派生
  - 内容：`question + similar_questions + 可选 category/tags`
  - 作用：问句召回主入口
  - 注意：不要带完整答案，避免问句向量被长答案稀释
- `answer`
  - 短答案：
    - 从 `qa_row` 派生一个 `answer` 单元
  - 长答案：
    - 从现有答案子块派生多个 `answer` 单元
  - 内容：
    - 建议保留问题锚点，例如“问题 + 答案”或“问题 + 答案片段”
  - 作用：答案语义召回入口

模式落地：

- `question_only`
  - 仅启用 `question` search unit
  - `answer` search unit 不生成，或生成但不激活
- `question_answer`
  - 同时启用 `question` 与 `answer` search unit
  - 检索后按 `content_group_id` 聚合
  - 推荐聚合方式：
    - `final_score = question_score * query_weight + answer_score * answer_weight`
  - 其中：
    - `question_score` 取该 QA 下 `question` 单元最高分
    - `answer_score` 取该 QA 下 `answer` 单元最高分
  - 推荐先取 `max`，不建议对多个答案片段直接求和，避免“切得越碎分越高”

优点：

- “问句优先 / 问答联合”不再通过修改 chunk 本体实现
- 不推翻当前 QA 父子块展示协议
- 前端数据视图与 chunk 预览可以继续围绕现有 chunk 树工作
- 只调整搜索单元与聚合策略即可支持权重检索

### 6.3 表格型知识库

`chunks`

- 行上下文块
- 行级父子块

`chunk_search_units`

- `row`
- `row_group`
- `row_fragment`
- 可选 `summary`

不建议直接让所有结构列都膨胀成独立 chunk，可保留在 `search_text` 投影阶段做字段组合。

`row_group` 当前定义：

- 不是跨多行真实聚合结果
- 而是围绕单行构造的增强上下文投影
- 主要补：
  - 表头
  - 维度字段
  - 指标字段
  - 当前行筛选值

这样可以先解决“命中某一行，但模型不知道这一行在表里代表什么”的问题。

后续如果要做真实多行 `row_group`，建议：

- 优先按显式 `group_by_columns`
- 或按 schema 中的 `dimension / identifier / entity` 字段推断 group key
- 不建议按相邻行直接拼接

### 6.4 网页同步知识库

`chunks`

- 页面正文块
- 页面摘要块

`chunk_search_units`

- `page_title`
- `page_body`
- 可选 `summary`

后续切 ES 时，网页型知识天然受益最大。

## 7. 中文全文检索的最佳实践提醒

如果当前使用 PostgreSQL 自身全文检索能力，必须正视中文分词问题。

结论：

- 默认 `tsvector` 方案对中文并不是最佳体验
- 如果继续坚持 PG 路线，建议引入中文分词扩展或自定义分词策略
- 如果后续要求更高，ES/OpenSearch 往往更适合作为中文全文检索引擎

因此，本方案将全文检索层独立为 `chunk_search_unit_lexical_indexes`，就是为了后续平滑切换到 ES，而不推翻主表设计。

## 8. 检索执行链路（推荐）

推荐统一检索链路：

1. 查询向量检索投影
  基于 `pg_chunk_search_unit_vectors` 或外部向量引擎
2. 查询全文检索投影
  基于 `pg_chunk_search_unit_lexical_indexes` 或外部全文引擎
3. 结果统一映射回 `search_unit_id`
4. 再按 `content_group_id` 聚合
5. 根据 `search_scope` / 权重做融合
  对 QA 来说：
  - `content_group_id` 语义上应对应同一条 QA
  - `question_answer` 模式下，应在这一层合并问题侧与答案侧分数
6. 最终回捞 `chunks`
  只返回 `display_enabled = true` 的 canonical chunk

这样做的好处：

- 各检索引擎返回的是统一的逻辑单位
- 各知识库类型都按同一套路聚合
- 返回给前端 / 大模型的始终是 canonical content，而不是检索投影文本

补充说明：

- 当前如果使用 PostgreSQL，本地物理索引来自 `pg_chunk_search_unit_vectors` 与 `pg_chunk_search_unit_lexical_indexes`
- 未来如果改为 Qdrant / Elasticsearch，推荐做法是“替换索引实现层”，而不是要求与 PG 长期并存
- 因此，本方案稳定层是：
  - `chunks`
  - `chunk_search_units`
  - `.env` 中的全局默认后端配置

## 9. 本方案对 `chunk_relations` 的最终态度

最终建议：

- 当前主方案：不引入 `chunk_relations`
- 仅在以下场景再考虑补充：
  - 需要表达复杂跨块引用关系
  - 需要把知识图谱边映射到 chunk 间关系
  - 需要做冷路径分析、导航、知识编排

如果未来真的需要，可增加：

- `chunk_links`
但必须明确：
  - 它不进入主检索热路径
  - 它只作为扩展关系层

这比现在就加一张通用 `chunk_relations` 更稳，也更符合性能最佳实践。

## 10. 对当前旧表的调整建议

### 10.1 `chunks`

保留，但增加：

- `content_hash`
- `source_type`
- `display_enabled`
- `content_group_id`

### 10.2 废弃旧 `embeddings`

当前旧设计问题：

- 同时承担向量与全文职责
- 直接绑 `chunk_id`
- 难以兼容未来多 backend
- 不利于 QA/表格等多路检索投影

建议：

- 用 `pg_chunk_search_unit_vectors`
- 用 `pg_chunk_search_unit_lexical_indexes`
替代

### 10.3 新增 `chunk_search_units`

这是本次最核心的新表，必须补。

## 11. 最终结论

当前项目在开发阶段，最佳实践不是继续围绕：

- `chunks`
- `embeddings(embedding + search_vector)`

做局部修补。

最佳实践应该升级为：

1. `chunks`
  内容主表
2. `chunk_search_units`
  检索投影表
3. `pg_chunk_search_unit_vectors`
  当前 PGVector 本地实现层
4. `pg_chunk_search_unit_lexical_indexes`
  当前 PostgreSQL FTS 本地实现层
5. 当前主流程读取 `.env` 中的默认后端配置
6. 后续按需新增知识图谱相关表
  - `kg_entities`
  - `kg_relations`
  - `kg_chunk_links`

并明确：

- 现在支持 PGVector + PostgreSQL FTS
- 后续切换 Qdrant / ES 时，优先替换索引实现层，而不是推翻主表与检索投影设计
- QA / 表格 / 网页 / 通用文档统一接入
- 知识图谱以后作为独立层接入
- `chunk_relations` 当前不进入主方案

对于 QA 的专项结论，再单独强调一次：

- 现有 QA 父子 chunk 结构可以保留
- 问题 / 答案联合索引不应通过新增平行展示 chunk 来实现
- 正确做法是在 `chunk_search_units` 层生成 `question` 与 `answer` 两类检索投影
- `question_only / question_answer` 的差异体现在 search unit 的生成 / 启用策略，而不是前端 chunk 树结构

这才是更稳、更可演进、也更符合未来多引擎兼容目标的方案。

## 13. `kb_doc.summary` 是否已用于检索：当前结论与方案

### 13.1 当前结论

- 当前主代码链路中，`kb_doc.summary` 已不再只是文档列表展示字段；
- 已进入稳定检索主链中的辅路；
- 当前已通过：
  - `chunk_search_units(search_scope=doc_summary)` 正式索引投影
  - 旧索引场景的热路径兜底
  接入检索流程。

### 13.2 为什么仍然值得接入

- 它具备文档级主题压缩信息，适合做粗粒度召回扩展；
- 对“先找文档再找块”的路由场景有价值；
- 但信息密度低于正文块，不能取代主检索投影。

### 13.3 标准接入方式（辅路）

1. 投影层
  - 在 `chunk_search_units` 增加 `search_scope=doc_summary`
  - 每个 `kb_doc` 最多一条，`is_primary=false`
2. 索引层
  - 第一阶段只写全文索引
  - 第二阶段按效果决定是否补向量索引
3. 检索融合层
  - `doc_summary` 设置低权重
  - 仅用于扩大候选集或少量分数加成
4. 开关治理
  - 默认关闭
  - 当前已支持知识库级开关
  - 当前聊天侧可通过会话配置间接控制是否注入摘要上下文
  - 已支持调试可视化与灰度回滚
5. 质量门槛
  - 当前已在写入投影前做摘要清洗
  - 当前已过滤低信息占位摘要，例如“暂无摘要 / 待补充”
  - 当前已增加最小长度、最小 token、最小唯一词项门槛
  - 当前热路径兜底与正式索引投影已复用同一套校验逻辑

### 13.4 验收指标（建议）

- 命中率提升：是否提高“正确文档进入候选集”的比例
- 精度影响：Top-K 精确率是否下降
- 幻觉率变化：回答是否更易偏离原文
- token 成本：最终上下文长度是否明显增加
