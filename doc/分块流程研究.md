# 分块流程研究：FastGPT 架构分析与 本项目Python 最佳实践

## 一、FastGPT 架构分析

### 1.1 核心设计思想

FastGPT 采用**多阶段异步队列**架构，将文档处理流程拆分为三个独立阶段：

```
文件上传 → 解析队列（parse） → 训练队列（chunk/qa/...）
```

**关键特性**：
- **异步解耦**：每个阶段独立，互不阻塞
- **Worker 子线程**：CPU 密集型任务（文件解析、分块）在子线程执行
- **Mongo Watch**：利用数据库变更流自动触发任务
- **任务重试**：通过 `retryCount` 和 `lockTime` 实现失败重试
- **资源隔离**：解析和训练分离，可独立扩展

### 1.2 完整调用链路

```
[1] API 层：创建集合（不传 rawText）
    ↓
[2] 插入解析任务：mode=parse → dataset_trainings 表
    ↓
[3] Mongo Watch 触发：监听 insert 事件 → datasetParseQueue()
    ↓
[4] 解析队列消费：
    - 读取文件（readFile Worker 子线程）
    - 可选 LLM 段落优化
    - 分块（text2Chunks Worker 子线程）
    ↓
[5] 插入训练任务：mode=chunk/qa/... → dataset_trainings 表
    ↓
[6] 训练队列消费：
    - 向量化（embedding）
    - 生成摘要（summary）
    - 提取问题（qa）
    - 提取关键词（keywords）
```

### 1.3 Worker 子线程机制（Node.js）

**Node.js 为什么使用 Worker Threads？**

1. **CPU 密集型任务不阻塞主线程**：
   - 文件解析（PDF/Word/Excel）：CPU 密集
   - 文本分块（正则、递归切分）：CPU 密集
   - 主线程继续处理 HTTP 请求

2. **资源隔离**：
   - Worker 线程崩溃不影响主进程
   - 可限制 Worker 内存使用

3. **并发控制**：
   - 可同时运行多个 Worker
   - 避免单个大文件阻塞所有请求

**实现方式**：
- Node.js `worker_threads` 模块（真正的多线程）
- 主线程通过 `postMessage` 传递参数
- Worker 线程通过 `parentPort.postMessage` 返回结果
- 主线程 `await` Promise 获取结果

**注意**：Node.js 的 Worker Threads 是真正的多线程，可以并行执行 CPU 密集任务。

### 1.4 核心优势总结

| 优势 | 说明 | 价值 |
|------|------|------|
| **异步解耦** | 上传、解析、训练三阶段独立 | 用户上传后立即返回，后台异步处理 |
| **Worker Threads** | CPU 密集任务不阻塞主线程（Node.js 真多线程） | 主线程继续处理 HTTP 请求，提升吞吐量 |
| **Mongo Watch** | 数据库变更流自动触发 | 无需轮询，实时响应 |
| **任务重试** | 失败自动重试 | 提高可靠性 |
| **资源隔离** | 解析和训练分离 | 可独立扩展，避免相互影响 |
| **可观测性** | 任务状态存储在数据库 | 便于监控、调试、恢复 |

## 二、本项目Python 最佳实践方案

### 2.0 Python GIL 与并发策略

#### 2.0.1 GIL（全局解释器锁）问题

**什么是 GIL？**
- CPython 解释器的互斥锁
- 确保同一时刻仅一个线程执行 Python 字节码
- 即使在多核 CPU 上，Python 多线程也无法实现真正的并行

**GIL 的影响**：
```python

import threading

def parse_pdf(file_buffer):
    # CPU 密集型任务
    # 由于 GIL，多个线程无法并行执行
    pass

threads = []
for file in files:
    t = threading.Thread(target=parse_pdf, args=(file,))
    threads.append(t)
    t.start()

# 结果：多个线程轮流执行，总时间 = 单线程时间
```

#### 2.0.2 Python 并发策略对比

| 方案 | 适用场景 | GIL 影响 | 并行能力 | 通信成本 |
|------|----------|----------|----------|----------|
| **多线程 (threading)** | I/O 密集（网络、文件读写） | 有影响，但 I/O 时释放 GIL | ❌ 无法并行 CPU 任务 | 低（共享内存） |
| **多进程 (multiprocessing)** | CPU 密集（解析、计算） | 无影响（独立进程） | ✅ 真正并行 | 高（进程间通信） |
| **异步 (asyncio)** | I/O 密集（大量并发请求） | 有影响（单线程） | ❌ 无法并行 CPU 任务 | 低（协程切换） |
| **Celery 任务队列** | 异步任务、分布式 | 无影响（独立 Worker） | ✅ 真正并行 | 中（消息队列） |

#### 2.0.3 FastGPT vs Python 对比

| 维度 | FastGPT (Node.js) | Python 方案 |
|------|-------------------|-------------|
| **并发模型** | Worker Threads（真多线程） | Multiprocessing（多进程） |
| **GIL 限制** | 无 GIL | 有 GIL，必须用多进程 |
| **CPU 密集任务** | Worker Threads 并行 | 多进程并行 |
| **I/O 密集任务** | 异步 I/O | asyncio 或多线程 |
| **通信成本** | 低（共享内存） | 高（进程间通信） |
| **内存开销** | 低（线程共享内存） | 高（每个进程独立内存） |

**结论**：
- **Node.js**：可以用 Worker Threads 处理 CPU 密集任务
- **Python**：必须用 Multiprocessing 处理 CPU 密集任务

#### 2.3.2 解析任务（多策略）

```python
# tasks/document_tasks.py
from celery import shared_task
from multiprocessing import Process, Queue
import time

@shared_task(bind=True, max_retries=3)
def parse_document_task(
    self, 
    document_id: str, 
    file_id: str, 
    file_extension: str,
    parse_strategy: str = "auto"
):
    """
    解析文档任务（支持多种解析策略）
    """
    try:
        # 1. 更新文档状态
        update_document_status(document_id, DocumentStatus.PARSING)
        
        # 2. 下载文件
        file_buffer = download_file_from_storage(file_id)
        
        # 3. 自动选择解析策略
        if parse_strategy == "auto":
            parse_strategy = auto_select_parse_strategy(
                file_buffer, file_extension
            )
        
        # 4. 根据策略选择解析方法
        if parse_strategy == "mineru":
            raw_text, metadata = parse_with_mineru(file_buffer, file_extension)
        elif parse_strategy == "vision":
            raw_text, metadata = parse_with_vision_model(file_buffer, file_extension)
        elif parse_strategy == "ocr":
            raw_text, metadata = parse_with_ocr(file_buffer, file_extension)
        else:  # basic
            raw_text, metadata = parse_basic(file_buffer, file_extension)
        
        # 5. 保存原始文本和元数据
        await save_raw_text(document_id, raw_text, metadata)
        
        # 6. 触发分块任务
        chunk_document_task.delay(document_id, raw_text, metadata)
        
        # 7. 更新状态
        update_document_status(document_id, DocumentStatus.CHUNKING)
        
    except Exception as e:
        update_document_status(document_id, DocumentStatus.PARSE_FAILED)
        self.retry(exc=e, countdown=60)

def auto_select_parse_strategy(file_buffer: bytes, extension: str) -> str:
    """
    自动选择解析策略
    
    规则：
    1. 扫描件 PDF → OCR
    2. 复杂布局 PDF → MinerU 或 Vision
    3. 简单文档 → Basic
    """
    if extension == ".pdf":
        # 检测是否为扫描件
        if is_scanned_pdf(file_buffer):
            return "ocr"
        
        # 检测布局复杂度
        if is_complex_layout(file_buffer):
            return "mineru"  # 或 "vision"
        
        return "basic"
    
    elif extension in [".jpg", ".jpeg", ".png"]:
        return "ocr"
    
    else:
        return "basic"

def parse_basic(file_buffer: bytes, extension: str) -> tuple[str, dict]:
    """
    基础解析（使用多进程避免 GIL）
    """
    result_queue = Queue()
    process = Process(
        target=parse_basic_worker,
        args=(file_buffer, extension, result_queue)
    )
    process.start()
    process.join(timeout=300)
    
    if process.is_alive():
        process.terminate()
        raise TimeoutError("基础解析超时")
    
    return result_queue.get()

def parse_basic_worker(file_buffer: bytes, extension: str, result_queue: Queue):
    """基础解析 Worker"""
    try:
        if extension == ".pdf":
            text = parse_pdf_basic(file_buffer)
        elif extension in [".docx", ".doc"]:
            text = parse_word(file_buffer)
        elif extension in [".xlsx", ".xls"]:
            text = parse_excel(file_buffer)
        elif extension == ".md":
            text = parse_markdown(file_buffer)
        else:
            text = file_buffer.decode("utf-8")
        
        metadata = {"parse_method": "basic"}
        result_queue.put((text, metadata))
    except Exception as e:
        result_queue.put((None, {"error": str(e)}))

def parse_with_mineru(file_buffer: bytes, extension: str) -> tuple[str, dict]:
    """
    使用 MinerU 解析（高质量 PDF 解析）
    
    MinerU 特点：
    - 保留文档结构（标题、段落、表格、公式）
    - 支持复杂布局
    - 输出 Markdown 格式
    """
    result_queue = Queue()
    process = Process(
        target=parse_mineru_worker,
        args=(file_buffer, extension, result_queue)
    )
    process.start()
    process.join(timeout=600)  # MinerU 较慢，10 分钟超时
    
    if process.is_alive():
        process.terminate()
        raise TimeoutError("MinerU 解析超时")
    
    return result_queue.get()

def parse_mineru_worker(file_buffer: bytes, extension: str, result_queue: Queue):
    """MinerU 解析 Worker"""
    try:
        from magic_pdf.pipe.UNIPipe import UNIPipe
        from magic_pdf.rw.DiskReaderWriter import DiskReaderWriter
        
        # 保存临时文件
        temp_path = save_temp_file(file_buffer, extension)
        
        # 使用 MinerU 解析
        reader = DiskReaderWriter(os.path.dirname(temp_path))
        pipe = UNIPipe(temp_path, reader)
        pipe.pipe_classify()
        pipe.pipe_parse()
        
        # 获取 Markdown 结果
        markdown_text = pipe.pipe_mk_markdown()
        
        # 提取元数据
        metadata = {
            "parse_method": "mineru",
            "layout_info": pipe.get_layout_info(),
            "table_count": len(pipe.get_tables()),
            "image_count": len(pipe.get_images())
        }
        
        result_queue.put((markdown_text, metadata))
        
    except Exception as e:
        result_queue.put((None, {"error": str(e)}))
    finally:
        cleanup_temp_file(temp_path)

def parse_with_vision_model(file_buffer: bytes, extension: str) -> tuple[str, dict]:
    """
    使用视觉大模型解析（适合复杂布局、图表密集文档）
    
    优势：
    - 理解图表、图片内容
    - 处理复杂布局
    - 提取视觉信息
    """
    # 将文档转为图片
    images = convert_to_images(file_buffer, extension)
    
    # 调用视觉大模型（如 GPT-4V、Claude 3、Qwen-VL）
    texts = []
    for image in images:
        prompt = """
        请分析这张文档图片，提取所有文本内容。
        要求：
        1. 保留文档结构（标题、段落、列表）
        2. 识别表格并转为 Markdown 表格
        3. 描述图表内容
        4. 保持原文顺序
        """
        text = call_vision_model(image, prompt)
        texts.append(text)
    
    full_text = "\n\n".join(texts)
    metadata = {
        "parse_method": "vision_model",
        "page_count": len(images)
    }
    
    return full_text, metadata

def parse_with_ocr(file_buffer: bytes, extension: str) -> tuple[str, dict]:
    """
    使用 OCR 解析（适合扫描件、图片）
    
    技术栈：
    - PaddleOCR：中文 OCR
    - Tesseract：通用 OCR
    - 版面分析：LayoutParser
    """
    result_queue = Queue()
    process = Process(
        target=parse_ocr_worker,
        args=(file_buffer, extension, result_queue)
    )
    process.start()
    process.join(timeout=600)
    
    if process.is_alive():
        process.terminate()
        raise TimeoutError("OCR 解析超时")
    
    return result_queue.get()

def parse_ocr_worker(file_buffer: bytes, extension: str, result_queue: Queue):
    """OCR 解析 Worker"""
    try:
        from paddleocr import PaddleOCR
        import layoutparser as lp
        
        # 1. 版面分析
        layout_model = lp.Detectron2LayoutModel(
            'lp://PubLayNet/faster_rcnn_R_50_FPN_3x/config'
        )
        
        # 转为图片
        images = convert_to_images(file_buffer, extension)
        
        # 2. OCR 识别
        ocr = PaddleOCR(use_angle_cls=True, lang='ch')
        
        texts = []
        for image in images:
            # 版面分析
            layout = layout_model.detect(image)
            
            # 按阅读顺序排序
            layout = layout.sort(key=lambda x: (x.coordinates[1], x.coordinates[0]))
            
            # OCR 识别
            for block in layout:
                block_image = image.crop(block.coordinates)
                result = ocr.ocr(block_image)
                block_text = "\n".join([line[1][0] for line in result[0]])
                texts.append(block_text)
        
        full_text = "\n\n".join(texts)
        metadata = {
            "parse_method": "ocr",
            "page_count": len(images),
            "layout_blocks": len(layout)
        }
        
        result_queue.put((full_text, metadata))
        
    except Exception as e:
        result_queue.put((None, {"error": str(e)}))
```

#### 2.3.3 分块任务

```python
@shared_task(bind=True, max_retries=3)
def chunk_document_task(self, document_id: str, raw_text: str):
    """
    分块任务
    
    选择：
    1. 单进程：适合简单分块（正则、固定长度）
    2. 多进程：适合复杂分块（递归、语义分块）
    3. 异步：适合需要调用 LLM 的分块
    """
    try:
        # 1. 获取分块配置
        config = get_chunk_config(document_id)
        
        # 2. 执行分块
        if config.use_llm_paragraph:
            # 需要调用 LLM，使用异步
            chunks = await chunk_with_llm(raw_text, config)
        elif config.chunk_strategy == "semantic":
            # 语义分块，CPU 密集，使用多进程
            chunks = chunk_with_multiprocess(raw_text, config)
        else:
            # 简单分块，单进程即可
            chunks = chunk_simple(raw_text, config)
        
        # 3. 保存分块结果
        segment_ids = await save_segments(document_id, chunks)
        
        # 4. 触发训练任务（向量化等）
        train_document_task.delay(document_id, segment_ids)
        
        # 5. 更新状态
        update_document_status(document_id, DocumentStatus.TRAINING)
        
    except Exception as e:
        update_document_status(document_id, DocumentStatus.CHUNK_FAILED)
        self.retry(exc=e, countdown=60)

def chunk_with_multiprocess(raw_text: str, config: dict) -> List[str]:
    """
    使用多进程分块（适合 CPU 密集型）
    """
    result_queue = Queue()
    process = Process(
        target=chunk_worker,
        args=(raw_text, config, result_queue)
    )
    process.start()
    process.join(timeout=180)  # 3 分钟超时
    
    if process.is_alive():
        process.terminate()
        raise TimeoutError("分块超时")
    
    return result_queue.get()

def chunk_worker(raw_text: str, config: dict, result_queue: Queue):
    """
    分块 Worker（在子进程中执行）
    """
    from llama_index.core.node_parser import MarkdownElementNodeParser
    
    parser = MarkdownElementNodeParser()
    nodes = parser.get_nodes_from_documents([Document(text=raw_text)])
    chunks = [node.text for node in nodes]
    
    result_queue.put(chunks)
```

```python
@shared_task(bind=True, max_retries=3)
def build_knowledge_graph_task(self, document_id: str, segment_ids: List[str]):
    """
    构建知识图谱任务
    
    策略：按分块构建，最后合并
    
    流程：
    1. 从每个分块中提取实体和关系
    2. 实体消歧和合并
    3. 构建图结构
    4. 存储到 Neo4j
    5. 在 PostgreSQL 中缓存实体和关系
    """
    try:
        update_document_status(document_id, DocumentStatus.BUILDING_KG)
        
        # 1. 并行提取实体和关系
        from celery import group
        
        extract_jobs = group(
            extract_entities_and_relations_task.s(segment_id)
            for segment_id in segment_ids
        )
        results = extract_jobs.apply_async().get()
        
        # 2. 合并所有实体和关系
        all_entities = []
        all_relations = []
        for entities, relations in results:
            all_entities.extend(entities)
            all_relations.extend(relations)
        
        # 3. 实体消歧和合并
        merged_entities = merge_entities(all_entities)
        merged_relations = merge_relations(all_relations, merged_entities)
        
        # 4. 存储到 Neo4j
        store_to_neo4j(document_id, merged_entities, merged_relations)
        
        # 5. 在 PostgreSQL 中缓存（用于快速查询）
        cache_to_postgres(document_id, merged_entities, merged_relations)
        
        # 6. 更新文档状态
        update_document_kg_status(document_id, "completed")
        
    except Exception as e:
        update_document_status(document_id, DocumentStatus.KG_FAILED)
        self.retry(exc=e, countdown=60)

@shared_task
def extract_entities_and_relations_task(segment_id: str) -> tuple[List[dict], List[dict]]:
    """
    从单个分块中提取实体和关系
    
    方法：
    1. LLM 提取（GPT-4/Claude）
    2. NER 模型（SpaCy/BERT-NER）
    3. 规则提取
    """
    segment = get_segment(segment_id)
    
    # 方法 1：使用 LLM 提取（推荐，效果最好）
    entities, relations = extract_with_llm(segment.text)
    
    # 方法 2：使用 NER 模型（备选，速度快）
    # entities, relations = extract_with_ner(segment.text)
    
    # 添加来源信息
    for entity in entities:
        entity["segment_id"] = segment_id
        entity["document_id"] = segment.document_id
    
    for relation in relations:
        relation["segment_id"] = segment_id
        relation["document_id"] = segment.document_id
    
    return entities, relations

def extract_with_llm(text: str) -> tuple[List[dict], List[dict]]:
    """
    使用 LLM 提取实体和关系
    
    Prompt 工程：
    - 明确实体类型（人物、组织、地点、概念等）
    - 明确关系类型（属于、包含、相关等）
    - 要求结构化输出（JSON）
    """
    prompt = """
    请从以下文本中提取实体和关系，以 JSON 格式返回。
    
    实体类型：
    - PERSON: 人物
    - ORGANIZATION: 组织/公司
    - LOCATION: 地点
    - CONCEPT: 概念/术语
    - PRODUCT: 产品
    - EVENT: 事件
    - DATE: 日期
    
    关系类型：
    - IS_A: 是一个（类别关系）
    - PART_OF: 属于/包含
    - RELATED_TO: 相关
    - LOCATED_IN: 位于
    - WORKS_FOR: 工作于
    - CREATED_BY: 创建者
    - HAPPENED_AT: 发生于
    
    返回格式：
    {
      "entities": [
        {"name": "实体名称", "type": "实体类型", "description": "简短描述"}
      ],
      "relations": [
        {"source": "源实体", "target": "目标实体", "type": "关系类型", "description": "关系描述"}
      ]
    }
    
    文本：
    {text}
    """
    
    response = call_llm_api(prompt.format(text=text))
    result = json.loads(response)
    
    return result["entities"], result["relations"]

def merge_entities(entities: List[dict]) -> List[dict]:
    """
    实体消歧和合并
    
    策略：
    1. 完全匹配：名称相同 → 合并
    2. 模糊匹配：名称相似（编辑距离） → 合并
    3. 语义匹配：使用 embedding 计算相似度 → 合并
    """
    # 按名称分组
    entity_groups = {}
    for entity in entities:
        name = entity["name"].lower().strip()
        if name not in entity_groups:
            entity_groups[name] = []
        entity_groups[name].append(entity)
    
    # 合并同名实体
    merged = []
    for name, group in entity_groups.items():
        # 取第一个作为主实体
        main_entity = group[0].copy()
        
        # 合并描述
        descriptions = [e.get("description", "") for e in group if e.get("description")]
        if descriptions:
            main_entity["description"] = " | ".join(set(descriptions))
        
        # 记录来源分块
        main_entity["segment_ids"] = list(set(
            e["segment_id"] for e in group
        ))
        
        # 记录出现次数
        main_entity["frequency"] = len(group)
        
        merged.append(main_entity)
    
    return merged

def merge_relations(relations: List[dict], entities: List[dict]) -> List[dict]:
    """
    关系合并和验证
    
    策略：
    1. 验证实体存在性
    2. 去重
    3. 合并相同关系
    """
    # 构建实体名称索引
    entity_names = {e["name"].lower().strip() for e in entities}
    
    # 过滤无效关系（实体不存在）
    valid_relations = []
    for rel in relations:
        source = rel["source"].lower().strip()
        target = rel["target"].lower().strip()
        
        if source in entity_names and target in entity_names:
            valid_relations.append(rel)
    
    # 去重和合并
    relation_key_map = {}
    for rel in valid_relations:
        key = (
            rel["source"].lower().strip(),
            rel["target"].lower().strip(),
            rel["type"]
        )
        
        if key not in relation_key_map:
            relation_key_map[key] = rel.copy()
            relation_key_map[key]["segment_ids"] = [rel["segment_id"]]
            relation_key_map[key]["frequency"] = 1
        else:
            relation_key_map[key]["segment_ids"].append(rel["segment_id"])
            relation_key_map[key]["frequency"] += 1
    
    return list(relation_key_map.values())

def store_to_neo4j(document_id: str, entities: List[dict], relations: List[dict]):
    """
    存储到 Neo4j 图数据库
    
    节点类型：
    - Document: 文档节点
    - Chunk: 分块节点
    - Entity: 实体节点（按类型细分）
    
    关系类型：
    - CONTAINS: 文档包含分块
    - MENTIONS: 分块提及实体
    - 业务关系: IS_A, PART_OF, RELATED_TO 等
    """
    from neo4j import GraphDatabase
    
    driver = GraphDatabase.driver(
        NEO4J_URI,
        auth=(NEO4J_USER, NEO4J_PASSWORD)
    )
    
    with driver.session() as session:
        # 1. 创建文档节点
        session.run("""
            MERGE (d:Document {id: $document_id})
            SET d.created_at = datetime()
        """, document_id=document_id)
        
        # 2. 创建实体节点
        for entity in entities:
            session.run("""
                MERGE (e:Entity {name: $name, type: $type})
                SET e.description = $description,
                    e.frequency = $frequency
                WITH e
                MATCH (d:Document {id: $document_id})
                MERGE (d)-[:HAS_ENTITY]->(e)
            """, 
                name=entity["name"],
                type=entity["type"],
                description=entity.get("description", ""),
                frequency=entity.get("frequency", 1),
                document_id=document_id
            )
            
            # 创建实体与分块的关系
            for segment_id in entity.get("segment_ids", []):
                session.run("""
                    MATCH (e:Entity {name: $name})
                    MERGE (c:Chunk {id: $segment_id})
                    MERGE (c)-[:MENTIONS]->(e)
                """,
                    name=entity["name"],
                    segment_id=segment_id
                )
        
        # 3. 创建关系
        for relation in relations:
            session.run("""
                MATCH (source:Entity {name: $source})
                MATCH (target:Entity {name: $target})
                MERGE (source)-[r:%s]->(target)
                SET r.description = $description,
                    r.frequency = $frequency
            """ % relation["type"],
                source=relation["source"],
                target=relation["target"],
                description=relation.get("description", ""),
                frequency=relation.get("frequency", 1)
            )
    
    driver.close()

def cache_to_postgres(document_id: str, entities: List[dict], relations: List[dict]):
    """
    在 PostgreSQL 中缓存实体和关系
    
    目的：
    1. 快速查询（不依赖 Neo4j）
    2. 支持全文检索
    3. 支持 SQL 聚合分析
    """
    # 保存实体
    for entity in entities:
        save_entity_to_postgres(
            document_id=document_id,
            name=entity["name"],
            type=entity["type"],
            description=entity.get("description", ""),
            frequency=entity.get("frequency", 1),
            segment_ids=entity.get("segment_ids", [])
        )
    
    # 保存关系
    for relation in relations:
        save_relation_to_postgres(
            document_id=document_id,
            source=relation["source"],
            target=relation["target"],
            type=relation["type"],
            description=relation.get("description", ""),
            frequency=relation.get("frequency", 1),
            segment_ids=relation.get("segment_ids", [])
        )
```

### 4.3 核心要点

1. **文件解析必须用多进程**：Python GIL 限制，CPU 密集型任务必须多进程

```python
class SearchStrategy(str, Enum):
    VECTOR_ONLY = "vector_only"           # 纯向量检索
    KG_ONLY = "kg_only"                   # 纯知识图谱检索
    HYBRID = "hybrid"                     # 混合检索（推荐）
    VECTOR_THEN_KG = "vector_then_kg"     # 先向量后图谱
    KG_THEN_VECTOR = "kg_then_vector"     # 先图谱后向量

@router.post("/search")
async def search_knowledge_base(
    kb_id: UUID,
    query: str,
    strategy: SearchStrategy = SearchStrategy.HYBRID,
    top_k: int = 10,
    current_user: User = Depends(get_current_user)
):
    """
    知识库检索 API
    
    支持多种检索策略：
    1. 纯向量检索：适合语义相似度搜索
    2. 纯知识图谱检索：适合实体关系查询
    3. 混合检索：结合两者优势（推荐）
    """
    if strategy == SearchStrategy.VECTOR_ONLY:
        results = await vector_search(kb_id, query, top_k)
    
    elif strategy == SearchStrategy.KG_ONLY:
        results = await kg_search(kb_id, query, top_k)
    
    elif strategy == SearchStrategy.HYBRID:
        results = await hybrid_search(kb_id, query, top_k)
    
    elif strategy == SearchStrategy.VECTOR_THEN_KG:
        results = await vector_then_kg_search(kb_id, query, top_k)
    
    elif strategy == SearchStrategy.KG_THEN_VECTOR:
        results = await kg_then_vector_search(kb_id, query, top_k)
    
    return {"data": results, "total": len(results)}
```

#### 2.6.2 纯向量检索

```python
async def vector_search(kb_id: UUID, query: str, top_k: int) -> List[dict]:
    """
    纯向量检索
    
    流程：
    1. 查询向量化
    2. 向量相似度搜索（cosine/dot product）
    3. 返回最相似的分块
    """
    # 1. 查询向量化
    query_embedding = call_embedding_api(query)
    
    # 2. 向量检索
    stmt = select(
        Segment.id,
        Segment.text,
        Segment.metadata,
        (1 - func.cosine_distance(Embedding.embedding, query_embedding)).label("score")
    ).join(
        Embedding, Segment.id == Embedding.segment_id
    ).where(
        Segment.kb_id == kb_id
    ).order_by(
        text("score DESC")
    ).limit(top_k)
    
    results = await session.execute(stmt)
    
    return [
        {
            "segment_id": row.id,
            "text": row.text,
            "metadata": row.metadata,
            "score": row.score,
            "source": "vector"
        }
        for row in results
    ]
```

#### 2.6.3 纯知识图谱检索

```python
async def kg_search(kb_id: UUID, query: str, top_k: int) -> List[dict]:
    """
    纯知识图谱检索
    
    流程：
    1. 实体识别（从查询中提取实体）
    2. 图谱查询（查找相关实体和关系）
    3. 返回相关分块
    """
    # 1. 从查询中提取实体
    entities = extract_entities_from_query(query)
    
    if not entities:
        return []
    
    # 2. 在 Neo4j 中查询相关实体和关系
    from neo4j import GraphDatabase
    
    driver = GraphDatabase.driver(NEO4J_URI, auth=(NEO4J_USER, NEO4J_PASSWORD))
    
    with driver.session() as session:
        # 查询：找到提及这些实体的分块
        cypher = """
        MATCH (d:Document {id: $kb_id})-[:HAS_ENTITY]->(e:Entity)
        WHERE e.name IN $entity_names
        MATCH (c:Chunk)-[:MENTIONS]->(e)
        RETURN DISTINCT c.id AS segment_id, 
               collect(DISTINCT e.name) AS mentioned_entities,
               count(DISTINCT e) AS entity_count
        ORDER BY entity_count DESC
        LIMIT $top_k
        """
        
        result = session.run(
            cypher,
            kb_id=str(kb_id),
            entity_names=[e["name"] for e in entities],
            top_k=top_k
        )
        
        segment_ids = [record["segment_id"] for record in result]
    
    driver.close()
    
    # 3. 从 PostgreSQL 获取分块内容
    if not segment_ids:
        return []
    
    stmt = select(Segment).where(Segment.id.in_(segment_ids))
    segments = await session.execute(stmt)
    
    return [
        {
            "segment_id": seg.id,
            "text": seg.text,
            "metadata": seg.metadata,
            "score": 1.0,  # 知识图谱匹配
            "source": "kg",
            "matched_entities": entities
        }
        for seg in segments.scalars()
    ]
```

#### 2.6.4 混合检索（推荐）

```python
async def hybrid_search(kb_id: UUID, query: str, top_k: int) -> List[dict]:
    """
    混合检索：向量检索 + 知识图谱检索 + RRF 融合
    
    流程：
    1. 并行执行向量检索和知识图谱检索
    2. 使用 RRF（Reciprocal Rank Fusion）融合结果
    3. 返回融合后的 top_k 结果
    
    优势：
    - 向量检索：捕捉语义相似度
    - 知识图谱检索：捕捉实体关系
    - RRF 融合：平衡两者权重
    """
    # 1. 并行执行两种检索
    vector_results = await vector_search(kb_id, query, top_k * 2)
    kg_results = await kg_search(kb_id, query, top_k * 2)
    
    # 2. RRF 融合
    k = 60  # RRF 参数
    scores = {}
    
    # 向量检索结果
    for rank, result in enumerate(vector_results, 1):
        segment_id = result["segment_id"]
        scores[segment_id] = scores.get(segment_id, 0) + 1 / (k + rank)
    
    # 知识图谱检索结果
    for rank, result in enumerate(kg_results, 1):
        segment_id = result["segment_id"]
        scores[segment_id] = scores.get(segment_id, 0) + 1 / (k + rank)
    
    # 3. 排序并获取 top_k
    top_segment_ids = sorted(scores.keys(), key=lambda x: scores[x], reverse=True)[:top_k]
    
    # 4. 获取完整信息
    stmt = select(Segment).where(Segment.id.in_(top_segment_ids))
    segments = await session.execute(stmt)
    segment_map = {seg.id: seg for seg in segments.scalars()}
    
    # 5. 构建结果
    results = []
    for segment_id in top_segment_ids:
        segment = segment_map.get(segment_id)
        if segment:
            results.append({
                "segment_id": segment.id,
                "text": segment.text,
                "metadata": segment.metadata,
                "score": scores[segment_id],
                "source": "hybrid"
            })
    
    return results
```

#### 2.6.5 先向量后图谱（扩展检索）

```python
async def vector_then_kg_search(kb_id: UUID, query: str, top_k: int) -> List[dict]:
    """
    先向量后图谱：先用向量检索找到相关分块，再用知识图谱扩展
    
    流程：
    1. 向量检索找到初始相关分块
    2. 从这些分块中提取实体
    3. 在知识图谱中查找相关实体
    4. 返回扩展后的分块
    
    适用场景：
    - 用户查询较模糊
    - 需要发现隐含关联
    """
    # 1. 向量检索
    initial_results = await vector_search(kb_id, query, top_k // 2)
    
    if not initial_results:
        return []
    
    # 2. 提取实体
    segment_ids = [r["segment_id"] for r in initial_results]
    entities = await get_entities_from_segments(segment_ids)
    
    if not entities:
        return initial_results
    
    # 3. 知识图谱扩展
    from neo4j import GraphDatabase
    
    driver = GraphDatabase.driver(NEO4J_URI, auth=(NEO4J_USER, NEO4J_PASSWORD))
    
    with driver.session() as session:
        # 查询：找到与这些实体相关的其他实体
        cypher = """
        MATCH (e1:Entity)-[r]-(e2:Entity)
        WHERE e1.name IN $entity_names
        MATCH (c:Chunk)-[:MENTIONS]->(e2)
        RETURN DISTINCT c.id AS segment_id,
               collect(DISTINCT e2.name) AS related_entities
        LIMIT $top_k
        """
        
        result = session.run(
            cypher,
            entity_names=[e["name"] for e in entities],
            top_k=top_k // 2
        )
        
        expanded_segment_ids = [record["segment_id"] for record in result]
    
    driver.close()
    
    # 4. 合并结果
    all_segment_ids = list(set(segment_ids + expanded_segment_ids))
    
    stmt = select(Segment).where(Segment.id.in_(all_segment_ids))
    segments = await session.execute(stmt)
    
    return [
        {
            "segment_id": seg.id,
            "text": seg.text,
            "metadata": seg.metadata,
            "score": 1.0 if seg.id in segment_ids else 0.5,
            "source": "vector_then_kg"
        }
        for seg in segments.scalars()
    ]
```

#### 2.6.6 先图谱后向量（精确检索）

```python
async def kg_then_vector_search(kb_id: UUID, query: str, top_k: int) -> List[dict]:
    """
    先图谱后向量：先用知识图谱找到相关实体，再用向量检索排序
    
    流程：
    1. 从查询中提取实体
    2. 在知识图谱中查找相关分块
    3. 对这些分块进行向量相似度排序
    4. 返回排序后的结果
    
    适用场景：
    - 用户查询包含明确实体
    - 需要精确匹配
    """
    # 1. 提取实体
    entities = extract_entities_from_query(query)
    
    if not entities:
        # 降级到纯向量检索
        return await vector_search(kb_id, query, top_k)
    
    # 2. 知识图谱查询
    kg_results = await kg_search(kb_id, query, top_k * 2)
    
    if not kg_results:
        return []
    
    # 3. 向量相似度排序
    query_embedding = call_embedding_api(query)
    segment_ids = [r["segment_id"] for r in kg_results]
    
    stmt = select(
        Segment.id,
        Segment.text,
        Segment.metadata,
        (1 - func.cosine_distance(Embedding.embedding, query_embedding)).label("score")
    ).join(
        Embedding, Segment.id == Embedding.segment_id
    ).where(
        Segment.id.in_(segment_ids)
    ).order_by(
        text("score DESC")
    ).limit(top_k)
    
    results = await session.execute(stmt)
    
    return [
        {
            "segment_id": row.id,
            "text": row.text,
            "metadata": row.metadata,
            "score": row.score,
            "source": "kg_then_vector"
        }
        for row in results
    ]
```

## 三、知识图谱构建策略对比

### 3.1 按分块构建 vs 按文档构建

| 维度 | 按分块构建（推荐） | 按文档构建 |
|------|-------------------|-----------|
| **粒度** | 细粒度，每个分块独立提取 | 粗粒度，整个文档一次提取 |
| **并行性** | 高，可并行处理多个分块 | 低，必须等待整个文档 |
| **准确性** | 高，上下文更聚焦 | 中，长文档容易遗漏 |
| **性能** | 快，单个分块处理时间短 | 慢，长文档处理时间长 |
| **实体消歧** | 需要后处理合并 | 相对简单 |
| **适用场景** | 长文档、复杂文档 | 短文档、简单文档 |

### 3.2 推荐方案：按分块构建 + 后处理合并

```
[1] 并行提取：每个分块独立提取实体和关系
    ↓
[2] 实体消歧：合并同名/相似实体
    ↓
[3] 关系验证：验证实体存在性，去重
    ↓
[4] 图谱构建：存储到 Neo4j
    ↓
[5] 缓存同步：同步到 PostgreSQL
```

**优势**：
- 并行处理，速度快
- 细粒度提取，准确性高
- 支持增量更新（新增分块时只处理新分块）
- 可追溯（知道每个实体/关系来自哪些分块）

## 五、RAGFlow 架构借鉴

### 5.1 核心设计亮点

#### 5.1.1 任务队列：Redis Stream

**RAGFlow 方案**：
- 使用 Redis Stream 而非简单的 List
- 支持消费者组（Consumer Group）
- 未确认消息自动重试（UNACKED_ITERATOR）

**优势**：
- 消息不丢失（持久化）
- 支持多消费者负载均衡
- 自动重试机制

**Python 实现**：
```python
# 生产者
def queue_product(queue_name: str, message: dict):
    """投递任务到 Redis Stream"""
    redis_client.xadd(
        queue_name,
        {"data": json.dumps(message)},
        maxlen=10000  # 限制队列长度
    )

# 消费者
async def collect_tasks():
    """消费任务（先处理未确认，再拉新）"""
    # 1. 处理未确认消息
    pending = redis_client.xpending_range(
        queue_name,
        group_name,
        min="-",
        max="+",
        count=10
    )
    
    for msg in pending:
        task = json.loads(msg["data"])
        await handle_task(task)
        redis_client.xack(queue_name, group_name, msg["id"])
    
    # 2. 拉取新消息
    messages = redis_client.xreadgroup(
        group_name,
        consumer_name,
        {queue_name: ">"},
        count=10,
        block=5000
    )
    
    for stream, msgs in messages:
        for msg_id, msg_data in msgs:
            task = json.loads(msg_data["data"])
            await handle_task(task)
            redis_client.xack(queue_name, group_name, msg_id)
```

**对比 Celery**：
- Celery 已经封装了类似机制
- 但 Redis Stream 更轻量，适合简单场景
- 推荐：继续使用 Celery（成熟稳定）

#### 5.1.2 多层并发控制

**RAGFlow 方案**：

```python
from functools import wraps

def llm_cache(cache_ttl: int = 3600):
    """LLM 结果缓存装饰器"""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            # 生成缓存 key
            cache_key = f"llm:{func.__name__}:{hashlib.md5(str(args).encode()).hexdigest()}"
            
            # 检查缓存
            cached = await redis_client.get(cache_key)
            if cached:
                return json.loads(cached)
            
            # 调用函数
            result = await func(*args, **kwargs)
            
            # 写入缓存
            await redis_client.setex(
                cache_key,
                cache_ttl,
                json.dumps(result)
            )
            
            return result
        return wrapper
    return decorator
```

```python
import hashlib
import json
from functools import wraps

def llm_cache(cache_ttl: int = 3600, key_prefix: str = "llm"):
    """
    LLM 结果缓存装饰器
    
    优势：
    1. 避免重复调用 LLM（节省成本 80%+）
    2. 加速处理（缓存命中时快 10-100x）
    3. 适合批量处理
    """
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            # 生成缓存 key
            cache_data = {
                "func": func.__name__,
                "args": str(args),
                "kwargs": str(sorted(kwargs.items()))
            }
            cache_hash = hashlib.md5(
                json.dumps(cache_data, sort_keys=True).encode()
            ).hexdigest()
            cache_key = f"{key_prefix}:{func.__name__}:{cache_hash}"
            
            # 检查缓存
            cached = await asyncio.to_thread(redis_client.get, cache_key)
            if cached:
                return json.loads(cached)
            
            # 调用函数
            result = await func(*args, **kwargs)
            
            # 写入缓存
            await asyncio.to_thread(
                redis_client.setex,
                cache_key,
                cache_ttl,
                json.dumps(result, ensure_ascii=False)
            )
            
            return result
        return wrapper
    return decorator
```

**问题**：当前方案在异步任务中调用同步库（PyPDF2、LlamaIndex）会阻塞事件循环。

**RAGFlow 方案**：
```python
# 将同步阻塞操作放到线程池执行
binary = await asyncio.to_thread(
    settings.STORAGE_IMPL.get,
    bucket,
    name
)

docs = await asyncio.to_thread(
    chunker.chunk,
    filename,
    binary
)
```

**补充方案**：
```python
@shared_task
async def parse_document_task(document_id: str, file_id: str):
    """
    异步任务中处理同步阻塞操作
    
    优势：
    1. 避免阻塞事件循环
    2. 充分利用线程池
    3. 代码简洁
    """
    # 1. 下载文件（I/O 阻塞）
    file_buffer = await asyncio.to_thread(
        download_file_from_storage,
        file_id
    )
    
    # 2. 解析文件（CPU 密集 + 同步库）
    text = await asyncio.to_thread(
        parse_pdf,  # PyPDF2 是同步的
        file_buffer
    )
    
    # 3. 分块（CPU 密集 + 同步库）
    chunks = await asyncio.to_thread(
        chunk_text,  # LlamaIndex 是同步的
        text
    )
    
    # 4. 保存结果（I/O 阻塞）
    await asyncio.to_thread(
        save_chunks,
        document_id,
        chunks
    )
```

**注意**：
- `asyncio.to_thread` 使用线程池，不是进程
- 仍受 GIL 限制，CPU 密集任务效果有限
- 主要用于避免阻塞事件循环

#### 8.2.5 批量处理（推荐补充）

**问题**：当前方案逐个调用 API，效率低。

**补充方案**：
```python
async def vectorize_batch(texts: list[str], batch_size: int = 100):
    """
    批量向量化
    
    优势：
    1. 减少 API 调用次数
    2. 提升吞吐量
    3. 降低成本
    """
    embeddings = []
    
    for i in range(0, len(texts), batch_size):
        batch = texts[i:i + batch_size]
        
        # 批量调用 API
        async with aiohttp.ClientSession() as session:
            async with session.post(
                "https://api.openai.com/v1/embeddings",
                json={
                    "input": batch,  # 批量输入
                    "model": "text-embedding-3-small"
                },
                headers={"Authorization": f"Bearer {API_KEY}"}
            ) as response:
                data = await response.json()
                batch_embeddings = [item["embedding"] for item in data["data"]]
                embeddings.extend(batch_embeddings)
    
    return embeddings
```

**预期效果**：
- API 调用次数减少 100x
- 吞吐量提升 2-3x
- 成本降低（批量折扣）
