

## 一、当前开发阶段目标

我想开发一个知识库与支持智能体可视化编排的平台，适配小数据量100万数据以内的场景，我的技术选型是只用用PG作为主业务数据库，同时也作为向量存储和全文检索的存储，结合下面我将来要支持的功能，帮我设计表结构。遵循最佳实践。 要求扩展性好。
注意：这阶段，我只做RAG，不做智能体。等完成RAG部分再考虑智能体部分。
但是目前要设计好基础的东西，比如精细的权限管理。

## 三、技术最终选型

#### 3.1 前端

- **框架**: Vite 7.x + React 19.x
- **路由**: TanStack Router 1.x（文件系统路由，类型安全）
- **UI 组件**: Radix UI + shadcn/ui（无样式组件原语）
- **样式**: Tailwind CSS 4.x（原子化 CSS）
- **状态管理**: 
  - TanStack Query 5.x（服务端状态）
  - Zustand 5.x（客户端状态）
- **表单**: React Hook Form 7.x + Zod 4.x（验证）
- **表格**: TanStack Table 8.x
- **图表**: Recharts 3.x
- **可视化流程**: ReactFlow 11.x（用于未来智能体编排）
- **图标**: Lucide React
- **HTTP 客户端**: Axios 1.x
- **语言**: TypeScript 5.9.x

#### 3.2 后端-语言

- **Python 3.12.x**
- **包管理**: uv
- **Web 框架**: FastAPI 0.128+
- **ORM**: SQLAlchemy 2.0+ (asyncio) + fastcrud 0.20+
- **认证**: 纯 Redis Session（不使用 JWT）+ passlib[argon2,bcrypt]
  - Session ID 作为 token，所有 session 数据存储在 Redis
  - 支持即时撤销、Token 轮换、重放检测
- **任务队列**: Redis 6.4+ + Celery 5.6+
- **LLM 管理**: LiteLLM（未来集成）
- **RAG 框架**: LlamaIndex 0.14+ / LangChain 1.2+

#### 3.3 后端-组件

混合检索（全文 + 向量 + RRF 重排）已成为 RAG 项目的强制标准。

| 场景量级          | 推荐组合 (全文 + 向量 + 混合方式)                            | 主要代表框架               | 为什么这个组合最佳？                                         |
| :---------------- | :----------------------------------------------------------- | :------------------------- | :----------------------------------------------------------- |
| **小** (<100万)   | **PostgreSQL 全家桶**<br>· 全文: tsvector / pg_search<br>· 向量: pgvector (HNSW)<br>· 混合: SQL 原生多路召回 | **MaxKB / 典型私有化方案** | **最优起步**：一个 DB 全搞定，事务一致性强，成本最低。支持复杂 JSONB 过滤。<br>*(注：若服务器资源受限如 RAM < 8GB，优先选择 PG 方案，以规避多套引擎的基础开销，8GB 内存足以承载 100 万量级检索)* |
| **中** (1-5000万) | **Qdrant (Mono-DB)**<br>· 全文: Native Sparse Vectors<br>· 向量: HNSW + Payload Filter<br>· 混合: 原生双路召回 + RRF | **高性能通用方案**         | **性能平衡王**：Qdrant 1.16+ 依靠 Rust 高效处理 Payload 过滤。**利用单库实现全文+向量，省去 ES**。 |
| **大** (>5000万)  | **Milvus 2.6/3.0 (Distributed)**<br>· 全文: Built-in BM25 Engine<br>· 向量: HNSW / IVF (GPU 加速)<br>· 混合: 原生多模态索引 | **Dify / RAGFlow**         | **大规模性能王**：分布式/GPU 加速。Milvus 2.6 引入的内置 BM25 性能高，不再需要外挂 ES。 |

**说明**：
  - **小规模 (<100万)**：重点是**“快起步、不出错、好运维”**。PG 让你的系统组件只有 1 个，**维护成本接近 0。**
  - **中大规模 (100万~5000万)**：此时 PG 的 B-Tree/GIN 索引会变得非常臃肿，查询变慢。这时候 Qdrant 的向量压缩技术和 Rust 带来的性能红利开始抵消“多维护一个数据库”的成本，平衡性达到最优。

**最终组件选型**：
  - **主数据库**: PostgreSQL 16+ (含 pgvector 扩展用于向量存储)
  - **全文检索**: PostgreSQL tsvector (中文支持使用 zhparser)
  - **缓存**: Redis 6.4+ (hiredis)
  - **文件存储**: SeaweedFS / 本地文件系统 (S3 兼容接口)
  - **任务队列**: Celery 5.6+ + Redis
  - **未来扩展**: 
    - Qdrant 1.12+（可选，向量数据库）
    - Neo4j（可选，知识图谱）
