# 备忘：自动标签与元数据过滤设计

本文用于记录通用知识库里“自动标签”和“自动元数据过滤”的推荐设计，重点解决这类冲突：

- 用户问“南康地区在哪里修理笔记本”，答案可能在没有地区标签的通用产品说明书里。
- 用户问“南康地区的企业开办联系电话”，赣州地区文档不应该参与回答。

核心结论：

> 显式过滤是硬约束；自动标签主要做排序信号；自动元数据只允许在配置白名单里生效，并使用“匹配或缺失”的通用回退语义。
> 元数据多个 key 默认是 AND；同一个 key 的多个值默认是 OR。

## 1. 问题本质

标签自动过滤的问题不在标签本身，而在“从 query 中命中标签”不等于“用户要求只查这个标签”。

例如 query 中出现“南康”：

- 它可能是用户所在地，答案来自通用文档。
- 它也可能是互斥业务范围，其他地区的联系电话就是错误答案。

如果自动命中的标签直接写入 `tag_ids` / `folder_tag_ids` 做硬过滤，会误杀通用文档。

如果完全不处理自动标签，又可能让赣州这类互斥文档进入最终答案。

所以通用知识库不应该依赖系统理解“南康、赣州是不是地区”这种业务语义，而应该把可过滤语义交给元数据白名单配置。

## 2. 过滤来源分层

所有过滤条件先按来源分层：

| 来源 | 示例 | 默认行为 |
| --- | --- | --- |
| 显式过滤 | UI 勾选标签、API 传入 `folder_ids` / `tag_ids` / `metadata` / `filter_expression` | 硬过滤 |
| 自动标签 | 规则或 LLM 从 query 里命中“南康”标签 | 加权，不默认硬过滤 |
| 自动元数据 | LLM 从 query 里提取 `地区=南康` | 按字段配置执行 |
| 检索词表 / 同义词 | “笔记本”扩展为业务标准词 | 查询增强，不过滤 |

显式过滤必须优先，因为这是调用方明确表达的检索范围。

自动标签和自动元数据不能覆盖显式过滤，只能在显式范围内进一步排序或收窄。

这里建议把术语固定下来：

```text
标签过滤：
  tag_ids / folder_tag_ids

元数据过滤：
  metadata(document_metadata) / search_unit_metadata

统一过滤表达式：
  filter_expression
  可同时表达标签、目录、文档元数据、分块元数据

LLM 统一过滤表达式：
  LLM 输出并经后端白名单校验后的 filter_expression
```

元数据关系默认规则：

```text
多个 key：AND
同一 key 多个值：OR
单一 key 否定：NOT / not_in
跨字段 OR：第一阶段不支持
```

## 3. 自动标签设计

### 3.1 标签默认不做硬过滤

自动命中的文档标签、文件夹标签都默认作为 `boost signal`。

推荐行为：

```text
命中 query 中自动标签的文档：加分
没有标签或没有相关标签的文档：保留
有其他标签但未命中自动标签的文档：轻微降权，可配置
```

这里不要求系统知道标签类型，也不要求判断标签是否互斥。

### 3.2 文档标签评分

文档标签是直接贴在文档上的信号，建议权重略高。

推荐评分：

```text
doc_tag_boost = max_candidate_confidence * source_weight * 0.07
```

其中：

| 来源 | source_weight |
| --- | --- |
| 用户显式选择 | 不走 boost，直接 hard filter |
| LLM 候选 | 1.00 |
| 规则标签名命中 | 0.95 |
| 规则别名命中 | 0.90 |

单个文档的自动文档标签加分建议封顶：

```text
doc_tag_boost <= 0.07
```

避免标签分数压过语义相关性。

### 3.3 文件夹标签分层评分

文件夹标签有层级，需要区分“直接所在文件夹命中”和“祖先文件夹继承命中”。

推荐用距离衰减：

```text
folder_tag_boost = confidence * source_weight * distance_decay * 0.09
```

`distance_decay`：

| 命中关系 | distance_decay |
| --- | --- |
| 文档所在文件夹直接有该标签 | 1.00 |
| 父级文件夹有该标签 | 0.75 |
| 祖父级文件夹有该标签 | 0.55 |
| 更高祖先 | max(0.30, 0.75 ^ distance) |

单个文档的自动文件夹标签加分建议封顶：

```text
folder_tag_boost <= 0.09
```

文档标签和文件夹标签合计建议封顶：

```text
auto_tag_total_boost <= 0.10
```

这样直接或近祖先文件夹标签可以强于文档标签，但总加分仍然受控，不会把语义明显不相关的内容顶到前面。

### 3.4 统一评分体系

评分不能只看标签。当前检索里实际有多类信号：

| 信号 | 示例 | 建议职责 |
| --- | --- | --- |
| 基础召回分 | 向量相似度、全文相关性 | 主排序信号 |
| 检索投影权重 | `default`、`summary`、`question`、`answer`、`keyword`、`doc_summary`、`row` | 表示命中的索引域是否适合当前任务 |
| 多路命中奖励 | 同一块被向量和全文同时命中、同一组有多个片段命中 | 召回稳定性加分 |
| 分块增强信号 | 分块摘要、关键词、假设问题 / 相似问题 | 通过 scope 权重和命中分进入排序，不额外再大幅加分 |
| 文档 / 文件夹标签 | 文档标签、文件夹祖先标签 | 小幅业务相关性加分 |
| 自动元数据 | `地区=南康` | 只做过滤，不做评分 |
| 非过滤投影元数据 | QA 问题文本、表格字段名、行标识、表格上下文 | 轻量词面补充，不能使用过滤字段 |
| rerank 分 | 模型重排返回分 | 二阶段精排信号 |

设计原则：

```text
基础召回分决定“内容像不像答案”。
检索投影权重决定“命中的字段是不是适合这个问题”。
标签 boost 只做轻量业务偏好，不应该压过基础召回分。
自动元数据负责排除互斥范围，不参与加分。
过滤型元数据字段不能进入 metadata_bonus。
rerank 是二阶段精排，不被标签分直接覆盖。
```

#### 3.4.1 先统一到 0-1 分数空间

所有进入融合的分数都应视为 `0-1`。

```text
vector_score_norm = clamp(vector_score, 0, 1)
keyword_score_norm = clamp(keyword_score, 0, 1)
rerank_score_norm = clamp(rerank_score, 0, 1)
```

如果某一路后端天然不是 `0-1`，需要在后端入口先做归一化，而不是在 UI 或最终排序时再猜。

全文分尤其要注意：PG FTS 原始 rank、结构化 QA 归一化分、词表短语补分的分布可能不同。推荐在全文检索后端统一输出 `0-1` 的 `keyword_score_norm`，同时在诊断里保留原始分：

```text
keyword_raw_score
keyword_score_norm
keyword_normalization_method
```

第一阶段如果已经能保证全文后端输出 `0-1`，可以只做 `clamp`，但诊断字段要预留。

#### 3.4.2 基础融合分

基础融合分只使用向量和全文两路：

```text
base_score =
  vector_score_norm * vector_weight
  + keyword_score_norm * (1 - vector_weight)
```

如果只有一路命中，另一侧按 0 处理，但会通过后面的 backend bonus 区分“单路命中”和“双路命中”。

#### 3.4.3 scope 权重

scope 权重用于表达不同索引域的可信度和适配度。

当前可参考的默认方向：

| scope | 建议含义 | 权重倾向 |
| --- | --- | --- |
| `default` | 原文主内容 | 最高，约 1.00 |
| `question` | QA 问题 / 假设问题 | 很高，适合问法匹配 |
| `answer` | QA 答案 | 高，适合答案语义匹配 |
| `summary` | 分块摘要 | 中高，适合泛化召回 |
| `page_body` | 页面正文 | 中高 |
| `row` | 表格行 | 表格知识库最高 |
| `row_group` | 表格行组 | 略低于行 |
| `row_fragment` | 表格行片段 | 再低一些 |
| `keyword` | 分块关键词 | 辅助召回，不能太高 |
| `doc_summary` | 文档摘要 | 文档级粗召回，权重要低 |

推荐用法：

```text
scope_adjusted_score = base_score * scope_weight
```

分块关键词、摘要、检索问题不要再各自额外加一堆独立分。它们应该主要通过：

```text
命中的 scope
该 scope 的 hit.score
scope_weight
```

进入统一评分。这样不会出现“关键词命中 + 摘要命中 + 问题命中”叠加过猛，把正文更相关的块挤掉。

如果按“辅助信号可信度”排序，推荐理解为：

```text
人工治理的范围信号 > 文档级整体语义 > 自动生成的辅助投影
```

但这不是一个绝对大于号，要按命中距离和知识库类型调整：

```text
直接文件夹标签 / 近祖先文件夹标签：
  可以强于文档摘要、关键词这类自动生成投影。

远祖先文件夹标签：
  只保留弱加分，因为范围可能很宽。

文档标签：
  通常比远祖先文件夹标签更精确，但未必比直接文件夹标签更强。

文档级整体评分 / doc_summary：
  适合做粗召回，权重应低于正文分块、QA 问题、表格行等精确命中。

关键词：
  辅助召回，权重应低。

摘要：
  分块摘要通常高于关键词；文档摘要低于分块摘要。

检索问题 / question：
  在普通文档的假设问题场景里是辅助投影；
  在 QA 知识库里是核心字段，权重应接近正文或高于答案片段。
```

所以不建议写成统一的：

```text
文件夹标签 > 文档级整体评分 > 关键词/摘要/检索问题
```

更稳的层级是：

```text
1. 精确内容命中：
   default / row / QA question / answer

2. 人工治理范围信号：
   直接文件夹标签、近祖先文件夹标签、文档标签

3. 文档级粗语义：
   doc_summary、文档标题、文档级摘要

4. 自动生成辅助投影：
   chunk summary、keywords、普通文档的 hypothetical questions

5. 宽泛或远距离范围信号：
   远祖先文件夹标签、低置信自动标签
```

其中第 2 层不能直接压过第 1 层，只能作为 `auto_tag_boost` 小幅加分；第 3、4 层主要通过 scope 命中分和 scope 权重进入基础融合，不再额外加大 bonus。

#### 3.4.4 召回质量 bonus

召回质量 bonus 只奖励“证据更稳定”的情况。

推荐：

```text
backend_bonus:
  向量和全文都命中同一聚合结果：+0.04 ~ +0.05

scope_coverage_bonus:
  多个不同 scope 命中同一聚合结果：每多一个 +0.02 ~ +0.03，封顶 0.08

repeated_hit_bonus:
  QA answer 碎片或表格 row_fragment 多片段命中：封顶 0.05 ~ 0.08
```

这些 bonus 合计建议封顶：

```text
retrieval_quality_bonus <= 0.12
```

#### 3.4.5 分块增强信号与非过滤投影元数据

分块增强包括：

```text
summary
keywords
question
similar_questions
doc_summary
```

推荐处理方式：

```text
summary / question / answer / keyword / doc_summary 都作为 search_scope 参与召回。
命中后只通过 scope_weight 和基础 hit.score 影响排序。
不要因为 metadata 里有 keywords 或 summary，再额外做大分值加权。
```

可以保留非常轻量的 query intent bonus：

```text
QA 问法命中 question scope：+0.015 ~ +0.035
答案/步骤类问法命中 answer scope：+0.015 ~ +0.030
表格查询命中 row / row_group：+0.020 ~ +0.035
```

`query_intent_bonus` 合计建议封顶：

```text
query_intent_bonus <= 0.04
```

关键词命中不建议再单独加 query intent bonus，因为关键词 scope 本身已经是辅助召回域，默认权重应低于正文和问句。

当前代码里有 `metadata_bonus` 这类轻量词面加分时，必须限制为非过滤投影字段：

```text
允许：
  QA question_text
  表格 sheet_name
  row_identity_text
  field_names
  dimension_field_names
  metric_field_names
  table_context_text

不允许：
  document_metadata 自动过滤字段
  search_unit_metadata.filter_fields
  qa_fields.category
  qa_fields.tags
```

原因：

```text
过滤型元数据已经用于候选裁剪。
如果再拿来加分，会让“范围约束”和“相关性排序”混在一起。
```

#### 3.4.6 标签 boost

自动标签 boost 是业务相关性补充，不是主排序信号。

推荐：

```text
doc_tag_boost <= 0.07
folder_tag_boost <= 0.09
auto_tag_total_boost <= 0.10
```

这里把总上限从 `0.12` 收敛到 `0.10` 更稳，因为系统里已经存在 scope、backend、coverage、query intent 等信号。

推荐公式：

```text
auto_tag_boost = min(
  0.10,
  max_doc_tag_boost + max_folder_tag_boost
)
```

还需要加一个语义地板，避免标签把明显不相关的内容顶上来：

```text
semantic_score = base_score * scope_weight

如果 semantic_score < 0.20：
  effective_auto_tag_boost = 0

如果 0.20 <= semantic_score < 0.35：
  effective_auto_tag_boost = auto_tag_boost * 0.5

如果 semantic_score >= 0.35：
  effective_auto_tag_boost = auto_tag_boost
```

也就是说，标签只在“内容基本相关”的候选之间调序，不负责救活弱语义命中。

如果同一文档命中多个自动标签，不建议线性累加所有标签。使用 `max + 少量覆盖奖励` 更安全：

```text
doc_tag_boost =
  max(single_doc_tag_boost)
  + min(0.02, max(0, matched_doc_tag_count - 1) * 0.01)
```

文件夹标签同理：

```text
folder_tag_boost =
  max(single_folder_tag_boost)
  + min(0.015, max(0, matched_folder_tag_count - 1) * 0.0075)
```

#### 3.4.7 最终 rerank 前分数

推荐汇总：

```text
pre_rerank_score = clamp(
  base_score * scope_weight
  + retrieval_quality_bonus
  + query_intent_bonus
  + effective_auto_tag_boost,
  0,
  1
)
```

注意：

```text
自动元数据不在这里出现。
分块摘要、关键词、问题不作为额外 bonus 出现，它们已经体现在 scope 和 hit.score 里。
```

#### 3.4.8 rerank 后排序

第一阶段建议保持简单：

```text
启用 rerank：
  最终排序以 rerank_score 为主。
  pre_rerank_score 只用于进入 rerank 前的候选选择和诊断展示。

未启用 rerank：
  最终排序使用 pre_rerank_score。
```

如果后续发现 rerank 模型对业务标签完全不敏感，再考虑混合：

```text
final_sort_score = rerank_score_norm * 0.90 + pre_rerank_score * 0.10
```

但不建议第一阶段就混合，因为会让“模型分”和“业务规则分”的解释变复杂。

#### 3.4.9 推荐上限

建议所有非基础召回加分有明确上限：

| 信号 | 建议上限 |
| --- | --- |
| `retrieval_quality_bonus` | 0.12 |
| `query_intent_bonus` | 0.04 |
| `auto_tag_boost` | 0.10 |
| 所有 bonus 合计 | 0.20 |

也就是说：

```text
bonus_total = min(
  0.20,
  retrieval_quality_bonus + query_intent_bonus + effective_auto_tag_boost
)
```

最终：

```text
pre_rerank_score = clamp(base_score * scope_weight + bonus_total, 0, 1)
```

这样做的好处：

```text
语义召回分仍然是主体。
标签、关键词、摘要、问题不会各自叠加到失控。
诊断界面能解释每一类信号的贡献。
未来新增信号时必须进入某个 bucket，而不是随手加分。
```

### 3.5 其他标签的降权

如果 query 自动命中了标签 A，而文档命中其他标签 B 但没有命中 A，可以做轻微降权。

推荐默认值：

```text
other_tag_penalty = 0.03
```

建议只在“该文档确实有同类候选线索”时使用。由于当前通用知识库没有标签类型，第一阶段可以先不启用降权，只做命中加权。

如果后续要启用，应做成配置：

```text
auto_tag_policy = boost_only | boost_and_penalty
```

推荐默认：

```text
boost_only
```

## 4. 自动元数据设计

### 4.1 元数据负责表达可过滤语义

如果业务上需要表达“地区互斥”，不要依赖标签名推理，应该由文档元数据表达：

```text
文档1：地区=南康
文档2：地区=赣州
文档3：没有地区字段
```

用户问：

```text
南康地区的企业开办联系电话
```

LLM 提取：

```json
{
  "document_metadata": {
    "地区": "南康"
  }
}
```

过滤语义：

```text
文档包含 地区 字段：
  只有 地区=南康 才匹配

文档不包含 地区 字段：
  默认保留，视为通用文档
```

这叫“匹配或缺失”：

```text
match_or_missing
```

它和硬过滤不同：

```text
hard filter:
  只保留 地区=南康

match_or_missing:
  保留 地区=南康 + 没有地区字段的通用文档
```

`match_or_missing` 落地时不建议把这两类文档混成一个无差别候选池，而应拆成：

```text
主候选池：
  字段存在且值匹配的文档。

通用回退池：
  字段缺失的文档。

排除池：
  字段存在但值不匹配的文档。
```

推荐截断策略：

```text
先从主候选池取结果。
主候选池不足时，再从通用回退池补足。
通用回退池不因为缺失字段加分，只保留参与语义排序的资格。
```

这样可以保留无地区字段的通用文档，又不会让大量未治理文档淹没真正匹配 `地区=南康` 的文档。

### 4.2 自动元数据必须走白名单配置

LLM 不能随便创造字段。

前端需要配置允许自动提取的字段。字段来源可以有两种：

```text
1. 从文件列表中已有的文档元数据 key 里选择。
2. 手动新建一个 key，用于文档还没补齐元数据、但管理员已经确定后续会治理的字段。
```

推荐字段配置：

```json
{
  "key": "region",
  "name": "地区",
  "target": "document_metadata",
  "metadata_path": ["region"],
  "aliases": ["地区", "区域", "所在地"],
  "options": [
    {
      "value": "南康",
      "label": "南康",
      "aliases": ["南康区"]
    },
    {
      "value": "赣州",
      "label": "赣州",
      "aliases": ["赣州市"]
    }
  ],
  "auto_extract": "llm",
  "match_mode": "match_or_missing",
  "confidence_threshold": 0.78
}
```

关键字段：

| 字段 | 说明 |
| --- | --- |
| `key` | 配置项稳定 ID |
| `name` | 给用户和 LLM 看的显示名 |
| `target` | `document_metadata` 或 `search_unit_metadata` |
| `metadata_path` | 实际落库路径 |
| `aliases` | 字段别名 |
| `options` | 允许取值，建议枚举 |
| `auto_extract` | 推荐 `llm` |
| `match_mode` | `match_or_missing` / `match_only` |
| `confidence_threshold` | LLM 候选最低置信度 |

### 4.3 前端字段配置界面

知识库配置页建议把“元数据识别字段”设计成一个独立配置区，标题可以是：

```text
自动元数据过滤字段
```

界面建议：

```text
字段来源：
  - 从已有文档元数据选择
  - 新建字段

字段 Key：
  - 选择已有 key 时自动填充
  - 新建字段时手动输入，保存后进入白名单

显示名称：
  - 给 LLM 和调试界面使用，例如“地区”

字段别名：
  - 例如“地区、区域、所在地”

目标层级：
  - 文档级 metadata
  - 分块 / 搜索单元级 metadata

匹配模式：
  - 匹配或缺失
  - 必须匹配

允许取值：
  - 从已有文档元数据值中选择
  - 手动新增
  - 支持值别名

自动提取：
  - LLM
  - 关闭

置信度阈值：
  - 默认 0.78
```

从已有文档元数据选择时，前端可以展示：

```text
字段 key
出现文档数
不同取值数
Top 取值
是否已加入自动过滤白名单
```

手动新建字段时要给出提示：

```text
当前可能还没有文档使用这个字段。保存后 LLM 可以识别该字段，但只有文档元数据补齐后才会产生有效过滤。
```

取值配置建议采用“可选值表”：

| 列 | 说明 |
| --- | --- |
| 显示名 | 给用户和 LLM 看 |
| 实际值 | 与文档元数据存储值匹配 |
| 别名 | LLM 识别时可用 |
| 出现次数 | 从已有文档统计得到，手动值显示 0 |

对于未枚举但字段已存在的情况，建议先不启用自动元数据过滤。自动元数据过滤最好只对有限枚举值生效。

### 4.4 是否必须 LLM

推荐：

```text
自动元数据过滤默认必须 LLM。
```

原因：

- 规则很难判断“南康”到底是地区、品牌、目录、还是普通词。
- 规则难以穷举字段别名、值别名和自然语言表达。
- 通用知识库里字段语义变化大，硬编码规则容易误杀。

规则可以保留为非常保守的辅助能力：

```text
只有字段和值都在 query 中明确出现，并且字段是枚举字段时，规则可产出候选。
默认不直接启用规则型自动元数据过滤；如果后续启用，也必须经过字段白名单和高置信阈值校验。
```

例如：

```text
“地区是南康的联系电话”
```

这种才适合规则识别。

### 4.5 元数据过滤模式

每个字段只需要支持两种过滤模式：

| 模式 | 行为 | 适用场景 |
| --- | --- | --- |
| `match_or_missing` | 字段匹配或字段缺失都保留 | 地区、部门、产品线这类可能存在通用文档的字段 |
| `match_only` | 必须字段存在且匹配 | 严格结构化库、报表、表格 |

推荐默认：

```text
document_metadata: match_or_missing
search_unit_metadata: match_only 或 match_or_missing，按业务配置
```

如果字段用于 QA / 表格行级过滤，`match_only` 通常更合理，因为每行就是结构化事实。

如果字段用于普通文档级过滤，`match_or_missing` 更稳，因为缺失字段常常表示通用资料。

### 4.6 自动元数据不参与评分

自动元数据的职责是结构化过滤，不再同时承担评分加权。

如果某个字段治理还不成熟，不应该把它改成加权字段，而应该先关闭该字段的自动元数据过滤：

```text
auto_extract = off
```

这样职责清晰：

```text
自动标签：负责相关性加权。
自动元数据：负责结构化范围过滤。
向量 / 全文 / rerank：负责最终相关性排序。
```

### 4.7 标签和元数据的且 / 或关系

过滤和加权都必须明确关系语义，不能只把所有条件扔进一个列表。

#### 4.7.1 显式过滤

显式过滤来自 UI 或 API，默认应按“强约束”处理。

推荐语义：

```text
不同过滤类型之间：AND
  folder_ids AND tag_ids AND document_metadata AND search_unit_metadata

同一文档标签列表 tag_ids：AND
  文档必须同时拥有这些标签

同一文件夹标签列表 folder_tag_ids：AND
  文件夹标签先解析成满足全部标签的文件夹集合，再展开到文档范围

同一个元数据字段多个值：
  默认 OR，例如 地区 in [南康, 赣州]

不同元数据字段之间：
  AND，例如 地区=南康 AND 年份=2024
```

也就是说：

```text
metadata = {
  "地区": ["南康", "赣州"],
  "年份": ["2024"],
  "状态": {"op": "not_in", "values": ["已废止"]}
}

等价于：
  (地区=南康 OR 地区=赣州)
  AND 年份=2024
  AND 状态 NOT IN [已废止]
```

默认不支持：

```text
地区=南康 OR 年份=2024
```

因为这是跨 key OR，会明显放宽候选范围，第一阶段应拒绝或降级为不应用。

如果前端未来要支持“任一标签”，应该显式增加配置：

```text
tag_match_mode = all | any
metadata_field_match_mode = all | any
```

不建议默认把多个标签改成 OR，因为这会放宽用户显式过滤范围。

#### 4.7.2 自动元数据

自动元数据默认只允许 LLM 从 query 中提取白名单字段。

推荐语义：

```text
不同字段之间：AND
  地区=南康 AND 部门=市场监管

同一字段多个值：OR
  地区 in [南康, 赣州]
```

再次明确：自动元数据多个 key 默认也是 AND，不因为它来自 LLM 就自动放宽成 OR。

自然语言里的“且、并、同时”通常映射为 AND：

```text
南康并且 2024 年的企业开办电话
=> 地区=南康 AND 年份=2024
```

自然语言里的“或、或者、任一”通常映射为同字段 OR：

```text
南康或赣州的企业开办电话
=> 地区 in [南康, 赣州]
```

如果 LLM 输出跨字段 OR，例如：

```text
地区=南康 OR 部门=行政审批
```

第一阶段建议拒绝或降级为不应用，因为跨字段 OR 会显著增加候选范围，也更难解释。

第一阶段可以支持单字段否定：

```text
不是赣州的企业开办电话
=> 地区 not in [赣州]
```

`not_in + match_or_missing` 的推荐语义：

```text
字段存在且值在排除列表中：
  排除

字段存在且值不在排除列表中：
  保留

字段缺失：
  保留，仍视为通用文档
```

第一阶段推荐支持的表达能力：

```text
根层级：AND
同一 key 多值：OR
单一 key 否定：NOT / not_in
```

等价于：

```text
地区 in [南康, 赣州]
AND 年份 in [2024]
AND 状态 not in [已废止]
```

暂不支持：

```text
跨字段 OR
括号表达式
多层嵌套 AND / OR / NOT
```

但数据结构需要为第二阶段预留表达式树，避免后续推倒重来。

推荐内部结构：

```json
{
  "operator": "and",
  "conditions": [
    {
      "field": "region",
      "op": "in",
      "values": ["南康", "赣州"],
      "match_mode": "match_or_missing"
    },
    {
      "field": "year",
      "op": "in",
      "values": ["2024"],
      "match_mode": "match_only"
    },
    {
      "field": "status",
      "op": "not_in",
      "values": ["已废止"],
      "match_mode": "match_only"
    }
  ]
}
```

第二阶段再开放：

```text
(地区=南康 OR 部门=行政审批) AND 年份=2024
```

第二阶段表达式树可以扩展为：

```json
{
  "operator": "and",
  "children": [
    {
      "operator": "or",
      "children": [
        {
          "field": "region",
          "op": "in",
          "values": ["南康"]
        },
        {
          "field": "department",
          "op": "in",
          "values": ["行政审批"]
        }
      ]
    },
    {
      "field": "year",
      "op": "in",
      "values": ["2024"]
    }
  ]
}
```

第二阶段开放前需要先解决：

```text
LLM 输出校验
前端可视化编辑
诊断解释
SQL / JSONB 条件生成
跨字段 OR 的候选范围膨胀
权限与审计
```

#### 4.7.3 自动标签

自动标签不是硬过滤，默认只做 boost。

推荐语义：

```text
命中任一自动标签：
  可以获得对应标签 boost。

命中多个自动标签：
  取 max + 少量覆盖奖励，不线性累加。

LLM 判断 query 明确表达“只查某标签”：
  第一阶段仍不建议自动升级为硬过滤；
  可以在诊断中提示“检测到强范围表达，但当前策略按 boost 处理”。
```

这样可以避免“南康 + 笔记本”两个词分别命中标签后，被错误解释成必须同时满足多个标签。

## 5. 诊断信息设计

检索测试页和分块诊断需要把标签、元数据的决策过程展示清楚。目标不是只展示最终过滤结果，而是能回答：

```text
LLM / 规则识别到了什么？
哪些被应用了？
哪些被拒绝了？
拒绝原因是什么？
过滤前后剩下多少候选？
某条结果为什么加分？
某条结果为什么被元数据排除？
```

### 5.1 查询级诊断

查询分析区域建议展示：

```text
自动标签候选：
  - 类型：文档标签 / 文件夹标签
  - 标签名、ID、别名命中词
  - 来源：规则 / LLM
  - 置信度
  - 使用方式：boost / ignored / hard_filter，默认 boost
  - 是否进入评分
  - 未应用原因

自动元数据候选：
  - 字段显示名、key、metadata_path
  - 提取值、规范化后的实际值
  - 命中表达
  - 来源：LLM
  - 置信度
  - 匹配模式：match_or_missing / match_only
  - 是否应用
  - 未应用原因
```

未应用原因建议枚举：

```text
low_confidence
field_not_allowed
value_not_allowed
conflict_with_explicit_filter
auto_extract_disabled
llm_parse_failed
no_candidate
```

### 5.2 过滤级诊断

过滤诊断摘要建议展示：

```text
显式过滤：
  - 文件夹、标签、文档元数据、搜索单元元数据

自动元数据过滤：
  - 字段=值
  - 匹配模式
  - 匹配文档数
  - 缺失字段保留文档数
  - 值不匹配排除文档数

自动标签评分：
  - 命中文档标签数
  - 命中文件夹标签数
  - 平均加分
  - 最大加分
```

对于 `match_or_missing`，尤其要展示三类数字：

```text
matched_value_count
missing_field_kept_count
mismatched_value_excluded_count
```

这能解释为什么“无地区字段”的通用文档还在，而“赣州”文档被排除。

### 5.3 结果级诊断

每条结果建议展示一个可展开的“过滤与评分”小节：

```text
标签评分：
  - 文档标签命中：南康，+0.076
  - 文件夹标签命中：政务/南康，距离 1，+0.052
  - 自动标签总加分：+0.10，上限 0.10

元数据过滤：
  - 地区=南康，文档值=南康，匹配
  - 或：地区=南康，文档缺失该字段，按通用文档保留

最终分数：
  - 向量分
  - 全文分
  - 融合分
  - 自动标签加分
  - rerank 分
```

被排除的结果不一定要默认展示在主结果列表里，但调试模式可以提供“被过滤样本”：

```text
被自动元数据排除的样本：
  - 文档：赣州企业开办指南
  - 原因：地区=赣州，不匹配 query 提取的 地区=南康
```

为了界面优雅，建议用三层展示：

```text
概览：一句话总结应用了哪些自动条件
详情：候选表格和计数
结果展开：单条结果的命中、保留、加分原因
```

## 6. API 配置优先级

### 6.1 默认规则

API 传入的 `raw_filters` 默认视为显式过滤。

例如：

```json
{
  "filters": {
    "tag_ids": ["..."],
    "folder_tag_ids": ["..."],
    "metadata": {
      "地区": "南康"
    }
  }
}
```

默认语义：

```text
tag_ids / folder_tag_ids: hard filter
metadata: hard filter
```

原因是 API 调用方已经明确传入过滤条件，后端不应猜测它只是想加权。

### 6.2 合并顺序

推荐流程：

```text
1. 解析 API / UI 显式过滤
2. 应用显式硬过滤范围
3. 执行同义词、检索词表、停用词等查询增强
4. 根据 auto_filter_mode 执行自动标签 / 自动元数据候选提取
5. 自动元数据只在白名单字段中生效
6. 自动元数据在显式范围内执行 match_or_missing / match_only
7. 自动标签只在显式范围内做 boost
8. 向量 + 全文召回
9. 融合分 + 自动标签加权
10. rerank
11. 最终答案生成时带上结构化约束说明
```

关键原则：

```text
自动条件不能扩大显式过滤范围。
自动条件不能覆盖显式过滤字段。
显式 metadata 与自动 metadata 冲突时，以显式 metadata 为准。
```

### 6.3 API 与知识库配置的关系

推荐不要让 API 请求默认覆盖知识库里的自动元数据字段配置。

原因：

```text
知识库配置代表管理员治理过的字段白名单。
API 请求代表一次检索调用的运行时意图。
如果 API 可以随便覆盖字段白名单，就会重新引入 LLM 任意字段过滤的风险。
```

推荐优先级：

```text
1. 显式 filters：最高优先级，永远作为 hard filter。
2. API auto_filter_config：只覆盖运行时开关和阈值，不默认覆盖字段白名单。
3. 知识库 query_analysis.metadata_fields：自动元数据字段白名单的默认来源。
4. 系统默认值：最后兜底。
```

API 可以做三类事情：

```text
关闭自动元数据：
  allow_auto_metadata_filter = false

限制本次请求可用字段：
  enabled_metadata_field_keys = ["region"]

追加临时字段：
  extra_metadata_fields = [...]
```

其中 `extra_metadata_fields` 默认不建议开放给普通业务调用。若开放，必须满足：

```text
字段必须有 key / name / metadata_path。
字段必须有枚举 options。
字段 match_mode 只能是 match_or_missing 或 match_only。
字段数量、取值数量有上限。
请求日志必须记录。
```

合并规则：

```text
effective_metadata_fields =
  kb_metadata_fields
  |> 按 API enabled_metadata_field_keys 限制
  |> 如果 allow_request_extra_metadata_fields=true，再追加 extra_metadata_fields
```

如果 API 传入字段和知识库字段同 key：

```text
默认不覆盖知识库字段。
只有 request_override_metadata_fields=true 且调用方有管理权限时，才允许覆盖。
```

推荐第一阶段不实现覆盖，只实现：

```text
API 可关闭自动元数据。
API 可限制本次启用哪些已配置字段。
API 不可新增或覆盖自动元数据字段白名单。
```

这样最稳。

### 6.4 API 是否支持传入过滤方式

建议支持，但需要和普通 `filters` 分开。

推荐 API 结构：

```json
{
  "filters": {
    "tag_ids": ["..."],
    "metadata": {
      "地区": "南康"
    }
  },
  "auto_filter_config": {
    "auto_filter_mode": "llm_candidate",
    "auto_tag_policy": "boost_only",
    "allow_auto_metadata_filter": true,
    "enabled_metadata_field_keys": ["region"]
  },
  "soft_filters": {
    "tag_ids": ["..."]
  }
}
```

语义：

| 字段 | 语义 |
| --- | --- |
| `filters` | 显式硬过滤 |
| `soft_filters` | 调用方显式要求的加权信号，第一阶段建议只支持标签 |
| `auto_filter_config` | 控制自动抽取和自动条件如何使用 |

不建议支持 `soft_filters.metadata`。

原因：

```text
元数据已经被定义为结构化过滤语义。
如果 API 同时支持“软元数据”，会重新引入元数据到底是过滤还是加权的歧义。
需要软相关性时，优先使用标签或普通 query 词。
```

如果短期不想扩 API，可以先约定：

```text
现有 filters 一律 hard filter。
自动提取只来自 query_analysis 配置，不和 API filters 混在一起。
```

这最不容易混乱。

## 7. 前端配置建议

前端知识库配置页应把“元数据识别字段”升级为“自动结构化过滤字段”。文件列表中的文档元数据编辑能力要和这里打通。

建议配置项：

```text
字段 Key
显示名称
字段别名
目标层级：文档级 / 分块或搜索单元级
元数据路径
允许取值
取值别名
自动提取方式：LLM / 关闭
匹配模式：匹配或缺失 / 必须匹配
置信度阈值
```

推荐交互：

```text
1. 在文件列表里，用户可以继续给单个或批量文档维护元数据。
2. 在知识库配置页，用户点击“新增自动元数据过滤字段”。
3. 弹窗先展示“从已有文档元数据选择”。
4. 用户也可以切到“新建字段”，提前配置后续要治理的字段。
5. 选择字段后，系统自动扫描已有取值，用户勾选允许 LLM 提取的值。
6. 用户可以给字段和值补别名。
7. 保存后，该字段进入自动元数据白名单。
```

文件列表中也可以提供快捷入口：

```text
把此元数据字段加入自动过滤
```

但点击后应该跳到配置弹窗，而不是直接启用。因为需要确认：

```text
字段别名
允许取值
匹配模式
置信度阈值
```

取值必须尽量枚举化。

不建议让 LLM 从无限自由文本里提取任意值，因为这会导致：

- 值拼写不一致
- 无法判断是否真的存在该值
- 过滤条件过窄
- 调试困难

对于“地区”这类字段，建议前端允许管理员维护：

```text
南康 | 南康 | 南康区/南康地区
赣州 | 赣州 | 赣州市/赣州地区
```

{

## 9. 对当前实现的落地建议

当前实现里 `RetrievalFilterSet` 只有硬过滤字段，自动标签一旦写入 `tag_ids` / `folder_tag_ids` 就会变成硬过滤。

建议后续拆出新的分析结果结构：

```python
@dataclass
class AutoFilterSignal:
    signal_type: str  # doc_tag / folder_tag / document_metadata / search_unit_metadata
    target_id: str
    value: str
    confidence: float
    source: str
    usage: str  # tag_boost / match_or_missing / match_only
```

然后：

```text
RetrievalFilterSet:
  继续只承载显式硬过滤和最终硬过滤。

AnalyzedQuery.auto_filter_signals:
  承载自动标签和自动元数据候选。

ResolvedCandidateFilter:
  支持显式硬过滤 + 自动元数据 match_or_missing 后的候选集解析。

score_trace:
  记录 tag_boost。
```

第一阶段最小改造：

```text
1. LLM 自动标签不再升级到 tag_ids / folder_tag_ids。
2. 增加 auto_filter_signals，并把自动标签写入 boost 信号。
3. metadata_fields 增加 match_mode / confidence_threshold。
4. LLM 只允许输出 metadata_fields 白名单里的字段和值。
5. 对 document_metadata 实现 match_or_missing 解析。
6. 检索测试页展示自动标签的分数贡献、自动元数据的过滤方式和命中范围。
```

第二阶段再考虑：

```text
1. API 支持 enabled_metadata_field_keys 限制本次可用字段。
2. 检索测试诊断展示 match_or_missing 的三类计数。
3. 结果级 score_trace 增加 auto_tag_boost 明细。
4. 管理员权限下支持 request_override_metadata_fields。
```

## 10. 推荐问答行为

用户问：

```text
南康地区在哪里修理笔记本
```

如果 LLM 提取 `地区=南康`：

```text
地区=南康 文档：保留
无地区字段文档：保留
地区=赣州 文档：排除
```

如果没有配置地区元数据字段，只命中“南康”标签：

```text
南康标签文档：加权
无标签或通用文档：保留
赣州标签文档：默认不硬排除，最多轻微降权
```

用户问：

```text
南康地区的企业开办联系电话
```

有地区元数据配置时：

```text
地区=南康 文档：保留
无地区字段文档：保留
地区=赣州 文档：排除
```

这能避免把赣州联系电话答成南康联系电话，同时保留通用说明书、通用流程、通用维修指南这类资料。

## 11. 一句话原则

```text
API / UI 显式选择才是硬过滤；
自动标签是排序信号；
自动元数据必须来自白名单配置；
互斥语义用 match_or_missing 元数据过滤表达；
缺失该元数据的文档默认当作通用文档保留。
```
