# 知识库工作台与多类型视图设计方案

## 1. 文档目的

本文档用于统一知识库详情页的前端工作台设计，解决以下问题：

- 表格知识库已经出现“文件治理”和“数据使用”两种主工作模式，不能再只按文件页组织
- 后续 `qa / web / media / connector` 等类型知识库也会逐步出现各自的主工作视图
- 如果继续为每种知识库单独做一整页，会导致交互割裂、代码分叉和维护成本持续上升
- 若盲目做一个“万能数据视图”，又会把所有类型的差异硬塞进一个大组件里，最终不可维护

本文档目标是形成一套可扩展、可维护、性能可控的知识库工作台设计。


## 2. 设计结论

知识库详情页应拆成两层导航：

### 2.1 一级导航：通用能力

一级导航继续承载所有知识库共通的能力：

- 文件管理
- 全局配置
- 标签管理
- 术语管理
- 同义词管理
- 检索测试

这层是“知识库通用能力层”，不随知识库类型变化而频繁调整。

### 2.2 二级视图：类型专属工作区

在一级导航进入主工作区后，再根据 `kb.type` 渲染该类型的二级视图。

例如：

- `table`
  - 文件列表
  - 数据视图
  - 结构定义
- `qa`
  - 问答列表
  - 导入视图
- `web`
  - 站点列表
  - 页面视图
  - 同步任务
- `media`
  - 文件列表
  - 转写片段
- `connector`
  - 连接源
  - 同步对象
  - 内容项

核心原则：

- 一级导航解决“做什么类型的事情”
- 二级视图解决“在当前类型知识库里，从哪个主视角工作”


## 3. 为什么不推荐两种极端方案

### 3.1 不推荐：每种类型都做成完全不同的详情页

问题：

- 用户心智不统一
- 通用能力入口位置不稳定
- 前端页面骨架会高度重复
- 新增知识库类型时，容易继续复制页面结构

### 3.2 不推荐：所有类型强行共用一个万能数据视图

问题：

- 类型差异会全部堆进一个超大组件
- `if kb.type === ...` 会越来越多
- 数据接口、列定义、筛选方式、交互形态根本不同
- 最终虽然“表面统一”，但维护性最差


## 4. 推荐方案：统一工作台骨架 + 类型专属视图注册

### 4.1 架构原则

前端应抽象为：

1. 通用工作台骨架
2. 类型视图注册表
3. 类型专属视图组件

也就是：

- **统一骨架**
  - 页面头部
  - 一级导航
  - 工作区容器
  - 空状态
  - 引导态
  - 数据缓存边界
  - 通用一级配置页签

- **类型注册**
  - 定义某种知识库有哪些二级视图
  - 定义默认视图
  - 定义每个视图的组件与加载策略
  - 定义该类型有哪些“一等配置区”

- **类型组件**
  - `TableFileListView`
  - `TableRecordView`
  - `QAItemListView`
  - `WebPageListView`
  - `MediaTranscriptView`
  - `ConnectorItemView`

### 4.2 推荐前端抽象

推荐形成类似结构：

```ts
type KnowledgeBaseWorkbenchView =
  | 'files'
  | 'records'
  | 'qa_items'
  | 'web_pages'
  | 'transcripts'
  | 'connector_items'

interface KnowledgeBaseTypeViewConfig {
  defaultView: string
  views: Array<{
    id: string
    label: string
    lazy?: boolean
  }>
}
```

示例注册：

```ts
{
  table: {
    defaultView: 'files',
    views: [
      { id: 'files', label: '文件列表' },
      { id: 'records', label: '数据视图' },
    ],
  },
  qa: {
    defaultView: 'qa_items',
    views: [
      { id: 'qa_items', label: '问答列表' },
      { id: 'files', label: '导入视图' },
    ],
  },
  web: {
    defaultView: 'web_pages',
    views: [
      { id: 'web_pages', label: '页面视图' },
      { id: 'files', label: '抓取文件' },
    ],
  }
}
```


## 5. 表格知识库的当前落地方式

当前阶段先在 `table` 类型知识库落地这套工作台方案：

- 一级导航仍然保留“文件管理”
- 进入“文件管理”工作区后，在顶部增加二级切换：
  - 文件列表
  - 数据视图
  - 结构定义

原因：

- 对用户来说，这三者本质上都属于“表格知识库主工作区”
- 文件列表负责导入治理
- 数据视图负责记录级使用
- 结构定义负责表格 schema、字段用途、导入校验等主流程配置
- 它们之间切换应该是平级工作模式切换，而不是跳到另一个完全独立页面

这样会比“左侧导航里再加一个独立数据页”更自然。

补充边界：

- 这里的“结构定义”指类型专属的主流程配置，不等同于整个“全局配置”
- “全局配置”仍然保留在一级导航，承载通用能力，例如模型、解析分块、智能索引
- 一旦某类型已经在工作区提供了成熟的专属配置视图，就不应在“全局配置”里再重复放一套同义入口
- 当某类知识库存在“高频、主流程、强类型相关”的配置面时，可以同时在工作区提供直达入口
- 后续 `qa / web / media / connector` 也可以参照这个规则，为自己的核心配置提供并排入口


## 6. 可维护性设计

### 6.1 不让类型分支扩散到全局页面

禁止在详情页里继续堆叠大量：

```ts
if (kb.type === 'table') ...
if (kb.type === 'qa') ...
if (kb.type === 'web') ...
```

更好的方式是：

- 详情页只负责拿到 `kb.type`
- 按注册表选择该类型的视图配置与工作区组件
- 全局配置页按注册表装配类型专属一级配置区

### 6.1.1 类型专属配置的扩展方式

对于 `table / qa / web / media / connector` 这类未来会逐步长出专属配置面的类型，不应继续采用“把专属配置埋到通用配置子栏里”的方式。

推荐做法：

- 全局配置页保留通用一级页签，例如：
  - 基础设置
  - 解析分块
  - 智能索引
- 再通过“类型配置注册表”追加该类型的一等配置区，例如：
  - `table -> 表格结构`
  - `qa -> 问答结构`
  - `web -> 同步规则`

这样可以保证：

- 主骨架稳定
- 类型差异集中在注册层
- 后续新增类型不需要频繁重构全局配置主页面

### 6.2 通用逻辑下沉到骨架层

应统一沉淀的能力：

- 空状态
- 首次引导
- 查询参数同步
- query key 约定
- 加载状态
- 刷新逻辑
- 权限校验
- 页面布局

### 6.3 类型差异留在专属视图层

不要试图把这些差异硬统一：

- 表格的记录列展示
- QA 的问答卡片编辑
- 网页的页面抓取状态
- 音视频的转写片段时间轴
- 同步应用的对象树结构

这些都应保留在各自类型组件中。


## 7. 性能设计

统一工作台方案如果设计不当，会把很多视图一次性加载出来，造成明显性能问题。因此必须提前约束。

### 7.1 按视图懒加载

原则：

- 进入知识库详情页时，只加载首屏必要数据
- 二级视图内容按需加载
- 未激活视图不请求数据

例如表格知识库：

- 打开“文件列表”时，不应同时请求 `kb_table_rows`
- 打开“数据视图”时，不应重复拉取完整文件分页

### 7.2 顶部摘要轻量化

详情页顶部只允许请求轻量摘要信息，例如：

- 文档数
- 行数 / item 数
- chunk 数
- 最近更新时间
- 关键状态

不应在首屏同步加载：

- 全量记录
- 全量统计
- 大体量明细

### 7.3 列表必须分页

所有“数据视图”都必须默认采用后端分页：

- `kb_table_rows`
- `kb_qa_rows`
- `kb_web_pages`
- `kb_content_items`

不能为了省事在前端一次性拿全量，再本地分页。

### 7.4 超大列表预留虚拟滚动能力

当单页展示量较大时，应预留虚拟滚动或分块渲染能力。

尤其后续这些场景会很容易变大：

- QA 题库
- 表格记录
- 网页页面
- 转写片段
- 同步对象

### 7.5 查询缓存必须按视图隔离

推荐 query key 约定：

- `['knowledge-base', kbId]`
- `['kb-files', kbId, folderId, page, filters]`
- `['kb-table-rows', kbDocId, page, filters]`
- `['kb-qa-items', kbDocId, page, filters]`
- `['kb-web-pages', kbId, page, filters]`

目的：

- 避免不同视图互相污染缓存
- 避免一次刷新误伤全部视图
- 保证切换视图时命中各自缓存

### 7.6 不做跨视图大接口

不建议设计这种接口：

- 一个接口同时返回文件列表、记录列表、统计、图表

原因：

- 任何一个区域的变化都会让整个大接口失效
- 前端无法按区域做精细缓存
- 接口性能和可维护性都会迅速恶化

应按视图和区域拆成小接口。


## 8. 状态与交互建议

### 8.1 首次引导

每种类型知识库在“还未具备主工作条件”时，都要有自己的引导态。

例如：

- `table`
  - 未定义结构时，引导用户“定义结构 / 上传 Excel 自动生成”
- `qa`
  - 无问答记录时，引导用户“手工新增 / 导入模板”
- `web`
  - 无页面时，引导用户“新增 URL / 导入站点”

补充约束：

- 对于 `table` 类型知识库，“表格结构定义”属于主配置能力，不应埋在“解析分块”的次级区域里
- 更推荐把它作为独立配置分区展示，保证用户首次进入时能快速理解“先定义结构，再导入与解析”的流程
- 首次引导卡片只应出现在主工作区的缺省状态中，不应在已经进入结构定义页后继续重复展示
- 若类型专属配置已经进入工作区二级视图，则引导应优先跳转到该工作区入口，而不是强制跳到一级“全局配置”

### 8.2 视图切换要可记忆

建议保存：

- 当前一级导航
- 当前二级视图

以便：

- 刷新后恢复
- 从创建流程跳转时直接落到目标视图
- 多类型知识库共享统一行为

### 8.3 目录树属于工作台通用组织能力

目录树不应被定义为“文件视图专属组件”，而应被定义为知识库工作台的通用组织导航能力。

原因：

- 所有类型知识库都可能需要目录进行分类与层级管理
- 目录树不仅服务文件治理，也服务数据集、页面、问答、转写片段等对象的组织与范围切换
- 不同视图的差异，更多在于“是否强依赖目录树”和“如何消费当前目录上下文”，而不是“有没有目录树”

因此推荐规则是：

- 原则上所有类型知识库都支持目录树
- 目录树作为工作台左侧稳定导航区存在
- 当前主视图消费“当前选中目录”作为过滤或作用域上下文
- 若某个视图并不强依赖目录树，也不建议系统自动隐藏；优先保持布局稳定，由用户自行折叠

例如表格知识库：

- 文件列表视图：按当前目录查看和治理文件
- 数据视图：按当前目录查看该目录下数据集对应的数据记录或数据范围

也就是说，数据视图不应简单理解为“不需要目录树”，而应理解为“对目录树的依赖方式与文件列表不同，但仍共享目录上下文”。


## 9. 当前实施建议

### P0

- 落地统一工作台文档
- 表格知识库改成“文件管理工作区下的二级切换：文件列表 / 数据视图 / 结构定义”
- 文件树按子视图显示隐藏
- 无 schema 时继续保留强引导

### P1

- 抽出类型视图注册表
- QA 知识库接入二级工作区
- 网页同步知识库接入二级工作区

### P2

- 统一工作台状态同步
- 统一空状态与首次引导组件
- 为超大列表增加虚拟滚动 / 分页增强


## 10. 结论

知识库详情页不应继续沿着“每种类型一个完全独立页面”发展，也不应做成一个“所有类型共用的万能数据页”。

最佳实践是：

- **统一工作台骨架**
- **类型视图注册**
- **类型专属内容组件**

表格知识库当前应作为第一种落地类型，先把：

- 文件列表
- 数据视图
- 结构定义

收敛到同一个主工作区中，作为后续 QA、网页同步、音视频、同步应用等类型扩展的基础模板。
