# 自定义规则分块设计方案

> 文档版本: v0.2  
> 状态: 待确认  
> 目标: 讨论并确认后再进入代码实现

## 1. 背景

当前“文本分块策略 > 自定义”在前端已经展示了分隔符与“支持正则”开关，但后端尚未完整消费这些字段。进一步讨论后，自定义分块不应只停留在“按一个分隔符 split”，否则对法律、制度、合同、标准规范等结构化长文本效果有限。

典型法律文本常见结构：

```text
第一章 总则
第一条 为了规范...
第二条 本办法适用于...

第二章 组织管理
第十条 ...

1.1 适用范围
1.2 定义
```

这类文档的关键不是简单切开，而是识别“章、节、条、款、编号标题”的层级关系，并把标题路径带入 chunk 元数据与检索上下文。

## 2. 设计目标

- 支持普通多分隔符，例如 `\n\n`、`\n`、`。`、`；`、空格等递归兜底切分。
- 支持多个正则规则，例如 `^第[一二三四五六七八九十百千万0-9]+章`、`^第[一二三四五六七八九十百千万0-9]+条`、`^\d+(\.\d+)+\s+`。
- 支持标题识别，把“第一章”“1.1”等识别为标题，而不是普通分隔符。
- 支持层级标题路径，例如 `第一章 总则 / 第一条 目的`，写入 `header_path`、`budget_header_text`、`prompt_header_text`。
- 保持统一 chunk 协议，不新增与 `content_blocks[].source_refs`、`metadata_info.page_numbers` 等冲突的字段。
- 复用现有 MarkdownChunker 的成熟能力，尽量避免重复实现复杂的章节、父子块、独立元素保护逻辑。

## 3. 核心结论

建议不要把自定义规则分块设计成“任意复杂正则脚本”。更稳的方案是：

```text
原始文本
  -> 标题规则识别
  -> 规范化为类 Markdown 章节结构
  -> 复用 MarkdownChunker 的按标题分节、章节内重组、父子块构建逻辑
  -> 输出统一 chunk 协议
```

也就是说，法律类、制度类、标准类文本最适合走“规则识别标题 + Markdown 化 + MarkdownChunker”的路径，而不是直接在 GeneralChunker 中塞很多分隔符逻辑。

## 4. 配置原则

### 4.1 不在后端内置业务预设

本项目是通用知识库，不应在后端固定内置“法律中文”“制度文档”等强业务预设。即使内置了法律规则，也很难覆盖所有真实文档，例如：

- `第一章 / 第一节 / 第一条`
- `一、 / （一） / 1.`
- `1 / 1.1 / 1.1.1`
- `Chapter 1 / Section 1.1`
- 企业内部制度里的自定义编号

因此建议：

- 后端只提供通用规则执行能力。
- 前端负责让用户配置多级标题规则。
- 前端可以提供“模板/快捷填充”，但模板本质仍是前端生成的配置，不作为后端写死逻辑。
- 后端必须对前端传入的规则做严格校验，不能信任前端正则。

### 4.2 标题层级先限制到 3 级

第一版建议只支持 3 级标题：

| 层级 | 语义 | 示例 |
| --- | --- | --- |
| level 1 | 一级标题 | `第一章 总则`、`1 总则` |
| level 2 | 二级标题 | `第一节 一般规定`、`1.1 适用范围` |
| level 3 | 三级标题 | `第一条 目的`、`1.1.1 定义` |

原因：

- 3 级已经覆盖绝大多数通用知识库结构。
- UI 更容易配置和预览。
- 后端层级树更稳定，避免一开始就开放过深结构导致 chunk 拓扑过复杂。
- 未来如果确实需要，可以把 `max_heading_level` 从 3 扩到 6，不破坏核心设计。

## 5. 模式划分

### 5.1 简单自定义模式

适合普通文本、日志、短文档。

能力：

- `separator`: 单个分隔符，兼容当前字段。
- `separators`: 多个普通分隔符，按优先级递归尝试。
- `is_regex`: 当前 `separator` 是否按正则解释。
- `regex_separators`: 多个正则分隔符，仅作为边界，不构建标题层级。

示例：

```json
{
  "chunk_strategy": "general",
  "chunk_size": 500,
  "overlap": 50,
  "separators": ["\n\n", "\n", "。", "！", "？", "；", " "],
  "regex_separators": []
}
```

边界：

- 不负责识别章/节/条层级。
- 不生成父子结构。
- 只用于“尽量按自然边界切分”。

### 5.2 规则分块模式

适合法律、制度、标准、合同、手册等结构化文本，但后端不内置具体行业规则。

建议新增策略名：

```text
rule_based
```

核心能力：

- 使用 `heading_rules` 识别标题。
- 把识别到的标题转换成 Markdown 标题，例如 `# 第一章 总则`、`## 第一条 目的`、`### 1.1 适用范围`。
- 转换后调用现有 MarkdownChunker。
- 标题路径、父子块、叶子块向量化规则沿用 MarkdownChunker。

示例：

```json
{
  "chunk_strategy": "rule_based",
  "chunk_size": 500,
  "overlap": 0,
  "preserve_headings": true,
  "max_heading_level": 3,
  "heading_rules": [
    {
      "name": "一级标题",
      "level": 1,
      "pattern": "^第[一二三四五六七八九十百千万0-9]+章\\s+.+$"
    },
    {
      "name": "二级标题",
      "level": 2,
      "pattern": "^第[一二三四五六七八九十百千万0-9]+节\\s+.+$"
    },
    {
      "name": "三级标题",
      "level": 3,
      "pattern": "^第[一二三四五六七八九十百千万0-9]+条\\s+.*$"
    }
  ],
  "fallback_separators": ["\n\n", "\n", "。", "；", " "]
}
```

如果用户要处理 `1 / 1.1 / 1.1.1`，则前端传入另一组规则：

```json
{
  "chunk_strategy": "rule_based",
  "chunk_size": 500,
  "overlap": 0,
  "preserve_headings": true,
  "max_heading_level": 3,
  "heading_rules": [
    {
      "name": "一级编号",
      "level": 1,
      "pattern": "^\\d+\\s+.+$"
    },
    {
      "name": "二级编号",
      "level": 2,
      "pattern": "^\\d+\\.\\d+\\s+.+$"
    },
    {
      "name": "三级编号",
      "level": 3,
      "pattern": "^\\d+\\.\\d+\\.\\d+\\s+.+$"
    }
  ],
  "fallback_separators": ["\n\n", "\n", "。", "；", " "]
}
```

## 6. 为什么适合 Markdown

现有 MarkdownChunker 已经具备以下能力：

- 按 Markdown 标题预分块，维护 `heading`、`level`、`header_path`。
- 对小章节保留完整块，对中等章节生成章节块 + 子块，对大章节生成章节块 + 父块 + 子块。
- 所有叶子块按 `is_leaf=true` 向量化，父块/章节块作为上下文层。
- 支持表格、代码块、列表、公式等独立元素保护。
- 统一输出 `content_blocks`、`metadata.source_anchors`、`metadata.page_numbers`、`metadata.source_element_indices` 等协议字段。

因此规则分块器不应重复造一套层级 chunker，而应做“标题识别和规范化”这一层适配。

## 7. 标题识别规则设计

### 7.1 通用 HeadingRule

后端只认识通用 `HeadingRule`：

```json
{
  "name": "一级标题",
  "level": 1,
  "pattern": "^第[一二三四五六七八九十百千万0-9]+章\\s+.+$",
  "keep_heading": true
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `name` | string | 规则名称，仅用于日志、预览、调试 |
| `level` | int | 标题层级，第一版只允许 1、2、3 |
| `pattern` | string | 单行正则，必须匹配整行标题 |
| `keep_heading` | bool | 是否把标题行保留到正文，默认 true |

### 7.2 规则匹配顺序

同一行可能命中多个规则，例如 `1.1 适用范围` 可能被宽泛的 `^\d+.*` 命中，也可能被 `^\d+\.\d+.*` 命中。

建议规则：

- 按前端传入的 `heading_rules` 顺序匹配。
- 命中第一条后停止。
- UI 中提示用户把更具体的规则放在前面。
- 后端不自动猜业务优先级，只保证执行确定性。

### 7.3 混合标题

实际文档可能同时出现：

```text
第一章 总则
1.1 适用范围
1.2 术语定义
第二章 管理要求
2.1 职责
```

建议处理方式：

- `第一章` 是否作为 level 1，由前端配置决定。
- `1.1`、`1.2` 是否作为 level 2，由前端配置决定。
- 如果出现 `第十条`，它是 level 2 还是 level 3，也由前端配置决定。
- 后端只根据 `level` 维护标题栈并构造 `header_path`。

## 8. 前端配置设计

### 8.1 UI 形态

建议自定义分块增加一个“规则类型”：

```text
自定义分块
  - 普通分隔符
  - 多级标题规则
```

选择“多级标题规则”后显示 3 级配置：

| 层级 | 名称 | 正则 |
| --- | --- | --- |
| 一级标题 | 用户可改 | 输入框 |
| 二级标题 | 用户可改 | 输入框 |
| 三级标题 | 用户可改 | 输入框 |

每一级可以为空。为空表示不启用该级标题。

### 8.2 前端模板

虽然后端不内置预设，但前端可以提供“填入示例模板”：

- 中文章/节/条模板。
- 数字编号模板。
- 中文大写序号模板。
- 清空模板。

模板只是在前端把输入框填上，不改变后端通用设计。

### 8.3 规则预览

强烈建议提供“预览识别结果”：

```text
第一章 总则
  第一条 目的
  第二条 适用范围
第二章 管理职责
  第十条 职责划分
```

预览可以先前端本地做一次轻量匹配，也可以调用后端接口。最终以后端校验为准。

## 9. 分块流程

### 9.1 规则预处理

```text
输入文本
  -> 按行扫描
  -> 对每行按 heading_rules 顺序匹配
  -> 命中标题规则时生成标题事件
  -> 未命中时作为普通正文
```

标题事件包含：

```json
{
  "line_no": 12,
  "raw_text": "第一章 总则",
  "level": 1,
  "rule_name": "chapter_cn",
  "normalized_heading": "第一章 总则"
}
```

### 9.2 Markdown 规范化

将标题行转为 Markdown 标题：

```text
第一章 总则
第一条 为了规范人工智能服务...
```

转换为：

```markdown
# 第一章 总则

## 第一条 为了规范人工智能服务...
```

对 `1.1` 类标题：

```text
1.1 适用范围
```

转换为：

```markdown
## 1.1 适用范围
```

### 9.3 调用 MarkdownChunker

```text
规范化 Markdown
  -> MarkdownChunker.parse_by_heading()
  -> MarkdownChunker._process_section()
  -> 输出 section / parent / child chunks
```

### 9.4 元数据补充

规则分块器应在 chunk metadata 中补充：

```json
{
  "chunk_strategy": "rule_based",
  "source_chunk_strategy": "markdown",
  "heading_rule_name": "三级标题",
  "heading_rule_level": 3
}
```

注意：

- 不新增与 chunk 协议重复的来源字段。
- `header_path`、`prompt_header_text`、`budget_header_text` 继续沿用 MarkdownChunker 字段。
- `source_anchors`、`page_numbers`、`source_element_indices` 仍作为 chunk 级聚合元数据。
- `content_blocks[].source_refs` 仍是块级权威来源字段，不在规则分块器里另造替代字段。

## 10. “条”或三级标题是否应该单独成块

法律文档中的“条”、制度文档中的 `1.1.1`、手册中的三级小节通常是较适合作为检索单元的粒度，但也不能绝对一条一块。

建议策略：

- 如果单条内容 `total_tokens <= chunk_size * 1.5` 且不超过 embedding 上限，则整条作为一个叶子块。
- 如果单条很长，则保留完整“条”作为上层块，再拆成多个叶子子块。
- 如果多个连续条文都很短，可以合并为一个叶子块，但必须保留 `prompt_header_paths`，避免召回后丢失条文定位。
- 一级/二级标题一般不向量化，作为上下文父层更合适。

示例：

```text
第一章 总则
  第一条 ...        -> 叶子块或条父块
  第二条 ...        -> 叶子块
第二章 管理职责
  第十条 ...        -> 叶子块或条父块
```

## 11. 与现有策略的边界

| 策略 | 适用 | 是否识别标题 | 是否建议用于法律 |
| --- | --- | --- | --- |
| `general` | 普通文本 | 否 | 不优先 |
| `recursive` | 多分隔符递归切分 | 否 | 可兜底 |
| `markdown` | 已有 Markdown 文档 | 是 | 推荐 |
| `pdf_layout` | PDF 版面定位优先 | 按版面元素 | PDF 精准定位推荐 |
| `rule_based` | 用户配置了多级标题规则的文本 | 是 | 推荐新增 |

如果解析器已经输出高质量 Markdown，直接用 `markdown`。如果解析器只输出纯文本，但文本里有“第一章、1.1”等可识别标题，则用 `rule_based`。

## 12. 前端交互建议

普通用户界面：

- 分块模式：智能 / 自定义 / 不分块。
- 自定义类型：普通分隔符 / 标题规则。
- 普通分隔符：输入多个分隔符，支持一行一个。
- 标题规则：用户配置 1 到 3 级标题正则。

高级配置界面：

- 允许编辑 `heading_rules`，第一版最多 3 条有效层级。
- 每条规则包含：名称、层级、正则、是否保留标题、是否作为独立边界。
- 提供“测试规则”按钮，展示识别出的标题树。

示例预览：

```text
第一章 总则
  第一条 目的
  第二条 适用范围
第二章 管理职责
  第十条 职责划分
    1.1 管理部门
    1.2 执行部门
```

## 13. 后端实现建议

### 13.1 Schema

建议先保留兼容字段：

```python
separator: str = "\n\n"
is_regex: bool = False
```

新增结构化字段：

```python
separators: list[str] = []
regex_separators: list[str] = []
rule_preset: str | None = None
heading_rules: list[HeadingRule] = []
fallback_separators: list[str] = []
preserve_headings: bool = True
max_heading_level: int = 3
```

其中 `rule_preset` 可不实现，或仅作为前端透传的显示字段；后端不根据它加载内置业务规则。

### 13.2 分块器

新增 `RuleBasedChunker`：

```text
RuleBasedChunker
  -> HeadingRuleDetector
  -> MarkdownNormalizer
  -> MarkdownChunker
```

职责边界：

- `HeadingRuleDetector`: 只负责识别标题事件和层级。
- `MarkdownNormalizer`: 只负责把标题行转换为 Markdown 标题。
- `MarkdownChunker`: 继续负责真正的章节分块、层级构建、独立元素保护。

### 13.3 正则安全

必须处理：

- 正则编译失败：创建/更新配置时直接报错，或任务执行时明确失败。
- 空匹配风险：禁止会匹配空字符串的标题规则。
- 规则数量上限：第一版建议最多 3 个启用层级，最多 10 条规则。
- 单条 pattern 长度上限：建议最多 512 字符。
- 行级匹配：标题规则只对单行执行，避免跨全文灾难性回溯。
- 标题规则必须是行级规则，建议前端自动补充 `^...$`，后端不强制改写用户正则，但文档提示必须匹配整行。

## 14. 测试用例

必须覆盖：

- 单层法律标题：`第一章`、`第二章`。
- 章/节/条混合：`第一章`、`第一节`、`第一条`。
- 数字编号：`1`、`1.1`、`1.1.1`。
- 混合编号：`第一章` 下包含 `1.1`、`1.2`。
- 条文超长时，能拆成子块并保留标题路径。
- 短条文相邻合并时，不丢 `prompt_header_paths`。
- 非法律普通文本走 fallback 分隔符，不误识别大量正文。
- 非法正则给出明确错误。
- 中文内容文件保持 UTF-8 无 BOM。
- 只配置一级标题时，能正确按一级标题分节。
- 配置一二三级标题时，能正确构造 `header_path`。
- 空规则时不能启用 `rule_based`，应提示至少配置一级标题规则。

## 15. 推荐实施路线

第一阶段：

- 修复当前 `separator/is_regex` 未贯通问题。
- 支持 `separators` 多分隔符递归切分。
- 增加基础测试。

第二阶段：

- 新增 `rule_based` 策略。
- 实现 `HeadingRuleDetector` 和 `MarkdownNormalizer`。
- 复用 MarkdownChunker 输出层级 chunk。
- 支持前端传入最多 3 级标题规则，不做后端业务预设。

第三阶段：

- 前端增加“标题规则”高级配置。
- 增加规则识别预览。
- 支持前端模板快速填充，但模板不下沉为后端固定预设。

## 16. 待确认问题

- `rule_based` 是否作为新的 `chunk_strategy`，还是作为 `markdown` 的预处理模式。
- 第一版是否严格限制最多 3 级标题。
- 短条文是否允许合并，默认阈值是多少。
- 正则配置错误时，是保存时校验失败，还是任务执行时失败。
- 是否需要后端提供“规则预览”接口，还是前端本地预览即可。

## 17. 当前建议

建议采用“新增 `rule_based` 策略 + 前端配置最多 3 级标题规则 + 后端预处理成 Markdown + 复用 MarkdownChunker”的方案。

理由：

- 法律/制度/手册类文档的核心是标题层级，不是单纯分隔符。
- 通用知识库不应在后端写死行业规则，标题规则由前端配置更灵活。
- 现有 MarkdownChunker 已经实现了较完整的层级 chunk、父子块、叶子向量化和独立元素保护。
- 规则分块器只做标题识别与 Markdown 规范化，职责清晰，后续维护成本低。
- 协议上继续复用现有 `header_path`、`is_root`、`is_leaf`、`depth` 等字段，不破坏统一 chunk 协议。

## 18. 设计复盘与补强

### 18.1 是否足够强大

当前方案足够覆盖第一阶段的通用知识库需求，原因如下：

- 多级标题规则由前端配置，不绑定法律、制度、手册等固定业务场景。
- 后端只做通用结构识别，规则可表达 `第一章`、`一、`、`（一）`、`1.1`、`Chapter 1` 等不同编号体系。
- 识别后转成 Markdown，再复用 MarkdownChunker，可直接获得标题路径、父子块、叶子块向量化、独立元素保护等能力。
- 限制 3 级标题能覆盖大多数通用知识库文档，也能降低 UI、校验、调试复杂度。

但它不是无限强大的文档理解系统，第一版不建议支持以下能力：

- 跨行标题识别。
- 根据字体、字号、缩进自动识别标题。
- 复杂目录自动解析。
- 同一行拆成“标题 + 正文”后分别建不同块。
- 任意深度标题树。

这些能力应留给解析器、版面分析器或后续增强模块，而不是放进自定义规则分块第一版。

### 18.2 最佳实践判断

从 RAG 最佳实践看，推荐“结构优先，长度兜底”：

```text
标题结构识别
  -> 章节级语义边界
  -> 章节内按 token / 字符预算重组
  -> 超长内容再用自然分隔符兜底
  -> 叶子块用于向量化
  -> 父块/章节块用于上下文扩展
```

不推荐“先按长度硬切，再试图补标题”，因为这样容易把标题和正文拆散，检索召回时也更难恢复 `header_path`。

因此，本方案里 `rule_based -> MarkdownNormalizer -> MarkdownChunker` 是更好的实践。

### 18.3 需要补强的关键约束

为避免规则过强导致误识别，建议增加以下约束：

- 标题规则默认只匹配单行。
- 标题行建议长度上限，例如 120 字符，超过则不作为标题，除非用户显式关闭。
- 标题规则建议使用整行匹配，即 `^...$`。
- 同一行命中多个规则时，按配置顺序取第一条。
- 标题等级跳跃时允许自动修正，例如没有 level 1 直接出现 level 2，可以临时提升为 level 1，或挂到虚拟根节点。
- 如果全文标题命中率异常高，例如超过总行数 40%，应记录告警，提示规则可能过宽。
- 如果完全没有命中标题规则，降级到 `fallback_separators` 或 `general`，并记录任务日志。

### 18.4 3 级标题限制是否合理

第一版限制 3 级是合理的，但实现上不要把内部结构写死成只能 3 层。

建议：

- UI 第一版只展示 3 级。
- Schema 第一版限制 `level` 只能为 1、2、3。
- 后端标题栈实现用通用 `dict[int, str]`，不要写死三个变量。
- Markdown 规范化按 `#` 数量生成，未来扩到 6 级时只改校验和前端。
- Chunk 输出继续使用 `is_root/is_leaf/depth`，不要恢复旧的固定 L0/L1/L2 设计。

### 18.5 前端规则配置的最佳形态

建议前端不要直接给用户一个大 JSON，而是表单化：

| 启用 | 层级 | 名称 | 正则 |
| --- | --- | --- | --- |
| 是 | 1 | 一级标题 | 输入框 |
| 是 | 2 | 二级标题 | 输入框 |
| 是 | 3 | 三级标题 | 输入框 |

配套能力：

- 提供“插入示例模板”，但模板只在前端填充，不作为后端固定预设。
- 提供“测试前 200 行”预览，展示标题树和命中规则。
- 对过宽规则给前端提示，例如 `^.*$`、`.*`、空匹配。
- 保存前调用后端校验接口，避免任务运行时才失败。

### 18.6 后端执行的最佳边界

后端应该做：

- 校验规则。
- 执行规则。
- 生成规范化 Markdown。
- 复用 MarkdownChunker。
- 记录命中统计与降级原因。

后端不应该做：

- 根据 `rule_preset` 隐式加载法律规则。
- 自动猜测用户想要的业务标题层级。
- 静默吞掉非法正则。
- 在 `metadata_info` 中新增与 `content_blocks[].source_refs` 重叠的来源字段。

### 18.7 降级策略

建议明确降级优先级：

```text
rule_based
  -> 如果规则非法: 配置校验失败，不进入任务
  -> 如果规则合法但无标题命中: fallback_separators
  -> 如果 fallback 也无法产生合理块: general
  -> 如果 general 不可用: fixed_size
```

所有降级都应写入任务日志，方便用户知道“为什么我的标题规则没有生效”。

### 18.8 最终复盘结论

当前方向是强大且符合最佳实践的，但前提是把它定位为“可配置标题结构识别器”，不是“任意正则分块器”。

第一版推荐收敛为：

- 最多 3 级标题规则。
- 前端配置规则，后端不内置业务规则。
- 后端做严格校验和单行匹配。
- 命中标题后规范化为 Markdown。
- 复用 MarkdownChunker 做真正分块。
- 规则无命中时明确降级并记录日志。

这套方案既保留通用性，又能覆盖法律、制度、标准、合同、手册等强结构文档，同时不会破坏现有统一 chunk 协议。
