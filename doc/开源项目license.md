# 开源项目协议及限制说明

本项目的开发依赖于众多的开源项目。为了确保合规使用，本篇文档列出了项目中使用的主要开源库及其对应的开源协议，并对常见开源协议的限制进行了说明。

## 1. 常见开源协议深度对比

| 许可协议 | 类型 | 传染强度 | 修改后需开源 | 闭源可用 | 与 AGPL 兼容 | 典型场景 | 限制/注意点 |
| :--- | :--- | :---: | :---: | :---: | :---: | :--- | :--- |
| **AGPL-3.0** | 强 Copyleft | **最强** | 是 (含网络) | 否 | 是 | SaaS、强开源项目 | 最严格，网络交互也算分发 |
| **GPL-3.0** | 强 Copyleft | 强 | 是 | 否 | 是 | 桌面/服务器软件 | 不包括网络交互豁免 |
| **LGPL-3.0** | 弱 Copyleft | 中 | 仅修改的部分 | 是 | 否 | 库/组件 | 允许链接引用，修改需源码 |
| **MPL-2.0** | 文件级 | 中弱 | 仅修改的文件 | 是 | 否 | Mozilla 相关 | 只传染修改的文件，不传染项目 |
| **Apache 2.0**| Permissive | 无 | 否 | 是 | 否 | 企业级、最大兼容 | 需保留版权声明与专利授权 |
| **MIT** | Permissive | 无 | 否 | 是 | 否 | 最大采用率 | 几乎无限制，仅需保留声明 |
| **BSD-3** | Permissive | 无 | 否 | 是 | 否 | 类似 MIT | 不允许使用原作者名义推广 |

## 2. 协议兼容性说明 (核心风险)

*   **兼容 AGPL 的选择**：如果项目核心依赖了 AGPL 组件（如 PyMuPDF），项目本身基本只能选择 **AGPL-3.0** 或 **GPL-3.0**。
*   **不兼容的冲突**：MIT、Apache 2.0、BSD 等宽松许可**不能**直接作为依赖 AGPL 组件的项目许可。这意味着如果你坚持要用 MIT 开源 Genesis AI，你必须先移除对 PyMuPDF 的依赖。

## 3. Genesis AI 的实际选择 (项目演进路径)

根据目前项目使用 PyMuPDF 的现状，你有以下四种选择：

1.  **追求极致宽松 (MIT/Apache)**：
    *   **动作**：必须换掉 PyMuPDF。
    *   **替代方案**：使用 `pdfplumber`、`pypdf`、`pikepdf` 或 `pdfminer.six`。
2.  **全拥抱开源 (AGPL)**：
    *   **动作**：继续使用 PyMuPDF，但 Genesis AI 整体采用 **AGPL-3.0** 许可。
    *   **建议**：在项目的 README 和 LICENSE 文件中明确标明。
3.  **未来商业化/不想传染**：
    *   **动作**：现在就购买 Artifex 的商业许可。
    *   **结果**：你的主项目可以继续使用 MIT 协议且不被传染，适合做闭源商业产品。
4.  **混合隔离方案 (高难度)**：
    *   **动作**：将 PyMuPDF 相关的解析代码做成独立的 AGPL 微服务。
    *   **结果**：主平台通过 API 与该服务交互。虽然在法律界有争议，但能物理上减少代码直接传染的面积。

## 4. 后端/前端核心组件协议一览 (简表)

1.  **商业化分发**：如果计划对 Genesis AI 进行闭源商业化包装并分发，请务必核查 **PyMuPDF** 的使用方式。如果仅作为内部工具或 SaaS 服务使用，根据协议要求可能仍需提供修改后的源码（特别是 AGPL 协议）。
2.  **保留声明**：在发布软件时，应在软件的“关于”或“法律信息”页面保留所使用开源库的版权声明和协议原文。
3.  **协议传染性**：尽量避免直接修改 **GPL/AGPL** 类协议项目的源代码。如果必须修改，请评估其对整体项目代码开源的要求。

## 4. 避坑指南：关于 AGPL 与项目开源的 Q&A

**Q: 我只是 `pip install` 并没有修改 PyMuPDF 的源码，我的项目需要开源吗？**
*   **A:** **很有可能需要。** 如果你使用的是 PyMuPDF 的最新版本（AGPL 协议），且你的项目是一个 Web 平台（如 Genesis AI），根据 AGPL 的“远程网络交互”条款，只要用户通过网络使用你的服务，你就必须应用户要求提供整个项目的源码。无论你是否修改了库的源码，只要存在调用关系，就会触发“传染性”。

**Q: 商业项目如何合法避开 PyMuPDF 的开源传染？**
1.  **购买商业授权**：PyMuPDF 是双协议模式，联系 Artifex 公司购买商业 License 即可在闭源项目中使用。
2.  **替换技术栈**：使用 MIT/Apache 协议的替代品，如 `pypdf`（原 PyPDF2）、`pdfminer.six` 或 `pdfplumber`。这些库虽然性能或解析质量可能略逊于 PyMuPDF，但商业使用无任何风险。
3.  **解耦部署**：将 PDF 解析逻辑剥离成一个独立的微服务（只包含解析代码并开源这一个微服务），主平台通过 API 调用它。这种方式在法律界仍有争议，但被许多企业作为折中方案。

## 附录：如何查询实时 License 列表

如果您新增了依赖项，可以参考以下命令重新生成详细的 `licenses.md` 文件。

~~~shell
# 1. 切换到平台目录
cd genesis-ai-platform

# 2. 安装依赖 (如果尚未安装)
uv add --dev pip-licenses

# 3. 生成 markdown 格式的详细列表
pip-licenses --format=markdown --with-system > ../doc/licenses_detail.md
~~~
