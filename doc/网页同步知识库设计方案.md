# 网页同步知识库设计方案

## 1. 文档目的

本文档用于明确 `web` 类型知识库的初版落地方案，目标是：

- 复用现有 `documents / knowledge_base_documents / chunks` 主链路
- 为网页同步引入独立的 `kb_` 业务表，避免将网页特有语义塞入通用挂载表
- 支持人工同步与定时同步
- 支持定时抓取完成后自动触发分块与后续训练链路
- 为后续扩展飞书文档、GitHub、云雀笔记等远程资源类型提供统一设计参考


## 2. 当前设计结论

网页同步知识库采用如下分层：

1. `documents`
   作为统一载体对象表，网页资源在这里创建一条远程载体记录。

2. `knowledge_base_documents`
   作为知识库内的挂载实例与运行态入口，负责目录归属、启停状态、解析状态、chunk 数、运行阶段等。

3. `kb_web_pages`
   作为网页主事实表，一条 URL 对应一个网页资源实体。

4. `kb_web_page_versions`
   作为网页抓取版本表，仅在“首次成功”或“内容发生变化”时沉淀版本/快照信息。

5. `kb_web_sync_schedules`
   作为同步调度规则表，支持知识库默认规则与页面级覆盖规则，支持多个时间配置。

6. `kb_web_sync_runs`
   作为同步执行记录表，用于保存每次同步任务的触发来源、状态、错误、变更结果与耗时。

7. `chunks`
   作为网页正文的检索投影层，由网页版本内容分块生成。


## 2.1 关键决策（最终）

- 表名保持 `kb_web_sync_runs`，不改为 `kb_web_sync_logs`
  原因：该表表示一次同步执行实例（run），不是纯日志流；`logs_json` 只是执行明细摘要字段。

- 版本保留采用“保留历史 + 生命周期清理”
  原因：仅保留最新版本不利于排障和回滚；固定只保留 3-5 个版本通常过短。

- 变更检测采用 `content_hash`，并维护最近同步时间
  规则：内容无变化不重建 chunks，但同步成功后必须更新最近同步时间字段。


## 3. 设计原则

- 一条 URL 就是一个独立的网页资源对象
- `documents` 只表达载体，不承担网页业务语义
- `knowledge_base_documents` 只表达挂载与处理状态，不承担网页主事实
- 网页特有字段全部下沉到 `kb_web_*` 表
- `knowledge_bases / knowledge_base_documents` 只承载抓取与处理配置，不承载调度规则实体
- 定时规则按“单条规则单行存储”设计，天然支持多个时间配置
- 抓取完成后，以“内容版本”为上游事实源，统一进入分块链路
- 若内容无变化，可根据配置决定是否跳过重建 chunks


## 4. 与现有载体模型的协同方式

网页资源创建时，`documents` 按当前载体模型固定赋值：

- `carrier_type = 'web_page'`
- `asset_kind = 'remote'`
- `source_type = 'crawl'`

同时建议：

- `name`
  默认使用页面标题；若首次创建时尚未抓取成功，则先使用 URL 或域名路径生成占位名称

- `source_url`
  保存用户录入的原始 URL

- `mime_type`
  初版可写 `text/html`

- `file_key`
  对远程网页载体可使用稳定逻辑键，例如 `remote/web/{kb_id}/{sha256(normalized_url)}`

- `file_size`
  初版可置 `0`，后续若需要也可记录最近一次抓取正文大小

- `metadata`
  可记录 `domain / normalized_url / canonical_url / fetch_mode` 等非核心展示信息

说明：

- 不将网页正文直接存入 `documents`
- 不在 `documents` 中承载同步配置、调度规则、抓取状态等网页业务字段


## 5. 数据关系

推荐关系如下：

- 1 个 `documents`
  - 对应 1 个 `knowledge_base_documents`
  - 对应 1 个 `kb_web_pages`

- 1 个 `kb_web_pages`
  - 对应 0..N 个 `kb_web_page_versions`
  - 对应 0..N 个 `kb_web_sync_schedules`
  - 对应 0..N 个 `kb_web_sync_runs`

- 1 个 `kb_web_page_versions`
  - 对应 0..N 个 `chunks`


## 6. 表结构设计

说明：

- `knowledge_bases`
  适合保存网页知识库默认抓取配置，例如 `fetch_mode / timeout / retry / force_rechunk_on_sync / extractor_preference`

- `knowledge_base_documents`
  适合保存某一网页资源在当前知识库挂载下的页面级处理配置覆盖，例如 `enable_js_render / custom_headers / parse_overrides`

- 当前实现中，网页文档的 `parse/chunk/enhance/train` 并未自建旁路
  `web_sync` 负责“抓取 + 正文抽取 + 版本化”，`parse` 阶段直接复用 `kb_web_page_versions.content_text`，随后继续进入统一分块与训练链路

- 调度规则本身不放入上述两张表的 `metadata / config`
  因为调度规则属于独立业务对象，必须支持多条规则、单条启停、作用域继承、索引查询、`next_trigger_at` 计算与后台调度扫描

- 因此，调度规则按最佳实践必须独立落在 `kb_web_sync_schedules`

  - 建议值：`manual / sitemap / discovered`
- `fetch_mode`
  - 建议值：`auto / static / browser`
- `sync_status`
  - 建议值：`idle / queued / syncing / success / partial_success / failed`
- `content_status`
  - 建议值：`new / unchanged / changed / gone`
- `last_success_version_id`
- `last_synced_at`
- `last_content_changed_at`
- `next_sync_at`
- `last_error`

### 6.2 `kb_web_page_versions`

网页版本表，用于保存每次抓取后的稳定版本信息。

建议字段：

- `id`
- `tenant_id`
- `kb_id`
- `kb_web_page_id`
- `kb_doc_id`
- `document_id`
- `version_no`
- `is_current`
- `is_material_change`
- `trigger_type`
  - 建议值：`manual / scheduled / created / retry`
- `fetch_started_at`
- `fetch_ended_at`
- `fetch_status`
  - 建议值：`success / failed`
- `http_status`
- `final_url`
- `canonical_url`
- `title`
- `site_name`
- `extractor`
  - 建议值：`trafilatura / playwright / playwright+readability / crawl4ai_sdk / crawl4ai_service / custom`
- `raw_html_document_id`
  - 可选，若保留原始 HTML 快照载体则关联到 `documents`
- `markdown_document_id`
  - 可选，若保存抽取后 Markdown 快照载体则关联到 `documents`
- `content_text`
  - 当前版本的清洗后正文
  - 当前实现中主链路为 Markdown 风格正文；仅在最终 `readability-lxml` 兜底分支时可能退化为纯文本
- `content_hash`
- `text_length`
- `change_summary`
- `diff_from_version_id`
- `chunk_status`
  - 建议值：`pending / queued / processing / completed / failed`
- `chunk_count`
- `extra_metadata`
- 审计字段

说明：

- 初版可以不强制为每个版本都创建物理快照载体
- 但推荐保留 `markdown_document_id` 扩展位，后续前端预览和问题排查会更方便
- 建议仅在“首次抓取成功”或“内容发生变化”时创建版本记录
- 内容无变化的同步尝试应记录在 `kb_web_sync_runs`，不新增版本行
- 当前实现中会尽量保存 `raw_html` 最新快照；`content_text` 作为后续 parse/chunk 的事实输入
- 若后续引入 `Crawl4AI`，建议仍保持 `content_text / raw_html_document_id / markdown_document_id` 这组统一出口，不让下游感知底层抽取器差异


### 6.3 `kb_web_sync_schedules`

同步规则表，一条规则一行，支持多个时间配置。

设置独立表的原因：

- 调度规则不只是配置项，而是需要被调度器直接消费的执行实体
- 一条页面可能存在多条规则，知识库默认规则与页面覆盖规则也需要共存
- 调度器需要高效查询“当前应执行的规则”，因此需要基于 `is_enabled / next_trigger_at / scope_level` 建立索引
- `last_triggered_at / next_trigger_at / start_at / end_at` 属于运行态调度字段，不应与抓取配置混放在 JSON 中
- 单独成表后，前端规则管理、启停控制、审计与排障都更清晰

建议字段：

- `id`
- `tenant_id`
- `kb_id`
- `kb_web_page_id` 可空
- `scope_level`
  - 建议值：`kb_default / page_override`
- `schedule_type`
  - 建议值：`manual / interval / daily / weekly / monthly / once / cron`
- `cron_expr` 可空
- `timezone`
- `interval_value` 可空
- `interval_unit`
  - 建议值：`minute / hour / day`
- `run_time` 可空
- `run_date` 可空
- `weekdays`
  - JSONB 数组，如 `[1,2,3,4,5]`
- `monthdays`
  - JSONB 数组，如 `[1,15,30]`
- `start_at`
- `end_at`
- `priority`
- `is_enabled`
- `last_triggered_at`
- `next_trigger_at`
- `extra_metadata`
- 审计字段

规则说明：

- `kb_web_page_id IS NULL`
  表示知识库级默认规则

- `kb_web_page_id IS NOT NULL`
  表示页面级覆盖规则

- 页面可以选择：
  - 继承知识库默认规则
  - 使用自己的独立规则

- `kb_web_page_id`
- `kb_doc_id`
- `version_id` 可空
- `schedule_id` 可空
- `trigger_type`
  - 建议值：`manual / scheduled / created / retry`
- `status`
  - 建议值：`queued / running / success / partial_success / failed / cancelled`
- `started_at`
- `ended_at`
- `duration_ms`
- `http_status`
- `content_changed`
- `old_content_hash`
- `new_content_hash`
- `chunks_before`
- `chunks_after`
- `error_message`
- `logs_json`
- `triggered_by_id`
- `triggered_by_name`
- `created_at`

命名说明：

- 主表采用 `runs` 命名，表达“一次同步执行实例”的完整生命周期（触发、执行、结束、结果）。
- 如果后续需要毫秒级/步骤级日志流，建议新增从表（如 `kb_web_sync_run_events`），不建议把主表改名为 `logs`。


## 7. 配置继承策略

建议采用“处理配置”和“调度规则”分层设计。

### 7.1 处理配置继承

处理配置采用三层覆盖：

1. 知识库全局默认配置
   放在 `knowledge_bases` 现有配置体系中，用于定义网页抓取默认策略与处理参数。

2. 知识库默认调度规则
   不放在 `knowledge_bases` 中，而是放在 `kb_web_sync_schedules(scope_level='kb_default')` 中。

3. 页面级覆盖配置
   放在 `kb_web_pages.config_json` 中，用于覆盖抓取与正文抽取参数。

优先级：

- 页面级显式配置
- 知识库级默认配置
- 系统默认值

### 7.2 调度规则继承

调度规则单独继承，不与处理配置混用：

1. 知识库默认调度规则
   位于 `kb_web_sync_schedules(scope_level='kb_default')`

2. 页面级覆盖调度规则
   位于 `kb_web_sync_schedules(scope_level='page_override')`

页面调度建议支持两种模式：

- 继承知识库默认规则
- 使用页面独立规则

说明：

- 不建议把 `cron_expr / interval / multiple_times / next_trigger_at` 这类字段直接塞进 `knowledge_bases` 或 `knowledge_base_documents` 的 `metadata`
- 这些字段需要被调度器直接查询与更新，独立表更符合最佳实践


## 8. 定时规则表达方式

前端界面建议支持“简单模式 + Cron 高级模式”。

### 8.1 简单模式支持

- 手动同步
- 每隔多久执行一次
  - 例如每 30 分钟
  - 例如每 2 小时
  - 例如每 3 天
- 每天固定时间
  - 支持多个时间点，例如每天 `09:00`、`18:00`
- 每周固定时间
  - 支持周一到周五 `09:00`
- 每月固定日期时间
  - 支持每月 `1 日 09:00`、`15 日 18:00`
- 指定日期时间执行一次

### 8.2 高级模式支持

- 直接输入 `cron_expr`
- 实时校验表达式
- 实时展示未来若干次触发时间
- 保存后回写到 `kb_web_sync_schedules`

### 8.3 多个时间配置的落地方式

多个时间配置不放在一个 JSON 字段里，而是：

- 每一条配置对应一条 `kb_web_sync_schedules`
- 同一个页面可以存在多条启用规则
- 同一个知识库默认规则也可以存在多条启用规则

### 8.4 动态调度执行架构（当前推荐）

针对“调度时间动态变化”的特点，当前阶段推荐：

- 不引入新的调度框架
- 继续使用现有 `Celery + Celery Beat`
- 保留 `beat` 独立进程（必须运行）

执行方式采用“数据库驱动分发器”：

1. `Celery Beat` 固定频率触发分发任务（例如每 30 秒或 1 分钟）
2. 分发任务查询 `kb_web_sync_schedules`：
   - `is_enabled = true`
   - `next_trigger_at <= now()`
3. 将命中规则写入 `kb_web_sync_runs`（`status='queued'`）并投递抓取任务
4. 原子更新规则：
   - `last_triggered_at`
   - `next_trigger_at`
5. 使用分布式锁（Redis）避免多分发实例重复投递

说明：

- 该模式天然支持规则动态增删改，不需要把每条规则注册到静态 `beat_schedule`
- 运维成本最低，且与本项目已有 Celery 体系最匹配

### 8.4.1 分布式锁实现建议（Redis）

结论：

- 分布式锁采用 Redis 是推荐方案
- 但必须配合数据库幂等约束，不能只依赖锁

推荐锁设计：

- 分发级全局锁（可选）：
  - `lock:web_sync:dispatch:{slot}`
  - 用于避免同一时间窗口被多个分发器重复扫描

- 规则级锁（建议）：
  - `lock:web_sync:schedule:{schedule_id}`
  - 用于保证同一调度规则在同一时刻只被一个分发实例处理

- 页面级执行锁（强烈建议）：
  - `lock:web_sync:page:{kb_web_page_id}`
  - 用于保证同一页面不会并发执行多个抓取任务

加锁参数建议：

- `SET key value NX EX <ttl>` 原子加锁
- `ttl` 建议：
  - 分发级：60-120 秒
  - 规则级：120-300 秒
  - 页面级：按平均抓取时长设置（例如 10-20 分钟）

- 锁值使用随机 token，解锁时仅允许持有者删除（Lua 脚本校验 token）
- 长任务需续租（watchdog 心跳）避免锁提前过期

失败与兜底：

- 抢锁失败：当前实例直接跳过该规则/页面
- 锁过期后重试：允许后续周期重新争抢
- Worker 异常退出：依赖 TTL 自动释放，避免死锁

数据库幂等兜底（必须）：

- `kb_web_sync_runs` 建议增加幂等键（例如 `dedupe_key`）
  - 示例：`sha256(kb_web_page_id + schedule_id + trigger_window_start)`
- 对“同一页面 + 运行中状态”做唯一约束或条件约束
- 入队前二次检查是否已存在 `queued/running` 任务，避免重复执行

说明：

- Redis 锁解决“并发竞争”
- DB 幂等解决“最终一致性与重复写入”
- 两者组合才能在多实例场景下稳定防重

### 8.5 高调度量与同一时刻拥塞治理

当同一时刻命中大量规则时，需要防止“同步阻塞”。

建议策略：

- 阶段一：分发与执行解耦
  - 分发任务只负责“找出到期规则并投递”
  - 抓取、抽取、分块在 Worker 队列异步执行
  - 禁止分发任务内同步抓取网页

- 阶段二：队列与并发治理
  - 单独队列：`web_sync`（抓取）与 `chunk/train`（后处理）隔离
  - Worker 并发限流：按机器资源设置 `--concurrency`
  - 域名级限流：同域并发上限（避免被封与热点阻塞）
  - 全局速率限制：每分钟最大抓取数与重试退避

- 阶段三：批量分片投递
  - 分发任务按批次投递（例如每批 100 条）
  - 采用“下一次分发继续接力”的方式，避免单次分发任务过长

- 阶段四：幂等与去重
  - 同一 `kb_web_page_id` 同时只允许一个 `running` 任务
  - 重复触发时仅记录 `queued` 或直接跳过（按策略）
  - 通过 Redis 页面锁 + 数据库唯一键/乐观锁双保险避免重复入队

- 阶段五：削峰与优先级
  - `priority` 字段用于紧急页面先执行
  - 支持抖动（jitter）把同一整点触发分散到 1-5 分钟窗口
  - 对失败重试采用指数退避，避免雪崩

高调度量升级方案（仅在必要时启用）：

- 规则量达到较大规模（例如 10 万+）且分钟级扫描成本明显升高时，再评估引入：
  - `RedBeat`（Redis 存储的 Celery Beat 动态调度）
  - 或 `APScheduler` 作为独立调度服务

升级前提：

- 当前 `Celery Beat + DB 分发器` 已出现明显瓶颈
- 需要更细粒度触发精度或更复杂日历能力
- 团队可承担新增组件运维成本


## 9. 抓取与分块链路

网页同步不是“只抓取不处理”，而是完整接入现有 RAG 接入链路。

推荐链路如下：

1. 用户创建网页资源
2. 创建 `documents`
   - `carrier_type='web_page'`
   - `asset_kind='remote'`
   - `source_type='crawl'`
3. 创建 `knowledge_base_documents`
4. 创建 `kb_web_pages`
5. 若用户选择“创建后立即同步”，投递网页同步任务
6. 同步任务抓取网页正文并计算 `content_hash`
   - 当前实现入口位于 `genesis-ai-platform/services/web_content_extractor.py`
   - 主链路：`httpx -> trafilatura(markdown)`
   - 回退链路：当正文长度不足且 `fetch_mode in {"auto","browser"}` 时，切换 `Playwright` 获取渲染后 HTML，再次使用 `trafilatura(markdown)` 抽取
   - 最终兜底：若渲染后仍无法得到有效正文，则使用 `readability-lxml` 提取纯文本
   - `Playwright` 当前运行在独立子进程池中，以规避 Windows asyncio 子进程兼容问题并限制浏览器并发
7. 比较 `content_hash`
8. 若内容变化或配置要求强制重建，则：
   - 首次成功或内容变化时创建 `kb_web_page_versions`
   - 提交统一 parse pipeline
   - parse 阶段直接复用 `kb_web_page_versions.content_text`
   - 投递分块任务
   - 由分块任务生成统一协议 chunks
   - 后续继续进入训练/向量化任务
   若内容无变化：
   - 不新增版本记录
   - 跳过重建 chunks
9. 更新：
   - `kb_web_pages.last_synced_at`（同步成功即更新）
   - `kb_web_pages.next_sync_at`
   - `kb_web_pages.last_success_version_id`（仅在创建新成功版本时更新）
   - `knowledge_base_documents.parse_status`
   - `knowledge_base_documents.chunk_count`
10. 记录 `kb_web_sync_runs`


## 10. 定时抓取后自动触发分块

这是网页同步知识库的关键行为，必须在设计上明确。

结论：

- 定时抓取成功后，不能只更新网页快照
- 必须根据抓取结果决定是否触发分块与后续训练

推荐策略：

### 10.1 内容未变化

若新版本 `content_hash` 与最近成功版本一致：

- 默认跳过分块
- `kb_web_sync_runs.content_changed = false`
- 不新增 `kb_web_page_versions` 记录
- 更新 `kb_web_pages.last_synced_at`

可选扩展：

- 若页面配置 `force_rechunk_on_sync = true`
  则即使未变化也触发重新分块

### 10.2 内容已变化

若 `content_hash` 发生变化：

- 创建新版本记录
- 将该版本 `chunk_status` 标记为 `pending`
- 投递分块任务
- 分块成功后更新：
  - `chunk_status = 'completed'`
  - `chunk_count`
  - `knowledge_base_documents.chunk_count`
  - `kb_web_pages.latest_content_hash`
  - `kb_web_pages.last_content_changed_at`
  - `kb_web_pages.last_synced_at`

### 10.3 抓取失败

若抓取失败：

- 不删除现有 chunks
- 保留最近一次成功版本对应的检索能力
- 更新：
  - `kb_web_pages.sync_status = 'failed'`
  - `kb_web_pages.last_error`
  - `kb_web_sync_runs.status = 'failed'`

这样可以避免定时任务失败时导致线上检索结果被清空。

### 10.4 版本保留策略（最佳实践）

建议采用“双阈值清理”：

- 数量阈值：每个 URL 最多保留最近 `N` 个“成功且发生内容变化”的版本，建议 `N=20`
- 时间阈值：仅保留最近 `T` 天，建议 `T=180` 天

清理要求：

- 永远保留当前版本（`is_current=true`）
- 可额外保留最近一次成功版本的前一版本，便于快速回滚
- 清理旧版本时同步清理其快照载体与派生资源，避免孤儿数据

说明：

- 固定只保留 3-5 个版本通常不足以支撑线上问题定位
- 完全无限保留会导致持续存储膨胀，不符合长期运营最佳实践


## 11. Chunk 协议映射建议

网页分块也必须遵循统一 chunk 协议，不得单独发明结构。

至少保证：

- `content_blocks: list`
- `metadata_info.source_anchors: list`
- `metadata_info.page_numbers: list`
- `metadata_info.source_element_indices: list`

网页场景约定：

- `metadata_info.page_numbers = []`
- `metadata_info.source_anchors`
  可记录标题层级、锚点、URL fragment

- `metadata_info.source_element_indices`
  可记录抽取后正文块的顺序索引

- `content_blocks[].source_refs`
  作为块级权威来源，建议包含：
  - `url`
  - `canonical_url`
  - `version_id`
  - `anchor_text`
  - `heading_path`
  - `css_selector` 或 `xpath`
  - `dom_index`

- 名称
- URL

- 基本信息
- 抓取配置
- 同步计划
- 同步历史（来自 `kb_web_sync_runs`）
- 版本预览（来自 `kb_web_page_versions`）

### 12.3.1 版本预览策略（关键）

结论：

- 网页分块预览必须以“版本快照”为准，不能直接以内嵌线上 URL 作为主预览。

原因：

- 线上 URL 展示的是“当前最新页面”，而分块来自某个历史 `version`，两者可能不一致。
- 若直接嵌线上 URL，用户会看到 A 页面，但检索命中基于 B 版本，容易造成认知错位与排障困难。

推荐展示策略：

- 主预览：
  - 展示 `kb_web_page_versions.content_text`
  - 前端应支持 Markdown 阅读模式与纯文本模式切换
  - 默认优先按 Markdown 渲染；若版本来源为 `playwright+readability` 等纯文本结果，则退化为文本阅读模式
  - 明确显示 `version_no / content_hash / fetched_at` 等版本标识

- 辅助入口：
  - 提供“打开原网页”按钮（新窗口打开 URL）
  - 明确提示“该页面可能已更新，与当前版本快照不一致”

- 调试视图（可选）：
  - 若后续启用 `raw_html_document_id` 快照存储，可提供“查看原始 HTML”仅用于调试
  - 不建议将原始 HTML 作为默认主预览（样式依赖不完整、可读性差）

### 12.3.2 最新性校验按钮（手动校验 + 人工触发）

在页面详情或列表行提供“校验是否最新”按钮，建议行为：

1. 后端发起一次轻量校验（不落版本）：
   - 拉取当前网页并按同一抽取链路生成临时 `content_hash`
   - 与 `kb_web_pages.latest_content_hash` 比较

2. 返回校验结果：
   - `is_latest = true`：提示“当前版本已是最新”
   - `is_latest = false`：提示“发现新版本候选”

3. 当 `is_latest = false` 时：
   - 前端显示“立即同步”按钮
   - 用户确认后触发人工同步（写入 `kb_web_sync_runs`，走正式版本化与分块链路）

状态字段建议：

- `kb_web_pages.last_checked_at`
  - 最近一次“校验是否最新”时间（区分于 `last_synced_at`）
- `kb_web_pages.last_check_status`
  - 例如：`latest / outdated / failed`

说明：

- “校验”与“同步”分离可以降低误触发成本。
- 校验结果仅用于提示，不直接覆盖线上版本；真正入库仍通过正式同步任务完成。

### 12.4 前端缓存与查询 key 建议

- `['knowledge-base', kbId]`
- `['kb-web-sites', kbId, page, filters]`
- `['kb-web-pages', kbId, page, filters]`
- `['kb-web-sync-runs', kbId, page, filters]`
- `['kb-web-page-versions', kbWebPageId, page, filters]`

约束：

- 二级视图按需加载，未激活视图不请求数据
- 列表默认后端分页，禁止前端全量拉取后本地分页

### 12.5 列表复用与返回主表（关键实现建议）

`web_pages` 视图建议继续以 `knowledge_base_documents` 作为列表主查询表，原因：

- 现有文件列表的绝大多数能力都绑定在 `knowledge_base_documents` 语义上
- 可直接复用批量操作、启停、重命名、标签、元数据设置、分页筛选、权限校验等现有能力
- 无需为网页类型复制一套新的列表状态机和批量动作体系

推荐查询模型：

推荐返回形态：

- 基础字段来自 `kbd`：
  - `id`（即 `kb_doc_id`）
  - `parse_status`
  - `chunk_count`
  - `is_enabled`
  - `folder_id`
  - `display_name`
  - 审计字段

- 载体字段来自 `d`：
  - `name`
  - `carrier_type`
  - `asset_kind`
  - `source_type`
  - `source_url`

- 网页字段来自 `wp`：
  - `url`
  - `domain`
  - `sync_status`
  - `last_synced_at`
  - `last_content_changed_at`
  - `next_sync_at`
  - `latest_content_hash`

动作复用映射建议：

- 现有“解析/重新解析”按钮映射为“立即同步”
- “下载”按网页语义映射为“打开原网页/导出快照”
- 其余动作（批量启停、标签、元数据、重命名、移动目录）优先复用原有实现

### 12.6 开源框架借鉴与前端配套结论

参考文档 [开源框架-HTML与URL知识摄入对比.md](d:/workspace/python/genesis-ai/doc/analysis/开源框架-HTML与URL知识摄入对比.md) 后，前端与交互层建议吸收以下实践：

- 借鉴 FastGPT / MaxKB 的“统一列表 + 类型动作映射”思路
  不为网页类型重做一套页面骨架，而是在同一工作台骨架中按类型注册列与动作。

- 借鉴 RAGFlow / WeKnora 的“版本与运行态分离”
  页面视图重点看当前状态与可操作性，运行细节和历史问题排查放到 `web_sync_runs` 与版本预览中，不把所有历史噪声塞进主列表。

前端不应把底层网页抽取方案写死为当前实现，应预留 `Crawl4AI` 接入后的配置与观测能力。

建议预留以下能力：

  - `native_auto`
  - `crawl4ai_sdk`
  - `crawl4ai_service`

- 页面级或知识库级“浏览器策略”选择：
  - `auto / static / browser`
  - 对 `crawl4ai` 场景可继续沿用相同概念，避免前端出现两套互斥配置模型

- 详情页或版本页展示抽取器来源：
  - `extractor`
  - 是否浏览器渲染
  - 是否 Markdown 输出
  - 是否命中回退链路

- 若后续启用 Docker 独立服务版 `Crawl4AI`：
  - 前端无需感知服务部署细节
  - 仅通过后端返回的枚举值、状态字段和诊断信息展示当前页面实际采用的抽取通道

## 13.1 URL 解析技术选型（吸收开源实践后的最终方案）

结合开源对比与本项目现状，推荐解析链路如下：

1. 下载层：`httpx`
   - 用于默认 URL 获取、超时控制、重试与并发治理

2. 主抽取器：`trafilatura`
   - 作为正文抽取主方案，兼顾正文质量与元数据抽取能力

3. 动态页兜底：`Playwright`
   - 对 SPA 或必须执行 JS 才能拿到正文的页面启用

4. 备用抽取器：`readability-lxml`（可选）
   - 当主抽取器结果质量过低时作为回退

推荐回退链路：

- `httpx` 拉取 HTML
- `trafilatura` 以 `output_format="markdown"` 提取正文与结构化信息
- 质量不达标时切换 `Playwright` 获取渲染后 HTML，再次使用 `trafilatura(markdown)` 抽取
- 仍不达标时使用 `readability-lxml` 兜底
- 抽取文本标准化后计算 `content_hash`
- `content_hash` 不变则不重建 chunks，仅更新最近同步时间

说明：

- 该方案综合吸收 WeKnora（Playwright + trafilatura）、FastGPT（HTML 转文本质量重视）和 MaxKB（可扩展抓取策略）思路
- 与本方案中的 `kb_web_page_versions / kb_web_sync_runs` 结构天然匹配，便于后续问题定位和质量评估
- 当前实现的输出特征为：
  - 主链路输出 Markdown 风格正文
  - 最终 `readability-lxml` 兜底可能退化为纯文本
  - `parse` 阶段不会重新解析原始 HTML，而是直接复用版本表中的 `content_text`

### 13.1.1 当前实现与未来抽取引擎扩展边界

当前实现最稳定的扩展边界是“网页抽取层”，即：

- 上游输入：`url / fetch_mode / timeout / 页面级配置`
- 下游统一输出：
  - `extracted_text`
  - `raw_html`
  - `final_url`
  - `http_status`
  - `etag`
  - `last_modified`
  - `extractor`

建议后续无论底层接入：

- 原生 `httpx + trafilatura + Playwright + readability`
- `Crawl4AI` Python SDK（`pip install` 方式）
- `Crawl4AI` Docker 独立服务

都统一映射到上述结果模型，再进入版本化与统一 parse/chunk 链路。

这样可以保证：

- 版本表结构不需要因抽取引擎变化而重做
- `parse/chunk` 不感知底层抽取器
- 前后端可以通过 `extractor / extra_metadata` 做观测与排障


## 13.2 Firecrawl / Jina Reader / Crawl4AI 对比与选型

### 13.2.1 对比结论

- `Firecrawl`
  - 优势：云端能力成熟，批量抓取与复杂页面成功率高，Markdown 输出友好
  - 代价：云端依赖与成本；self-host 与 cloud 能力存在差异；**资源占用**与运维复杂度更高
  - 适配定位：作为“高成功率增强通道”而不是首选基础链路

- `Jina Reader`
  - 优势：单页提取体验好，接入速度快，适合快速原型与低门槛验证
  - 代价：长期生产链路对**外部服务**依赖较强；批量同步与私有化治理能力相对弱
  - 适配定位：作为“单页快速读取或临时补救”的辅助能力

- `Crawl4AI`
  - 优势：本地优先、开源可控、可直接 Python 集成，天然面向 Markdown / RAG 场景，且后续可平滑切换到 Docker 独立服务
  - 代价：浏览器中心型架构整体资源占用高于当前“静态优先”的原生链路；反爬、代理、浏览器指纹治理仍需自建策略
  - 适配定位：作为“复杂页增强抽取层”和“未来独立抓取服务”的首选升级方向

### 13.2.2 最终选型（当前阶段）

当前阶段最终选型保持：

- 主链路：`httpx + trafilatura`
- 兜底链路：`Playwright -> trafilatura -> readability-lxml`
- 新增预留：`Crawl4AI` 作为可插拔增强抽取引擎，支持 SDK 集成与未来服务化

原因：

- 与现有 Python 任务链路、`kb_web_page_versions / kb_web_sync_runs` 数据结构高度契合
- 无需引入额外服务即可落地，部署与维护成本最低
- 对“单 URL + 定时同步 + 内容变更触发分块”的核心目标已足够
- 当前实现已经明确把网页抽取层与下游版本化/解析/分块链路解耦，适合后续平滑接入 `Crawl4AI`

### 13.2.2.1 Crawl4AI 接入结论

当前建议对 `Crawl4AI` 采取“两阶段接入”：

1. 第一阶段：支持 `uv add crawl4ai` 的 SDK 集成
   - 作为应用内可选抽取通道接入
   - 适合 PoC、复杂页增强、低改造成本验证
   - 不要求当前就拆出独立服务

2. 第二阶段：预留 Docker 独立服务化能力
   - 当抓取规模、浏览器并发、资源隔离和监控需求上升时启用
   - 将 `Crawl4AI` 作为独立的网页抽取服务，通过 HTTP/SDK 适配层接入当前平台

结论：

- 现阶段不建议全量替换当前原生链路
- 建议把 `Crawl4AI` 作为“复杂页/动态页/抽取质量增强”通道接入
- 当前设计文档、表结构和前后端交互都应预留其存在，但下游协议保持不变

### 13.2.2.2 Crawl4AI 资源与格式判断

结论：

- `Crawl4AI` 无论是 `uv add` 方式还是未来 Docker 独立服务方式，均支持 Markdown 输出
- 安装形态不会改变其主要输出格式，区别在于“库内调用”还是“服务调用”
- 相比当前方案：
  - 与当前 `Playwright` 分支相比，资源消耗大体同量级
  - 与当前整体“`httpx` 静态优先”方案相比，通常更重

因此建议：

- 静态页仍优先保留当前原生链路
- `Crawl4AI` 优先用于：
  - JS 重页面
  - 结构复杂且当前抽取质量不稳定的页面
  - 后续需要统一浏览器池化与独立服务化的场景

### 13.2.3 何时启用 Firecrawl / Jina Reader / Crawl4AI

- 启用 `Crawl4AI` 的场景：
  - 每日同步规模显著上升（例如从几百 URL 增长到数千/上万）
  - 动态页、复杂页比例明显上升，当前 `Playwright + trafilatura` 定制成本变高
  - 需要更强的批量调度、抓取队列管理与抓取链路抽象
  - 仍坚持本地/私有化优先，不希望核心链路绑定外部云 API
  - 需要为未来 Docker 独立抓取服务提前统一接口模型

- 启用 `Firecrawl` 的场景：
  - 目标站点反爬强、JS 重、对抓取成功率要求极高
  - 团队可接受**云端成本**或具备 self-host 运维能力
  - 需要快速补齐“复杂网页抓取成功率”而非长期自研反爬体系

- 启用 `Jina Reader` 的场景：
  - 以单页快速抽取为主，追求最短接入路径
  - 作为失败回退通道或研发期实验工具
  - 不作为核心长期批量同步引擎


## 14. 后续扩展方向

该方案可以平滑扩展到其他远程资源类型：

- 飞书文档
- GitHub 仓库文件
- 云雀笔记
- Notion 页面
- Confluence 页面

统一方式：

- `documents` 继续表达远程载体
- `knowledge_base_documents` 继续表达知识库挂载与运行态
- 各资源类型使用各自 `kb_*` 主事实表
- 同步型资源统一拥有：
  - 版本表
  - 调度表
  - 运行记录表

### 14.1 后端架构预留建议（Crawl4AI）

后端建议显式预留“网页抽取 provider”层，而不是把 `Crawl4AI` 直接耦合进任务流程。

推荐抽象：

- `native_extractor`
  - 当前实现
  - `httpx + trafilatura + Playwright + readability-lxml`

- `crawl4ai_sdk_extractor`
  - 基于 `pip install crawl4ai` 在应用内直接调用

- `crawl4ai_service_extractor`
  - 面向未来 Docker 独立服务
  - 通过 HTTP 或适配客户端调用

统一约束：

- 三类 extractor 都必须返回同一结果模型
- `KBWebPageVersion`、`parse_task`、`chunk_task` 不感知底层 provider 差异
- `extractor` 与 `extra_metadata` 承担诊断与观测能力

推荐配置入口：

- 知识库级默认配置：
  - `web_extractor_engine`
  - `crawl4ai_mode`
  - `crawl4ai_endpoint`

- 页面级覆盖配置：
  - 是否强制使用 `crawl4ai`
  - 是否允许浏览器渲染
  - 是否仅对复杂页启用


## 15. 结论

网页同步知识库的正式方向为：

- 复用 `documents / knowledge_base_documents / chunks`
- 新增 `kb_web_pages / kb_web_page_versions / kb_web_sync_schedules / kb_web_sync_runs`
- 网页载体在 `documents` 中固定使用：
  - `carrier_type = 'web_page'`
  - `asset_kind = 'remote'`
  - `source_type = 'crawl'`
- 前端工作台遵循“一级通用导航 + 二级 `web_sites/web_pages/web_sync_runs`”的类型化视图注册方案
- `web_pages` 列表继续以 `knowledge_base_documents` 为主查询，关联 `documents + kb_web_pages` 扩展网页字段，最大化复用现有文件列表能力
- URL 解析技术栈采用 `httpx + trafilatura`，并以 `Playwright`（动态页）与 `readability-lxml`（抽取兜底）形成回退链路
- 当前主输出为 Markdown 风格正文，最终 `readability-lxml` 兜底时可能退化为纯文本；`parse` 阶段直接复用 `kb_web_page_versions.content_text`
- `Crawl4AI` 在设计上作为可插拔增强抽取引擎预留：
  - 近期支持 `pip install crawl4ai` 的 SDK 集成
  - 中长期支持 Docker 独立服务化接入
- 前后端均应围绕统一抽取结果模型预留 `Crawl4AI` 支持，而不改变下游版本化、统一 parse/chunk 协议与工作台交互骨架
- 外部/增强引擎按场景启用：规模与私有化优先时优先考虑 `Crawl4AI`，复杂反爬高成功率需求时选 `Firecrawl`，单页快速抽取/实验场景可用 `Jina Reader`
- 动态定时采用 `Celery Beat + DB 分发器`，保留 `beat` 独立进程；高并发场景通过队列隔离、批量分发、域名限流、幂等去重与削峰策略避免同步阻塞
- 分布式锁采用 Redis，执行防重采用“Redis 锁 + 数据库幂等约束”双保险
- 定时抓取成功后，必须根据内容变更结果决定是否自动触发分块
- 抓取失败时不清空旧 chunks，保证现有检索可用性

这套设计既能满足网页同步知识库的初版落地，也能为后续飞书、GitHub、云雀等同步型知识源提供统一骨架。
