## 2. 关键设计口径

前后端统一按这套模型理解：

- `chat_spaces`
  - 首页卡片层
  - 表示一个聊天空间/聊天入口容器
- `chat_sessions`
  - 进入空间后的具体会话
  - 左侧会话列表就是它
- `chat_messages`
  - 中间消息流
- `chat_turns`
  - 每轮执行快照

也就是说：

- 首页不是会话列表，而是空间列表
- 进入空间后，左侧才是 session 列表

## 6. 当前页面行为

### 6.1 首页

首页支持：

- 拉取聊天空间列表
- 搜索空间
- 按状态过滤 `active / archived`
- 卡片/列表视图切换
- 创建聊天空间
- 创建空间后自动创建第一条会话并跳转详情页
- 归档空间
- 删除空间

### 6.2 详情页

详情页支持：

- 根据 `spaceId` 拉取空间信息
- 拉取空间能力绑定
- 拉取当前空间下会话列表
- 若没有会话，自动创建首个会话
- 左侧切换不同 session
- 拉取当前 session 消息列表
- 流式发送消息
- 右侧保存模型、检索模板、知识库挂载、推理参数

如果在新会话继续开发，最推荐的顺序是：

1. 补详情页交互细节

2. 优化移动端体验

4. 等后端继续完善后再接：
   - 真实 RAG 引用

## 10. 最关键提醒

后续继续开发时，请始终保持这几个原则：

- 不要再把聊天页写回单个大文件
- API 层只负责请求与协议解析
- 页面层只做组合，不要塞太多业务逻辑
- 共享逻辑优先沉到 `hooks / utils / api / types`
- 继续沿用当前的“空间 -> 会话 -> 消息”心智，不要再混回单层聊天模型
